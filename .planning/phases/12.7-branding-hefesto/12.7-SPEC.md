# Phase 12.7: branding hefesto — Specification

**Created:** 2026-06-16
**Ambiguity score:** 0.14 (gate: ≤ 0.20)
**Requirements:** 6 locked

## Goal

Un clon fresco de los repos del portfolio no encuentra el literal `nemea` (case-insensitive) en ninguna parte de la superficie de clone-and-run: el string `nemea` pasa a `hefesto` en docker, DB, env de ejemplo, nombres de paquete, docs y refs de código, en `nemea-front` + `nemea-back` + los docs del orquestador root.

## Background

"Nemea" es marca registrada de los inversores de la marroquinería; para portfolio se rebrandea a **Hefesto** y se evita el conflicto. La Phase 12.6 ya hizo el branding **in-app** configurable por env con defaults neutros (`APP_NAME=Hefesto`, `demo@hefesto.com`, `admin@hefesto.com`) — un clon que arranca con `.env.example` ya muestra "Hefesto" en la UI. Lo que sigue hardcodeado con el literal `nemea` es **plomería**, no marca visible:

- `nemea-back/docker-compose.yml` — `container_name`/`POSTGRES_DB`/`POSTGRES_USER`/`POSTGRES_PASSWORD`/volume/healthcheck (servicios `postgres` y `postgres-test`).
- `nemea-back/.env.example` — `DATABASE_URL` y `DATABASE_URL_TEST`.
- `package.json` (+ `package-lock.json`) en ambos repos — `"name": "nemea-front"`/`"nemea-back"`.
- `nemea-back/src/app.controller.ts` (health `app: 'nemea-back'`) y `src/common/interceptors/logging.interceptor.ts` (`new Logger('Nemea')`).
- READMEs + `docs/ARCHITECTURE.md` en ambos repos; `CLAUDE.md` + `README.md` + `README.es.md` del root.
- Fixtures en 4 archivos de test de front y 8 specs/e2e de back.

Las migraciones **no** contienen `nemea` (los productos seed son dioses griegos: Hefesto, Ares, Hermes, Apolo, Poseidon, Artemisa, Atenea, Afrodita) → la data seed no se toca. Inventario completo y verificado (se corrió el find-replace real y se revirtió): `~/.claude/notes/Nemea-web/2026-06-15_rebrand-nemea-to-hefesto.md`.

## Requirements

1. **Infra naming (docker + DB)**: docker-compose y env de ejemplo usan `hefesto_*` de forma consistente.
   - Current: `docker-compose.yml` usa `nemea_postgres`/`nemea_db`/user `nemea`/pass `nemea_dev`/volume `nemea_pgdata` (+ los gemelos `_test`); `.env.example` `DATABASE_URL=postgresql://nemea:nemea_dev@localhost:5432/nemea_db` (+ `DATABASE_URL_TEST`).
   - Target: todos esos tokens → `hefesto_*`; los valores de `docker-compose.yml` (POSTGRES_DB/USER/PASSWORD/container/volume/healthcheck) y de `.env.example` (DATABASE_URL/DATABASE_URL_TEST) quedan mutuamente consistentes.
   - Acceptance: `git grep -i nemea -- docker-compose.yml .env.example` no devuelve nada; `docker compose up -d` levanta y el backend conecta a `hefesto_db` y corre migraciones sin error.

2. **Package identity (front + back)**: `name` de `package.json` + self-name del lockfile = `hefesto-front`/`hefesto-back`.
   - Current: `"name": "nemea-front"` / `"nemea-back"`; `package-lock.json` repite ese self-name (2 refs por repo).
   - Target: `hefesto-front`/`hefesto-back` en `package.json` y `package-lock.json`.
   - Acceptance: `git grep -i nemea -- package.json package-lock.json` vacío en ambos repos; `npm ci` (o install) corre sin romper.

3. **Code references (back)**: health response del controller + contexto del logger.
   - Current: `app.controller.ts` devuelve `app: 'nemea-back'`; `logging.interceptor.ts` usa `new Logger('Nemea')`.
   - Target: `'hefesto-back'` y `new Logger('Hefesto')`.
   - Acceptance: `GET /` (health) devuelve `app: "hefesto-back"`; `git grep -i nemea -- src` vacío en back.

4. **Docs (front + back + root)**: READMEs, ARCHITECTURE y docs del orquestador sin el literal `nemea` de marca.
   - Current: README + `docs/ARCHITECTURE.md` en front y back; `CLAUDE.md`/`README.md`/`README.es.md` del root referencian `Nemea`/`Nemea-web`.
   - Target: refs de marca → `Hefesto`/`Hefesto-web`. **En el repo root** solo cambian los tokens de marca en mayúscula `Nemea` (case-sensitive); los tokens de path de carpeta en minúscula `nemea-front/`/`nemea-back/` y el `.gitignore` se preservan (las carpetas locales NO se renombran).
   - Acceptance: front+back → `git grep -i nemea` vacío. Root → `git grep -n "Nemea" -- CLAUDE.md README.md README.es.md` vacío; `git grep -n "nemea-front\|nemea-back" -- .gitignore` sigue presente (paths intactos).

5. **Test fixtures**: specs/e2e rebrandeados para que la suite quede verde y grep-clean.
   - Current: 4 archivos de test de front + 8 specs/e2e de back contienen `nemea` en fixtures/strings.
   - Target: fixtures usan hefesto; fuente y fixtures cambian juntas (no se rompe ninguna aserción).
   - Acceptance: `npm test` (unit) verde en ambos repos; `git grep -i nemea` no encuentra fixtures con nemea.

6. **Mapping sin colateral**: solo cambian tokens de marca; migraciones y data seed intactas.
   - Current: las migraciones contienen cero `nemea`; mapping case-sensitive `nemea→hefesto`, `Nemea→Hefesto`, `NEMEA→HEFESTO`.
   - Target: `src/database/migrations/**` sin cambios; ninguna palabra que contenga `nemea` como substring queda mal reemplazada.
   - Acceptance: `git diff --stat` muestra cero cambios bajo `src/database/migrations/`; revisión manual del diff confirma que no hay substrings rotos.

## Boundaries

**In scope:**
- Reemplazo case-sensitive (`nemea→hefesto`, `Nemea→Hefesto`, `NEMEA→HEFESTO`) en los working trees de `nemea-front` y `nemea-back`: código, docker-compose, `.env.example`, `package.json` + lockfile, docs, fixtures de test.
- Docs del orquestador root: `Nemea`/`Nemea-web` (mayúscula) → `Hefesto`/`Hefesto-web` en `CLAUDE.md`, `README.md`, `README.es.md`.
- Un checklist de **follow-up manual** (en los docs de la fase) con las acciones de GitHub.

**Out of scope:**
- Renombrar las carpetas locales `nemea-front/`/`nemea-back/` — cosmético, rompe paths de sesión y no afecta al clonador (recibe la carpeta con el nombre del repo de GitHub). El `.gitignore` del root y los tokens de path se mantienen.
- Renames de repos en GitHub (`nemea-front`→`hefesto-front`, `nemea-back`→`hefesto-back`, `Nemea-web`→`Hefesto-web`), `git remote set-url`, push y About/description — outward/manual; GSD no pushea ni renombra sin OK explícito. Quedan como checklist de follow-up.
- Branding in-app (`APP_NAME`, mails demo/admin/contacto) — ya configurable por env con defaults Hefesto desde Phase 12.6. Sin cambios.
- Git history, mensajes de commit y artefactos pasados — explícitamente no es un riesgo (el conflicto de marca es por la superficie de clon viva, no por la historia).
- Artefactos internos de `.planning/` y `project_title` del config GSD — internos del orquestador privado, no portfolio-facing.

## Constraints

- El mapping es case-sensitive con 3 reglas; `nemea` no es substring de ninguna palabra real en estos repos, así que un replace global de token es seguro — pero el executor DEBE revisar el diff para confirmar que no hay colateral por substring.
- `docker-compose.yml` (POSTGRES_DB/USER/PASSWORD/volume) DEBE quedar consistente con `DATABASE_URL` de `.env.example` (ambos `hefesto`).
- Renombrar el volume orphanea el `nemea_pgdata` existente (data dev) → nuevo `hefesto_pgdata`; re-seedear con `docker compose down -v && docker compose up -d` + migraciones. Aceptable (data dev).
- El `.env` local del dev (gitignored) mantiene las creds viejas hasta que se actualice a mano; el clonador arranca limpio igual.
- Los tests e2e conectan a la DB de test → recrear el container `postgres-test` (hefesto_test) antes de `npm run test:e2e`.
- Branching: ambos sub-repos están hoy en `feature/phase-12.6-branding-configurable-por-env`; el executor crea branch por fase según config GSD del proyecto.
