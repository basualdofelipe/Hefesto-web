      **=========================================================================**
      **CRITICAL BUG #1 -- ALL ratePercent values from getAll() are NaN at runtime**
      **=========================================================================**

      Both `getLatestGatewayRates()` and `getInstallmentRates()` use raw SQL (`.query()`).
      Raw SQL returns PostgreSQL column names as-is: `rate_percent` (snake_case).
      The parse helpers do `parseFloat(rate.ratePercent as string)` but `rate.ratePercent`
      is `undefined` on raw rows -- the actual key is `rate_percent`.
      Result: `parseFloat(undefined) = NaN`. The `.ratePercent` property on every
      ParsedGatewayRate and ParsedInstallmentRate is **NaN**.

      The spread operator (`...rate`) copies the original `rate_percent` string, so
      at runtime each rate object has BOTH:
        - `rate_percent: "3.49"` (string, from spread -- the REAL value)
        - `ratePercent: NaN` (number, from parseFloat(undefined) -- BROKEN)

      **resolveRates MUST read the rate value using bracket notation:**
      ```typescript
      // For gateway rates:
      const gatewayRatePercent = parseFloat((matchedRate as any)['rate_percent'] as string);
      // Verify: if (isNaN(gatewayRatePercent)) throw new Error(...)

      // For installment rates:
      const installmentRatePercent = parseFloat((matchedInstallment as any)['rate_percent'] as string);
      // Verify: if (isNaN(installmentRatePercent)) throw new Error(...)
      ```

      Do NOT trust `.ratePercent` on any object from `config.rates` or `config.installments`.
      It is NaN. Always use `(obj as any)['rate_percent']` and parseFloat it yourself.

      **=========================================================================**
      **CRITICAL BUG #2 -- IVA/IIBB are percentages (21, 3.5), formulas need fractions (0.21, 0.035)**
      **=========================================================================**

      DB stores `iva_rate = 21.00` and `iibb_rate = 3.50`.
      `parseTaxConfig` (which uses `findOne`, NOT raw SQL) correctly returns:
        `{ ivaRate: 21, iibbRate: 3.5 }`
      These are numbers, not NaN -- parseTaxConfig works fine.

      BUT the prototype formulas expect fractions, not percentages:
        - `tasaConIVA = tasaBase * (1 + IVA_RATE)` -- expects 0.21, NOT 21
        - `costoProductoConIVA = costoProducto * (1 + IVA_RATE)` -- expects 0.21
        - `ivaCreditoProducto = costoProducto * IVA_RATE` -- expects 0.21
        - `retencionIIBB = netoRecibido * IIBB_RATE` -- expects 0.035

      If you pass ivaRate=21 without dividing, then:
        `tasaConIVA = 3.49 * (1 + 21) = 76.78` instead of `3.49 * 1.21 = 4.22`

      **resolveRates MUST convert percentages to fractions:**
      ```typescript
      return {
        gatewayRate: gatewayRatePercent,           // stays as percentage (3.49) -- divided by 100 in formula
        installmentRate: installmentRatePercent,    // stays as percentage -- divided by 100 in formula
        ivaRate: config.taxConfig.ivaRate / 100,   // 21 -> 0.21 (fraction for * (1 + ivaRate))
        iibbRate: config.taxConfig.iibbRate / 100,  // 3.5 -> 0.035 (fraction for * iibbRate)
        cptRate: ...,                               // stays as percentage -- divided by 100 in formula
      };
      ```

      **WHY gateway/installment/CPT stay as percentages but IVA/IIBB become fractions:**
      The prototype formulas use IVA/IIBB as multipliers: `* (1 + ivaRate)`, `* iibbRate`.
      But gateway/installment/CPT are divided inline: `* (tasaConIVA / 100)`, `* (cptRate / 100)`.
      Match the prototype exactly. Gateway rate 3.49 enters the formula as `3.49 / 100 = 0.0349`.
      IVA rate 0.21 enters as `1 + 0.21 = 1.21`. Both are correct in the prototype.

      **=========================================================================**

      **WARNING -- snake_case field names on gateway rates from raw SQL:**
      Rate objects from getAll() have snake_case field names at runtime because `getLatestGatewayRates()` uses raw SQL (`gr.*`). The `parseGatewayRate` helper only overrides `ratePercent` (to NaN -- see above) -- it does NOT rename `payment_method` to `paymentMethod` or `withdrawal_days` to `withdrawalDays`. The resolveRates filter MUST use:

      - `(rate as any)['withdrawal_days'] === withdrawalDays` (NOT `rate.withdrawalDays`)

    The app has a global `ResponseInterceptor` (in `nemea-back/src/common/interceptors/response.interceptor.ts`) that automatically wraps every controller return value in `{ data: T }`. Therefore:
    - Controller methods return `CalcResult` directly (NOT `{ data: CalcResult }`)
    - The client receives `{ data: CalcResult }` after the interceptor wraps it
    - If you return `{ data: CalcResult }`, the client gets `{ data: { data: CalcResult } }` -- WRONG

    - `POST /calculadora/forward` -- accepts CalcForwardDto. If productId provided, fetch cost from CostsService.calculateForProduct(productId) and use that cost (per CALC-03). Otherwise use costoProducto from DTO. Returns CalcResult directly (interceptor wraps to `{ data: CalcResult }`). Accessible to any authenticated user (both ADMIN and USER -- per CONTEXT.md "Both ADMIN and USER can access").
