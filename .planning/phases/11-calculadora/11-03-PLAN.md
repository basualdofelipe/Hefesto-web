<root_cause_analysis>
## Infinite Loop Chain (diagnosed from code read)

The crash is a circular re-render loop between CalculadoraClient and GatewaySelectors:

1. **GatewaySelectors mounts** -> useMemo computes effectiveMethod and effectiveDays
   -> useEffect (line 124) fires because selectedGateway + effectiveMethod are truthy
   -> calls notifyChange() -> calls onConfigChange({ ...new object })

2. **CalculadoraClient receives onConfigChange call** -> handleGatewayConfigChange
   (line 183, a plain function) calls setGatewayConfig(newConfig)
   -> CalculadoraClient re-renders

3. **On re-render**, handleGatewayConfigChange is a NEW function reference (not
   wrapped in useCallback) -> passed as onConfigChange prop to GatewaySelectors

4. **GatewaySelectors re-renders** -> notifyChange useCallback (line 106) has
   onConfigChange in its deps (line 121) -> onConfigChange is new -> notifyChange
   is recreated -> useEffect (line 128) has notifyChange in deps (line 135)
   -> useEffect fires again -> calls notifyChange() -> calls onConfigChange()
   -> GOTO step 2 -- INFINITE LOOP

## Fix Strategy (two changes, both required)

**Fix A (CalculadoraClient.tsx):** Wrap handleGatewayConfigChange in useCallback
so the function reference is stable across renders. This breaks the loop at step 3.

**Fix B (GatewaySelectors.tsx):** Remove the notifyChange useCallback + useEffect
indirection pattern entirely. Replace with:
- A single useEffect that calls onConfigChange directly when the computed config
  values change (using the actual values as deps, NOT the callback).
- Use a useRef for onConfigChange to avoid it being a dependency while still
  calling the latest version.

Fix A alone would stop the loop, but Fix B is needed for correctness -- without it,
any parent re-render (e.g., from typing in an input field) that creates a new
onConfigChange reference would still trigger a spurious useEffect re-fire.
Both fixes together make the component robust.
</root_cause_analysis>

Wrap handleGatewayConfigChange in useCallback to stabilize its reference:

```typescript
// BEFORE (plain function -- new reference every render):

  (newConfig: GatewayConfig): void => {
    setGatewayConfig(newConfig);
```

**File 2: nemea-front/src/components/calculadora/GatewaySelectors.tsx**

Replace the `notifyChange` useCallback + useEffect pattern (lines 106-136) with a single, safe useEffect that:

1. Uses a ref to hold the latest onConfigChange so it never appears as a dependency.
2. Depends ONLY on the actual config values (selectedGateway, effectiveMethod, effectiveDays, installments, showPlanOverride, planSlug).
3. Calls onConfigChangeRef.current() directly inside the effect.

Concrete changes:

A) Add a ref for the callback at the top of the component (after the state declarations):
```typescript
const onConfigChangeRef = useRef(onConfigChange);
onConfigChangeRef.current = onConfigChange;
```

B) Delete the entire notifyChange useCallback (lines 106-122).

C) Replace the useEffect (lines 124-136) with:
```typescript
useEffect(() => {
  if (selectedGateway && effectiveMethod) {
    onConfigChangeRef.current({
      gatewaySlug: selectedGateway,
      paymentMethod: effectiveMethod,
      withdrawalDays: effectiveDays,
      installments,
      planSlug: showPlanOverride ? planSlug : undefined,
    });
  }
}, [selectedGateway, effectiveMethod, effectiveDays, installments, showPlanOverride, planSlug]);
```

This eliminates onConfigChange from the dependency array entirely, breaking the circular dependency chain permanently. The ref pattern is the standard React solution for "call a callback in useEffect without it being a dependency."

D) Keep the existing useMemo derivations (effectiveMethod, effectiveDays) unchanged -- they are correct and not part of the loop.
