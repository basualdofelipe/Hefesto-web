# Phase 12.1: Code Review Report

**Reviewed:** 2026-04-09
**Depth:** standard
**Files Reviewed:** 38 (including supplementary context files)
**Status:** issues_found

## Summary

Phase 12.1 introduces a solid, well-structured dynamic permissions system. The core security model is sound: `JwtStrategy.validate()` hits the database on every request, ensuring permissions are always live (no stale JWT data used for authorization). The `PermissionsGuard` correctly maps snake_case permission keys to camelCase via `PERMISSION_TO_CAMEL`. The admin lockout protections (system role name is immutable, ADMIN cannot lose `canManageUsers`) are correctly enforced in `RolesService`. Guard registration order in `AuthModule` is correct (JwtAuthGuard before PermissionsGuard). The frontend middleware and server-side page guards provide redundant defense in depth.

Four warnings require attention before this ships:

1. `CreateUserDto` accepts an optional `roleId` but the DB column is `NOT NULL` — a missing `roleId` crashes with a raw DB error instead of a clean 400.
2. The backend has no endpoint to update a user's role after creation, but the frontend only exposes role selection at creation time — this is a functional gap that will require future backend work.
3. The migration `down()` function will fail with a PostgreSQL cast error when custom roles (non-ADMIN/USER names) exist, making rollback unsafe in production.
4. Self-deactivation is not blocked at the API level — only the UI hides the button, an admin calling `PATCH /api/users/:id/toggle-status` on their own ID via the API will lock themselves out.

## Warnings

### WR-01: Missing roleId validation causes DB-level crash instead of HTTP 400

**File:** `nemea-back/src/users/dto/create-user.dto.ts:19-22`
**Issue:** `roleId` is marked `@IsOptional()` so `POST /api/users` without a `roleId` passes validation. In `users.service.ts:52`, when `dto.roleId` is falsy the entity is built with `role: undefined`. TypeORM then attempts an `INSERT` with `role_id = NULL`, which PostgreSQL rejects with a NOT NULL constraint error — an unhandled exception that surfaces as a 500 instead of a user-friendly 400. The frontend always sends `roleId` (Zod schema requires it), but the backend contract is loose.

**Fix:** Either make `roleId` required in the DTO, or guard against it explicitly in the service:

```typescript
// Option A — make roleId required in create-user.dto.ts:
@ApiProperty({ description: 'UUID del rol a asignar' })
@IsUUID()
@IsNotEmpty()
roleId!: string;

// Option B — add a guard in users.service.ts create():
if (!dto.roleId) {
  throw new BadRequestException('El roleId es obligatorio');
}
```

### WR-02: Migration `down()` will fail for custom roles with a PostgreSQL cast error

**File:** `nemea-back/src/database/migrations/1773000000000-CreateRolesAndMigrateUsers.ts:118-129`
**Issue:** The `down()` path:
1. Copies role names from the `roles` table back into `users.role` (varchar).
2. Lowercases the value with `LOWER("role")`.
3. Casts to `users_role_enum('admin', 'user')`.

If any user has been assigned a custom role (e.g., `INVERSOR`), step 3 fails with:
```
ERROR: invalid input value for enum users_role_enum: "inversor"
```
This makes the migration irreversible once custom roles are in use, which is a production rollback risk.

**Fix:** Add a safety step before the cast that remaps unknown roles to the `user` enum value:

```sql
-- Add before the CAST ALTER TABLE:
UPDATE "users"
SET "role" = 'user'
WHERE "role" NOT IN ('admin', 'user');
```

### WR-03: No backend protection against self-deactivation

**File:** `nemea-back/src/users/users.controller.ts:66-92`
**Issue:** `PATCH /api/users/:id/toggle-status` has no check that the authenticated user is not deactivating themselves. The frontend hides the action button for the current user's own row (line 184: `user.id === currentUserId`), but the API itself has no such guard. A user with `can_manage_users` permission can call the endpoint directly (e.g., via curl or Swagger) with their own ID, immediately losing access. Since `JwtStrategy.validate()` checks `isActive` on every request, the lockout takes effect on the very next API call.

**Fix:** Inject `CurrentUser` and compare in the controller:

```typescript
// users.controller.ts
import { CurrentUser, JwtUser } from '../auth/decorators/current-user.decorator';

@Patch(':id/toggle-status')
@RequirePermission('can_manage_users')
async toggleStatus(
  @Param('id', ParseUUIDPipe) id: string,
  @CurrentUser() currentUser: JwtUser,
): Promise<User> {
  if (id === currentUser.id) {
    throw new BadRequestException('No puedes desactivar tu propia cuenta');
  }
  // ... rest of existing logic
}
```

### WR-04: `canManageUsers` used as admin bypass in `scenarios.service.ts` — semantic mismatch

**File:** `nemea-back/src/scenarios/scenarios.service.ts:173-177`
**Issue:** The `remove()` method uses `permissions.canManageUsers` to decide whether to bypass the owner check:

```typescript
const where = permissions.canManageUsers
  ? { id }
  : { id, user: { id: userId } };
```

`canManageUsers` is semantically about user/role administration, not scenario ownership. If a future admin role is created that has `canManageScenarios` but not `canManageUsers` (a legitimate configuration), scenario deletion behaves differently than a user would expect — they can delete their own scenarios but not orphaned ones from deactivated users. The comment in the controller says "owner or admin" but "admin" is implicitly defined as `canManageUsers`, not `canManageScenarios`.

**Fix:** Either document this assumption explicitly in the service, or introduce a dedicated check. If the intent is that any admin can clean up all scenarios, `canManageUsers` is acceptable but should be documented:

```typescript
// In scenarios.service.ts remove() — clarify the intent:
// Admin bypass: users with can_manage_users (system admins) can delete any scenario
// to allow cleanup of scenarios belonging to deactivated/removed users.
const isAdmin = permissions.canManageUsers;
const where = isAdmin ? { id } : { id, user: { id: userId } };
```

### IN-01: JWT token carries embedded permissions that are never used for authorization

**File:** `nemea-back/src/auth/auth.service.ts:44-49`, `nemea-back/src/auth/strategies/jwt.strategy.ts:32-43`
**Issue:** `AuthService.validateGoogleToken()` embeds `permissions` into the JWT payload (signed into the token). However, `JwtStrategy.validate()` ignores `payload.permissions` entirely and re-derives permissions from a fresh DB lookup of `user.role`. The embedded permissions in the token are surfaced to the frontend via `auth.ts:47` and used in the UI, but they are stale the moment a role is updated — the UI may show incorrect permissions until the user signs out and back in, even though the API correctly uses fresh permissions from the DB.

This is a design trade-off, not a security bug (API authorization is always correct). But the discrepancy means a role change is invisible to the UI until session refresh.

**Fix (low priority):** If real-time UI permission sync matters, consider removing `permissions` from the JWT and instead fetching them from a `/api/auth/me` endpoint on session load. Alternatively, document this known lag in the codebase.

### IN-02: Duplicate `PERMISSION_KEYS` constant defined in two frontend files

**File:** `nemea-front/src/app/(app)/roles/RolesClient.tsx:46-58`, `nemea-front/src/types/permissions.ts:29-41`
**Issue:** `PERMISSION_KEYS` is defined twice with identical content. `types/permissions.ts` exports it as `PERMISSION_KEYS` (typed as `readonly (keyof Permissions)[]`), and `RolesClient.tsx` re-declares a local `PERMISSION_KEYS` constant as a plain string array. The two are kept in sync manually, which is a maintenance risk.

**Fix:** Import and reuse the canonical one from `types/permissions.ts`:

```typescript
// In RolesClient.tsx — remove the local declaration and import:
import { PERMISSION_KEYS } from '@/types/permissions';
```

The type in `types/permissions.ts` is already `as const satisfies readonly (keyof Permissions)[]`, which is stricter and preferred.
