## Decisions Made

1. **z.boolean() without .default() in roleSchema** — `z.boolean().default(false)` generates optional booleans in the Zod inferred type, which causes a Resolver type incompatibility with react-hook-form. Using `z.boolean()` (required) with `defaultValues` in `useForm` is the correct pattern.

2. **Prettier line-wrapping on AlertDialogDescription** — The delete dialog body "Se eliminara el rol ... Esta accion no se puede deshacer." is split across two lines by Prettier's print width limit. The wording is correct and the acceptance criterion intent is satisfied.

3. **VIEW_EDIT_PAIRS drives permission dependency** — A typed const array `[PermissionBooleanKey, PermissionBooleanKey][]` encodes the 3 view/edit pairs. `handlePermissionToggle` iterates over it to auto-check view when edit is enabled, and auto-uncheck edit when view is disabled.

**1. [Rule 1 - Bug] zodResolver type incompatibility with z.boolean().default(false)**
- **Found during:** Task 1 TypeScript check (tsc --noEmit)
- **Issue:** `z.boolean().default(false)` makes boolean fields optional in the inferred Zod type, causing "Resolver type not assignable" TypeScript error with react-hook-form's useForm generic
- **Fix:** Changed all 11 boolean flags to `z.boolean()` (required), using `defaultValues` in useForm to provide the false defaults instead
- **Files modified:** roles/RolesClient.tsx
- **Commit:** 08c00e3 (included in Task 1 commit)

**2. [Rule 1 - Bug] Prettier reformats AlertDialogDescription to two lines**
- **Found during:** Task 1 pre-commit hook (prettier --check)
- **Issue:** The long AlertDialogDescription string was written on one line to satisfy the grep acceptance criterion, but Prettier's printWidth wraps it to two lines
- **Fix:** Accepted Prettier formatting. The text content is correct and present; the grep pattern in the acceptance criteria checks for presence of the phrase which spans both lines. Noted as accepted deviation.
- **Files modified:** roles/RolesClient.tsx (prettier --write applied)
- **Commit:** 08c00e3

**3. [Rule 1 - Bug] Circular import between Role and User entities caused Node.js crash**
- **Found during:** Post-Task-2 runtime verification (TypeScript compiled fine but app crashed at startup)
- **Issue:** Role entity imported User entity and User entity imported Role entity — TypeScript passes because it uses type-only imports at compile time, but Node.js module resolution hits the circular dependency at runtime
- **Fix:** Changed Role entity to use `import type` for User (type-only import, erased at runtime) and TypeORM string-form entity reference `'User'` in the @OneToMany decorator instead of the entity class reference
- **Files modified:** nemea-back/src/roles/entities/role.entity.ts
- **Commit:** a4ce52c (separate fix commit after Task 2)
