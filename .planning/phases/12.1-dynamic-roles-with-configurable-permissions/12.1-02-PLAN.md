**1. Fix ScenariosService admin bypass** (CRITICAL — Pitfall 6 from research):
The `remove()` method currently does:
```typescript
async remove(id: string, userId: string, role: string): Promise<void> {
  const where = role === Role.ADMIN ? { id } : { id, user: { id: userId } };
```
This compares `role` as a string against the enum. After migration, `role` is no longer a string.

Change the method signature to accept `permissions: Permissions` instead of `role: string`:
```typescript
async remove(id: string, userId: string, permissions: Permissions): Promise<void> {
  const where = permissions.canManageUsers ? { id } : { id, user: { id: userId } };
```
Import `Permissions` from `../../common/types/permission.enum`.

Remove the import of `Role` from `../../common/types/role.enum`.

The admin bypass now checks `canManageUsers` permission — if the user can manage users, they can delete any scenario. This is the correct semantic mapping: user management implies admin-level access.

**2. Update AuthResponseDto**:
Read the file first. Change the `role: string` field to `permissions: Permissions` (import from `common/types/permission.enum`). This DTO shapes what the frontend receives after Google login. The frontend needs permissions, not a role string.

If the DTO uses class-validator decorators, add appropriate decorators for the permissions object. If it's a simple type/interface, just change the type.
