# Phase 12.1: Dynamic Roles with Configurable Permissions - Research

**Researched:** 2026-03-29
**Domain:** NestJS authorization (guards, decorators, TypeORM relations) + NextAuth session extension + Next.js middleware permission routing
**Confidence:** HIGH

## Summary

This phase replaces a hardcoded `Role` enum (`ADMIN`/`USER`) with a dynamic `roles` table containing 11 boolean permission flags. The implementation spans four layers: database migration, backend auth system (guard + decorator + JWT), frontend session/permissions hook, and a new role management UI page.

The existing codebase has a clean, well-factored auth system: `RolesGuard` (global via `APP_GUARD`), `@Roles()` decorator, `JwtUser` interface, and `JwtStrategy`. All 40 `@Roles()` usages across 8 controllers follow identical patterns, making the migration mechanical. The frontend has two patterns for permission checking: `useIsAdmin()` hook (3 client components) and `session?.user?.role === 'admin'` checks (8 server component pages + sidebar). Both converge to the same boolean check.

**Primary recommendation:** Embed all 11 permission flags directly in the JWT payload (not role ID lookup). The flags are booleans totaling ~200 bytes -- negligible JWT overhead, eliminates per-request DB query, and the frontend already receives role data through the NextAuth JWT callback. Create a dedicated `RolesModule` (not extend `UsersModule`) since roles are a first-class entity with their own CRUD, and the `UsersModule` is already imported by `AuthModule` (adding roles there would create circular dependency risk).

### Pattern 1: Role Entity with Boolean Flags
**What:** TypeORM entity extending BaseEntity with 11 boolean permission columns
**When to use:** Core pattern for the entire phase

```typescript

import { BaseEntity } from '../../common/entities/base.entity';
import { User } from '../../users/entities/user.entity';

@Entity('roles')
@Unique(['name'])
export class Role extends BaseEntity {
  @Column({ type: 'varchar', length: 100 })
  name!: string;

  @Column({ type: 'varchar', length: 255, nullable: true })
  description!: string | null;

  @Column({ name: 'is_system', type: 'boolean', default: false })
  isSystem!: boolean;

  // Permission flags — all default false
  @Column({ name: 'can_view_products', type: 'boolean', default: false })
  canViewProducts!: boolean;

  @Column({ name: 'can_edit_products', type: 'boolean', default: false })
  canEditProducts!: boolean;

  // ... (9 more boolean columns following same pattern)

  @OneToMany(() => User, (user) => user.role)
  users!: User[];
}
```

### Pattern 2: Permission-Based Guard
**What:** Refactored RolesGuard that reads a `PERMISSIONS_KEY` from metadata and checks corresponding boolean flags on the user's role
**When to use:** Replaces current `RolesGuard` globally

```typescript
// nemea-back/src/auth/guards/permissions.guard.ts
import { CanActivate, ExecutionContext, ForbiddenException, Injectable } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { PERMISSIONS_KEY } from '../decorators/require-permission.decorator';

@Injectable()
export class PermissionsGuard implements CanActivate {
  constructor(private readonly reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredPermissions = this.reflector.getAllAndOverride<string[] | undefined>(
      PERMISSIONS_KEY,
      [context.getHandler(), context.getClass()],
    );

    if (!requiredPermissions || requiredPermissions.length === 0) {
      return true; // No permission requirement = allow authenticated user
    }

    const request = context.switchToHttp().getRequest();
    const user = request.user;

    if (!user || !user.permissions) {
      throw new ForbiddenException('Permisos insuficientes');
    }

    // ALL required permissions must be true (AND logic)
    const hasAll = requiredPermissions.every(
      (perm) => user.permissions[perm] === true,
    );

    if (!hasAll) {
      throw new ForbiddenException('Permisos insuficientes');
    }

    return true;
  }
}
```

### Pattern 3: @RequirePermission Decorator
**What:** `SetMetadata` decorator accepting one or more permission keys
**When to use:** On every controller method that was previously `@Roles()`

```typescript
// nemea-back/src/auth/decorators/require-permission.decorator.ts
import { SetMetadata } from '@nestjs/common';

export const PERMISSIONS_KEY = 'permissions';

// Accept single permission or array (AND logic — all must be true)
export const RequirePermission = (...permissions: string[]): ReturnType<typeof SetMetadata> =>
  SetMetadata(PERMISSIONS_KEY, permissions);
```

### Pattern 4: JWT Payload with Embedded Permissions
**What:** Sign all 11 permission booleans into the JWT payload so no DB lookup is needed per request
**When to use:** In `auth.service.ts` when generating token, and `jwt.strategy.ts` when validating

```typescript
// In auth.service.ts — after user loaded with role relation
const accessToken = this.jwtService.sign({
  sub: user.id,
  email: user.email,
  permissions: {
    canViewProducts: user.role.canViewProducts,
    canEditProducts: user.role.canEditProducts,
    // ... all 11 flags
  },
});

// In jwt.strategy.ts — validate returns permissions object
async validate(payload: JwtPayload): Promise<JwtUser> {
  const user = await this.usersService.findActiveByEmail(payload.email);
  if (!user) throw new UnauthorizedException('Usuario no autorizado');
  return {
    id: user.id,
    email: user.email,
    permissions: payload.permissions,
  };
}
```

### Anti-Patterns to Avoid
- **Checking role NAME in guards:** The whole point is to check permission FLAGS. Never `if (user.roleName === 'admin')` in guard logic.
- **Dual role+permission system:** Don't keep the old `@Roles()` decorator alongside the new `@RequirePermission()`. Complete replacement, not coexistence.
- **Lazy-loading role relation:** D-10 specifies `eager: true` on the User->Role ManyToOne. This ensures permissions are always available without explicit `relations: ['role']` in every query.
- **Mutable system roles:** ADMIN and USER roles must have `isSystem: true` flag to prevent deletion. Editing permissions is allowed, but deletion is blocked.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Permission decorator | Custom interceptor pipeline | NestJS `SetMetadata` + Reflector | Standard NestJS pattern, matches existing `@Roles()` exactly |
| Role-permission validation | Custom middleware chain | NestJS guard with `APP_GUARD` | Already using this pattern globally |
| Form validation (frontend) | Manual validation | zod + react-hook-form + zodResolver | Already established in UsersClient.tsx |
| Permission type safety | Loose strings | TypeScript const enum or string literal union | Prevents typos in permission names |

**Key insight:** The existing `@Roles()`/`RolesGuard` pattern is the exact same mechanism as `@RequirePermission()`/`PermissionsGuard` -- just checking different metadata. This is a refactor, not a new system.

## Migration Strategy

### Critical: Single Breaking Migration

The migration (D-06) must execute these steps in order within a single `up()` method:

1. **Create `roles` table** with UUID PK (from BaseEntity pattern), `name` (unique), `description` (nullable), `is_system` (boolean, default false), 11 boolean permission columns (all default false), + `created_at`/`updated_at` timestamps.

2. **Seed ADMIN role:**
   - All 11 flags = `true`
   - `is_system = true`
   - Use `RETURNING id` to capture the UUID for step 4.

3. **Seed USER role:**
   - View flags (`can_view_products`, `can_view_supplies`, `can_view_expenses`) = `true`
   - `can_use_calculator` = `true`
   - `can_manage_scenarios` = `true`
   - All edit/manage flags = `false`
   - `is_system = true`
   - Use `RETURNING id` to capture the UUID for step 4.

4. **Add `role_id` column** to `users` table as UUID, initially nullable (to allow UPDATE before NOT NULL).

5. **Migrate existing users:**
   - `UPDATE users SET role_id = '<admin-uuid>' WHERE role = 'admin'`
   - `UPDATE users SET role_id = '<user-uuid>' WHERE role = 'user'`

6. **Set `role_id` NOT NULL** after all users migrated.

7. **Add FK constraint** on `role_id` referencing `roles(id)`.

8. **Drop `role` column** from `users`.

9. **Drop `users_role_enum`** type (created in the first migration).

**Important:** The migration uses raw SQL with `RETURNING id` to capture seed UUIDs. This is the established pattern from existing seed migrations.

### JWT Impact

After migration, existing JWTs in the wild will have `{ sub, email, role: 'admin' }` (old format). The `jwt.strategy.ts` `validate()` method already does a DB lookup on every request, so the actual user data is fresh. The strategy should be updated to read permissions from the user entity (which now has the role relation). Old JWTs will still authenticate but get permissions from the DB, not the token payload. New JWTs after login will include the permissions object.

## Common Pitfalls

### Pitfall 1: Circular Dependency Between RolesModule and UsersModule
**What goes wrong:** If RolesModule imports UsersModule (to count users per role) and UsersModule imports RolesModule (to validate role existence), NestJS throws a circular dependency error.
**Why it happens:** TypeORM entities can reference each other via relations, but NestJS modules cannot circularly import.
**How to avoid:** RolesModule only needs `TypeOrmModule.forFeature([Role])`. User count per role can be done via a raw query or by injecting the User repository directly in RolesService. If UsersModule needs RolesService, use `forwardRef(() => RolesModule)`.
**Warning signs:** `Error: Nest can't resolve dependencies` at startup.

### Pitfall 2: TypeORM Eager Loading + FindAll Performance
**What goes wrong:** With `eager: true` on User->Role, every `usersRepository.find()` call JOINs the roles table. This is intentional for individual user loads but could be unexpected in bulk queries.
**Why it happens:** TypeORM eager relations load automatically unless explicitly excluded.
**How to avoid:** This is actually fine for this app's scale (2-10 users). If needed, specific queries can use `loadEagerRelations: false`. Document this as a known behavior.
**Warning signs:** Slow user list queries (unlikely at this scale).

### Pitfall 3: NextAuth Session Not Updating After Role Change
**What goes wrong:** Admin changes a user's role or edits a role's permissions, but the user's session still has old permissions until they re-login.
**Why it happens:** NextAuth JWT is signed once at login and persists for 7 days (current `expiresIn: '7d'`). The frontend session callback reads from the JWT, not from the DB.
**How to avoid:** Accept this limitation for now (it matches the current behavior where role changes require re-login). Document it. Future enhancement: session refresh mechanism.
**Warning signs:** User reports "I changed their role but they still see old stuff." Answer: they need to log out and back in.

### Pitfall 4: Migration Failure on Existing Data
**What goes wrong:** If any user has a role value that isn't 'admin' or 'user', the UPDATE in step 5 won't match them, and step 6 (SET NOT NULL) will fail.
**Why it happens:** The enum was ADMIN/USER only, so this shouldn't happen with valid data. But if there's test data or manual DB edits, it could.
**How to avoid:** Add a safety check: `UPDATE users SET role_id = '<user-uuid>' WHERE role_id IS NULL` before the NOT NULL constraint.
**Warning signs:** Migration error: "column role_id contains null values."

### Pitfall 5: Frontend isAdmin Prop Threading
**What goes wrong:** Many components receive `isAdmin: boolean` as a prop from parent pages. The refactor must change ALL of these to permission-specific props (e.g., `canEdit: boolean`) or the component tree breaks.
**Why it happens:** 10+ pages pass `isAdmin` to child components. Missing one creates a runtime error.
**How to avoid:** Grep for `isAdmin` across the entire frontend. Count: ~30 locations across 15 files. Consider replacing `isAdmin` with more specific props like `canEdit` per component, or passing the full permissions object.
**Warning signs:** TypeScript compilation errors (good -- they catch missing prop changes).

### Pitfall 6: Scenarios Service Admin Bypass
**What goes wrong:** `ScenariosService.remove()` currently checks `role === Role.ADMIN` to bypass owner filter. After migration, `role` is no longer a string but a relation.
**Why it happens:** The service does `if (role === Role.ADMIN)` which compares against the enum value string.
**How to avoid:** Change the admin bypass to check a permission flag instead: pass `canManageScenarios` or a new `isAdmin`-equivalent derived from permissions.
**Warning signs:** Admin can no longer delete other users' scenarios.

### Migration SQL Pattern (critical path)
```typescript
// Pattern from existing migrations — raw SQL with seed data
public async up(queryRunner: QueryRunner): Promise<void> {
  // 1. Create roles table
  await queryRunner.query(`
    CREATE TABLE "roles" (
      "id" uuid NOT NULL DEFAULT uuid_generate_v4(),
      "created_at" TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
      "updated_at" TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
      "name" character varying(100) NOT NULL,
      "description" character varying(255),
      "is_system" boolean NOT NULL DEFAULT false,
      "can_view_products" boolean NOT NULL DEFAULT false,
      "can_edit_products" boolean NOT NULL DEFAULT false,
      "can_view_supplies" boolean NOT NULL DEFAULT false,
      "can_edit_supplies" boolean NOT NULL DEFAULT false,
      "can_view_expenses" boolean NOT NULL DEFAULT false,
      "can_edit_expenses" boolean NOT NULL DEFAULT false,
      "can_use_calculator" boolean NOT NULL DEFAULT false,
      "can_manage_scenarios" boolean NOT NULL DEFAULT false,
      "can_view_dashboard" boolean NOT NULL DEFAULT false,
      "can_manage_config" boolean NOT NULL DEFAULT false,
      "can_manage_users" boolean NOT NULL DEFAULT false,
      CONSTRAINT "UQ_roles_name" UNIQUE ("name"),
      CONSTRAINT "PK_roles_id" PRIMARY KEY ("id")
    )
  `);

  // 2. Seed ADMIN role (all true)
  const [adminRole] = await queryRunner.query(`
    INSERT INTO "roles" ("name", "description", "is_system",
      "can_view_products", "can_edit_products",
      "can_view_supplies", "can_edit_supplies",
      "can_view_expenses", "can_edit_expenses",
      "can_use_calculator", "can_manage_scenarios",
      "can_view_dashboard", "can_manage_config", "can_manage_users")
    VALUES ('ADMIN', 'Administrador con acceso total', true,
      true, true, true, true, true, true, true, true, true, true, true)
    RETURNING id
  `);

  // 3. Seed USER role (view + calculator + scenarios)
  const [userRole] = await queryRunner.query(`
    INSERT INTO "roles" ("name", "description", "is_system",
      "can_view_products", "can_edit_products",
      "can_view_supplies", "can_edit_supplies",
      "can_view_expenses", "can_edit_expenses",
      "can_use_calculator", "can_manage_scenarios",
      "can_view_dashboard", "can_manage_config", "can_manage_users")
    VALUES ('USER', 'Usuario con acceso de lectura', true,
      true, false, true, false, true, false, true, true, false, false, false)
    RETURNING id
  `);

  // 4-7. Migrate users
  await queryRunner.query(`ALTER TABLE "users" ADD COLUMN "role_id" uuid`);
  await queryRunner.query(`UPDATE "users" SET "role_id" = '${adminRole.id}' WHERE "role" = 'admin'`);
  await queryRunner.query(`UPDATE "users" SET "role_id" = '${userRole.id}' WHERE "role" = 'user'`);
  await queryRunner.query(`UPDATE "users" SET "role_id" = '${userRole.id}' WHERE "role_id" IS NULL`);
  await queryRunner.query(`ALTER TABLE "users" ALTER COLUMN "role_id" SET NOT NULL`);
  await queryRunner.query(`ALTER TABLE "users" ADD CONSTRAINT "FK_users_role_id" FOREIGN KEY ("role_id") REFERENCES "roles"("id")`);

  // 8-9. Drop old column and enum
  await queryRunner.query(`ALTER TABLE "users" DROP COLUMN "role"`);
  await queryRunner.query(`DROP TYPE "public"."users_role_enum"`);
}
```

## Discretion Recommendations

### 1. @RequirePermission: Support Array (AND Logic)
**Recommendation:** Accept an array of permission strings. ALL must be true. This allows `@RequirePermission('can_view_products', 'can_edit_products')` for endpoints that need both read + write access.

In practice, most endpoints need only one permission (e.g., `can_edit_products` implies the user should be able to edit). But having array support costs nothing and future-proofs.

### 2. JWT Strategy: Embed All 11 Flags in JWT
**Recommendation:** Embed permissions in JWT payload.

**Trade-off analysis:**
- JWT size increase: ~200 bytes for 11 booleans. Current JWT is ~300 bytes. Total ~500 bytes -- still well within cookie limits.
- Eliminates DB query on every request in the guard (currently `jwt.strategy.ts` already does a DB lookup, but that validates the user exists, not roles).
- Downside: Permission changes don't take effect until re-login. This is acceptable per Pitfall 3 analysis and matches current behavior.

**However:** The `jwt.strategy.ts` ALREADY does `findActiveByEmail` on every request. Since user now has `eager: true` role, permissions are loaded for free. So we could read permissions from the DB lookup instead of the JWT. **Recommended hybrid approach:** Store permissions in JWT (for frontend session consumption) but also validate from DB in the strategy (for backend guard). The strategy returns `payload.permissions` directly (trusting the JWT for speed) since the user existence is already validated.

### 3. Permission Summary Display: Badges with Count
**Recommendation:** Show permission summary as a compact badge count (e.g., "8/11 permisos") with a tooltip or expandable detail showing which ones are active. This keeps the table clean while still being informative. Use color coding: green badge for full access, amber for partial, red for none.

### 4. Separate RolesModule
**Recommendation:** Create `RolesModule` as a separate module. Reasons:
- Roles are a first-class entity with their own CRUD, not a sub-feature of Users.
- `UsersModule` is already imported by `AuthModule`. Adding more to it increases coupling.
- Future RolesModule may need to be imported by other modules for permission checking.
- Register `TypeOrmModule.forFeature([Role])` in RolesModule and export it. UsersModule imports RolesModule to access the Role entity for the FK relationship.

## Open Questions

1. **GET endpoint protection with view permissions**
   - What we know: Current GET endpoints (list products, list supplies, etc.) have NO `@Roles()` decorator -- they're accessible to any authenticated user.
   - What's unclear: Should we add `@RequirePermission('can_view_products')` to GET endpoints, or keep them open to all authenticated users and rely on frontend route protection?
   - Recommendation: Add view permissions to GET endpoints for defense in depth. A user without `can_view_products` shouldn't be able to curl the API either. This is a small addition since the guard already runs.

2. **Server component permission access**
   - What we know: 8 pages are server components that do `const session = await auth()` and read `session?.user?.role === 'admin'`. These need to read permissions from the session.
   - What's unclear: NextAuth v5 beta session structure for custom fields in server components.
   - Recommendation: The current `auth.ts` session callback already extends the session with custom fields (`accessToken`, `role`, `id`). Add `permissions` the same way. Server components read `session?.user?.permissions?.canEditProducts`.
