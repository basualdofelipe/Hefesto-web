# Phase 12.1: Dynamic Roles with Configurable Permissions - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the hardcoded `Role` enum (ADMIN/USER) with a dynamic roles system. Admin can create custom roles and assign granular boolean permissions via a dedicated management page. Users are assigned a role via FK. Auth guards check permission flags instead of role names.

</domain>

<decisions>
## Implementation Decisions

### Permission Granularity
- **D-01:** 11 boolean permission flags, organized as view/edit per module:
  - `can_view_products` — See products, costs, BOM
  - `can_edit_products` — Create/edit/delete products, BOM, prices
  - `can_view_supplies` — See supplies, suppliers
  - `can_edit_supplies` — Create/edit supplies, prices, suppliers
  - `can_view_expenses` — See expenses
  - `can_edit_expenses` — Create/edit/delete expenses
  - `can_use_calculator` — Use Tiendanube calculator (forward/inverse)
  - `can_manage_scenarios` — Create/edit/delete own scenarios
  - `can_view_dashboard` — See investor dashboard
  - `can_manage_config` — Edit Tiendanube config (rates, plans, taxes)
  - `can_manage_users` — Create/edit/deactivate users, manage roles
- **D-02:** NOT per-resource (no CASL, no Spatie pattern). Simple boolean flags on the role entity. If more granularity needed in the future, add a new flag — don't redesign.
- **D-03:** User wants this SaaS-ready — dynamic role creation matters more than flag count.

### Role Management UI
- **D-04:** Dedicated `/roles` page with table + dialog. Same pattern as catalogs page.
  - Table shows: role name, user count assigned, permission summary (badges or icons)
  - Click row opens edit dialog with permission toggles (11 switches/checkboxes)
  - "Crear rol" button opens create dialog
  - ADMIN and USER roles are seed defaults — can be edited but not deleted (system roles)
- **D-05:** Route `/roles` is admin-only (requires `can_manage_users` permission)

### Migration Strategy
- **D-06:** Breaking migration — single migration that:
  1. Creates `roles` table with UUID PK, name (unique), description (nullable), 11 boolean permission columns (all default false), + BaseEntity timestamps
  2. Seeds ADMIN role (all flags true) and USER role (all view flags true, all edit flags false, can_use_calculator true, can_manage_scenarios true)
  3. Adds `role_id` UUID FK column to `users` table
  4. Migrates existing users: `role = 'admin'` → ADMIN role UUID, `role = 'user'` → USER role UUID
  5. Drops the `role` varchar/enum column from `users`
- **D-07:** No dual period, no soft migration. Clean cut.

### Backend Auth Changes
- **D-08:** `RolesGuard` refactored: instead of checking `requiredRoles.includes(user.role)`, it checks specific permission flags on the user's loaded role entity.
- **D-09:** Replace `@Roles(Role.ADMIN)` decorator pattern with a `@RequirePermission('can_edit_products')` decorator (or similar). Each endpoint declares which permission it needs.
- **D-10:** User entity gets `@ManyToOne(() => Role, { eager: true })` so permissions are always loaded with the user — no extra query.
- **D-11:** JWT payload includes role permissions (or role ID that gets looked up). Decision for planner: include all 11 flags in JWT vs look up role on each request.

### Frontend Permission Checks
- **D-12:** New `usePermissions()` hook replaces `useIsAdmin()`. Returns an object with all 11 flags: `const { canEditProducts, canViewDashboard } = usePermissions()`.
- **D-13:** Hook reads permissions from the session (NextAuth session object must include role permissions).
- **D-14:** `ADMIN_ONLY_ROUTES` in middleware.ts replaced with permission-based route mapping. Each route maps to a required permission flag.
- **D-15:** Components use `usePermissions()` to show/hide buttons, forms, and actions. Example: create/edit buttons only render if `canEditProducts` is true.

### Claude's Discretion
- How to structure the `@RequirePermission` decorator (single permission vs array)
- Whether to embed all 11 flags in JWT or do a role lookup on each request (trade-off: JWT size vs DB query)
- How to display permission summary in the roles table (badges, icons, or count)
- Whether to create a RolesModule or add role management to the existing UsersModule

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- Catalogs page pattern (table + create/edit dialog) — reuse for /roles page
- `BaseEntity` with UUID PK + timestamps — use for Role entity
- `@CurrentUser()` decorator — already supports extracting any JwtUser field
- Shadcn Switch component — available for permission toggles in role dialog

### Established Patterns
- TypeORM entities with `@Entity`, `@Column`, `@ManyToOne` — consistent across all modules
- Guards registered globally in AppModule via `APP_GUARD`
- Seed data in migrations (not separate seed files)
- Frontend uses `useSession()` from NextAuth for auth state

### Integration Points
- Every controller that uses `@Roles()` needs to switch to `@RequirePermission()`
- `middleware.ts` route protection needs to read permissions instead of role name
- NextAuth session callback needs to include role permissions
- AppSidebar uses `useIsAdmin()` for conditional navigation — needs usePermissions()

</code_context>

<specifics>
## Specific Ideas

- User wants this SaaS-ready: the ability for admin to CREATE custom roles is more important than having many pre-built roles
- ADMIN and USER are seed defaults that can be edited but not deleted
- Future roles the user envisions: INVESTOR (view dashboard + scenarios only), CM (view products + future stock features)
- The 11 flags were designed to cover all current app features with room for future flags via simple migration

</specifics>

<deferred>
## Deferred Ideas

- INVESTOR role pre-creation — admin creates this via the UI, not seeded (Phase 13 will document what permissions an investor needs)
- Per-resource permissions (CASL/Spatie) — explicitly rejected for now, revisit only if 11 flags prove insufficient
- Stock management features for CM role — v2+ scope

</deferred>
