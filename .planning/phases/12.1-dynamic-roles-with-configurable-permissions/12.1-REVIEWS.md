## Additional Findings (Pre-mortem, not caught by cross-AI reviewers)

### 8. Swagger docs lose permission visibility (LOW)

Endpoints decorated with `@RequirePermission('can_edit_products')` no longer expose the required permission in Swagger UI at `/api/docs`. The old `@Roles(Role.ADMIN)` had implicit Swagger visibility via the enum. The new decorator uses `SetMetadata` which Swagger doesn't pick up. Fix: add a custom `@ApiOperation({ summary: '...', description: 'Requires: can_edit_products' })` pattern, or create a small Swagger plugin/decorator that reads `PERMISSIONS_KEY` metadata and adds it to the operation description automatically. Not blocking but degrades API documentation for future devs and investors testing endpoints.

### 9. `canViewDashboard` toggle has no enforcement point (LOW)

The Role entity, frontend Permissions type, and /roles UI all include `canViewDashboard` as a toggleable permission. But the dashboard page doesn't exist yet (Phase 13), no route maps to it in `ROUTE_PERMISSIONS`, and no backend endpoint checks it. An admin toggling this flag in the /roles UI gets no visible effect. Fix: either add a disabled state / tooltip in the /roles UI saying "Dashboard (fase 13 — proximamente)" on the `canViewDashboard` toggle, or simply document it in the checkpoint verification notes so the tester knows it's expected to be inert.

### Overall Verdict

**NEEDS 4 TARGETED FIXES before execution. Plans are otherwise sound.**

## Senior Pre-mortem

**Reviewer:** Senior (internal)
**Date:** 2026-04-09

Context: Phase 12.1 was previously executed, but the session crashed and STATE.md got corrupted. User reverted all changes (`git checkout`) in both sub-repos. The plans were already revised with ROUND 3 fixes addressing all 9 consensus issues. This pre-mortem is against the REVISED plans, post-revert, before re-execution.

### Will break

1. **Spec files import deleted `role.enum.ts` — 4 files will fail `tsc --noEmit` in Plan 02.** Plan 02 Task 2 deletes `nemea-back/src/common/types/role.enum.ts`. Four spec files still import `Role` from it: `auth.controller.spec.ts`, `auth.service.spec.ts`, `users.controller.spec.ts`, `users.service.spec.ts`. `tsconfig.json` includes spec files (only `tsconfig.build.json` excludes them). Plan 02's verification runs `npx tsc --noEmit` which WILL fail on these imports. No plan task addresses spec file updates — the executor must improvise. The fix is mechanical (change imports, update mock shapes from `role: Role.ADMIN` to `permissions: { canManageUsers: true, ... }`, update assertions), but it adds unplanned work.

2. **Security gap: all `@Roles` decorators become inert after Plan 01.** Plan 01 Task 2 Step 10 REPLACES `RolesGuard` with `PermissionsGuard` in `auth.module.ts` (`grep -c 'RolesGuard' auth.module.ts` must return 0). After this, the 9 existing controllers still have `@Roles(Role.ADMIN)` decorators, but `PermissionsGuard` only reads `PERMISSIONS_KEY` metadata (not `ROLES_KEY`). All mutation endpoints (products, supplies, catalogs, expenses, suppliers, TN config) are temporarily protected only by JWT auth — any authenticated user can call them. This gap lasts until Plan 02 adds `@RequirePermission`. Not a production issue (plans run back-to-back, server isn't live), but be aware.

### Might break

1. **`forwardRef` vs `TypeOrmModule.forFeature` ambiguity.** Plan 01 Task 2 Step 7 discusses `forwardRef(() => RolesModule)` in `UsersModule`. Step 8 then says "Recommended approach: use `TypeOrmModule.forFeature([User, Role])` instead — eliminates circular dependency concern." These are contradictory approaches. If the executor follows BOTH (adds forwardRef AND changes forFeature), there's unnecessary complexity. The recommendation (Step 8) is correct — ignore Step 7.

2. **Auth spec mock shapes need rewriting, not just import changes.** Beyond fixing the `Role` import, the 4 spec files have deep assumptions about the data shape: `role: Role.ADMIN`, `result.user.role`, assertions like `expect(result.user.role).toBe(Role.ADMIN)`. These need structural changes to use `permissions` object. The executor needs to rewrite mock data and assertions, not just swap imports.

3. **Previous crash root cause unknown.** The first execution crashed and corrupted STATE.md. The plans haven't changed. If the crash was caused by context window exhaustion (likely — Plan 01 touches 20 files, Plan 02 touches 13, Plan 03 touches 30), the same risk exists. Wave 2 runs Plan 02 (backend, 13 files) and Plan 03 (frontend, 30 files) in parallel — that's 43 files worth of context for parallel executors. Consider running Plan 02 and Plan 03 sequentially if the crash recurs.

### Missing from the plans

1. **No spec file update task in any plan.** Add to Plan 02 Task 2 or create a new subtask: update the 4 spec files to use `Permissions` type and remove `Role` enum references.

2. **No `npm run test` step.** None of the 4 plans run `npm run test` as verification. After the auth system overhaul, running the existing test suite would catch integration issues. At minimum, Plan 01 should run tests after Task 2.

3. **`AuthResponseDto` update is in Plan 02 Task 2 but belongs in Plan 01.** Plan 01 Task 2 Step 5 changes `auth.service.ts` to return `permissions` instead of `role`, but the `AuthResponseDto` still has `role: string`. Plan 02 lists it in files_modified, but the Swagger docs would be wrong between Plan 01 and Plan 02. Since Plan 01 already changes the actual response shape, the DTO should be updated in the same plan. The plan mentions it in Step 5 ("Update the AuthResponseDto") but it's also listed in Plan 02 — ambiguous ownership.

The ROUND 3 fixes address all 9 consensus issues from the cross-AI review. The plans are architecturally sound. The spec file issue is the only guaranteed `tsc` failure, and the executor's verification step will catch it immediately. The fix is 15 minutes of mechanical work.
