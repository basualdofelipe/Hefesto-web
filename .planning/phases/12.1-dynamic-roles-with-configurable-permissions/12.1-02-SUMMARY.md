## Decisions Made
- TiendanubeConfig GET endpoints (getAll, gateways, installments, taxes, plans) have no @RequirePermission -- they contain public rate data needed by calculator users who may not have can_view_products
- Kept Plan 01's approach for ScenariosService.remove() which accepts `isAdmin: boolean` instead of full `Permissions` object -- cleaner interface, controller derives isAdmin from permissions.canManageUsers

## Deviations from Plan

### Minor Deviations

**1. ScenariosService admin bypass already fixed by Plan 01**
- **Context:** Plan specified changing service to accept `permissions: Permissions`, but Plan 01's deviation fix already changed it to `isAdmin: boolean`
- **Decision:** Kept existing approach -- controller computes `isAdmin = permissions.canManageScenarios && permissions.canManageUsers`, service receives boolean. Same security outcome, cleaner service interface.
- **Impact:** None -- the acceptance criteria intent (no role string comparison in service) is fully satisfied.

**2. AuthResponseDto already updated by Plan 01**
- **Context:** Plan specified updating auth-response.dto.ts from `role: string` to `permissions: Permissions`, but Plan 01 already did this.
- **Impact:** None -- no work needed.

**3. Additional GET endpoints protected beyond plan mapping table**
- **Context:** Plan's mapping table listed specific GET endpoints but missed sub-resource GETs (supplies/:id/prices, products/:id/bom, products/:id/prices, expenses/:id). Applied defense-in-depth consistently to ALL GET endpoints including sub-resources.
- **Impact:** Stronger security -- more endpoints protected than minimum specified.
