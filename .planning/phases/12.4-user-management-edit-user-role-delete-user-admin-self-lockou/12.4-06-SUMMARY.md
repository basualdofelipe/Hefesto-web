# Phase 12.4 Plan 06: UsersClient Integration + Controller Spec Rewrite Summary

**One-liner:** Integrates `EditUserDialog` (Plan 03) and `DeleteUserAlertDialog` (Plan 04) into `UsersClient.tsx` — replacing the legacy toggle-status button with Editar+Borrar row actions and own-row "(tu cuenta)" marker (D-08/D-09) — and rewrites `users.controller.spec.ts` to delete the toggle-status suite and add PATCH+DELETE coverage, closing the final 5 tsc errors that Plan 01 deliberately left for this plan to resolve.

## What Was Built

### Frontend — `nemea-front/src/app/(app)/usuarios/UsersClient.tsx` (MODIFIED)

**Removed:**
- `togglingId` state (`useState<string | null>(null)`).
- `handleToggleStatus(user)` async function (15 lines of toggle-status PATCH logic).
- Old single-button actions cell with Activar/Desactivar conditional.

**Added:**
- `Pencil` + `Trash2` imports alongside existing `Loader2, Plus` from `lucide-react`.
- `EditUserDialog` + `DeleteUserAlertDialog` named imports from co-located files.
- `editTarget` + `deleteTarget` state (both `UserRow | null`).
- New actions cell layout — `<div className='flex items-center justify-end gap-2'>` containing:
  - `Editar` button (always rendered, even on own row per D-08) with `<Pencil className='mr-1 size-3' />` leading icon.
  - On own row: `<span className='text-muted-foreground text-xs'>(tu cuenta)</span>` (D-09 hides Borrar).
  - On other rows: `Borrar` button (`variant='outline' className='text-destructive hover:text-destructive'`) with `<Trash2 className='mr-1 size-3' />` leading icon.
- Two dialog mounts after the existing Create dialog, both with:
  - `open={target !== null}` / `onOpenChange={(open) => { if (!open) setTarget(null); }}`
  - `onSuccess={() => { setTarget(null); router.refresh(); }}`
- `EditUserDialog` receives `roles` (passed through from props) and `isOwnRow={editTarget?.id === currentUserId}` — boolean coercion is automatic via `===`.

### Backend — `nemea-back/src/users/users.controller.spec.ts` (MODIFIED)

**Removed:**
- Entire `describe('PATCH /users/:id/toggle-status', ...)` block (3 it() cases — 65 lines).
- `deactivate: jest.fn()` and `activate: jest.fn()` from `mockUsersService`.

**Added:**
- `BadRequestException` to imports from `@nestjs/common`.
- `update: jest.fn()` and `remove: jest.fn()` to `mockUsersService`.
- `describe('PATCH /users/:id', ...)` with 3 it() cases:
  1. happy path — forwards `(id, dto, callerId)` to `service.update`, returns updated user.
  2. propagates `BadRequestException('No puedes cambiar tu propio rol')` from service (self-role-edit guard).
  3. propagates `NotFoundException('Usuario no encontrado')` from service.
- `describe('DELETE /users/:id', ...)` with 4 it() cases:
  1. happy path — forwards `(id, callerId)` to `service.remove`, returns `undefined`.
  2. propagates `BadRequestException('No puedes borrarte a vos mismo')` from service.
  3. propagates `BadRequestException('No se puede dejar el sistema sin administradores activos')` from service.
  4. propagates `NotFoundException('Usuario no encontrado')` from service.

**Test count delta:** 7 tests before (1 defined + 1 GET + 2 POST + 3 toggle-status) → 11 tests after (1 defined + 1 GET + 2 POST + 3 PATCH + 4 DELETE). Net **+4 tests**, with 3 toggle-status deleted and 7 new added.

## Decisions Made

- **Loader2 import retained in UsersClient.tsx** — still used in the create-user submit button (existing line 281). The plan flagged this as "may be unused" but a quick grep confirmed it stays in use after the toggle button removal.
- **Boolean coercion via `===` is sufficient for `isOwnRow`** — `editTarget?.id === currentUserId` evaluates to `false` when `editTarget` is `null` (and the right side is `undefined`/UUID, both falsy compare). No `!!` wrapper needed; matches the plan's note.
- **No guard tests in controller spec** — per D-04 the guards live in `UsersService`; controller spec only verifies (a) shape of `service.X(...)` call and (b) propagation of exceptions thrown by the mocked service. This avoids duplicating the guard coverage already in Plan 05's `users.service.spec.ts` (which has 24/24 passing tests).
- **Valid v4-style UUID literals in controller tests** — `VICTIM_ID`/`CALLER_ID`/`DIFFERENT_ROLE_ID` constants at top of each describe block. Even though `ParseUUIDPipe` doesn't run in the unit test path (the controller method receives strings directly), using real UUID shapes mirrors the Plan 03/05 spec patterns and reads as production-realistic data.
