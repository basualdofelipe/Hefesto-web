# Phase 12.4: User management — edit user, delete user, admin self-lockout — Specification

**Created:** 2026-05-18
**Ambiguity score:** 0.175 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

Admin can edit any user's name, role, and active status via a dialog on `/usuarios`, and hard-delete a user (transferring their scenarios to the deleting admin with a name suffix marking provenance), with self-lockout guards that prevent the system from ever ending up without an active admin.

## Background

Phase 12.1 introduced dynamic roles, but `/usuarios` only exposes "create user" and "toggle active/inactive". There is no flow to change a user's role once assigned, and there is no flow to hard-delete a user.

This gap was surfaced by Phase 12.1 UAT: a custom role cannot be deleted because `RolesService.remove()` blocks deletion of any role with users assigned (`nemea-back/src/roles/roles.service.ts:85`), and there is no UI to reassign those users to a different role.

Current state in code:
- `UsersController` (`nemea-back/src/users/users.controller.ts`) exposes GET `/`, POST `/`, PATCH `/:id/toggle-status` — no PATCH for editing user fields, no DELETE.
- `UsersService.remove()` (`nemea-back/src/users/users.service.ts:76`) exists but no route exposes it.
- `UsersClient.tsx` table has Edit-role column gap: only "Activar/Desactivar" button + "(tu cuenta)" guard.
- Self-lockout guards today: PATCH toggle-status rejects self-deactivation (`users.controller.ts:86`); `RolesService.update` rejects stripping `canManageUsers` from the ADMIN role (`roles.service.ts:59`). No guard prevents an admin from changing their OWN role.
- `scenarios.user_id` is FK to users with `ON DELETE CASCADE` (`1772900000000-CreateScenariosTables.ts:24`) — currently a hard delete would silently destroy all of the user's scenarios.

## Requirements

1. **Edit-user endpoint**: Backend exposes PATCH `/users/:id` accepting `{ name?, roleId?, isActive? }`.
   - Current: only PATCH `/users/:id/toggle-status` exists; no way to change name or role from outside `create()`.
   - Target: `PATCH /users/:id` with `UpdateUserDto` (all fields optional, validated). Requires `can_manage_users`. Returns the updated user (with eager-loaded role). 404 if user not found. 404 if `roleId` provided but role does not exist.
   - Acceptance: PATCH `/users/<uuid>` with `{ name: "X" }` returns 200 + updated user; with `{ roleId: <valid-uuid> }` returns 200 + user with new role; with `{ isActive: false }` returns 200 + deactivated user; unknown `roleId` returns 404.

2. **Edit-user UI**: Replace the "Activar/Desactivar" button with an "Editar" button that opens a Dialog editing name + role + active status in one form.
   - Current: only inline "Activar/Desactivar" button; role is not editable; name is not editable.
   - Target: "Editar" button per row opens `<EditUserDialog>` with Inputs for name + Select for role (loaded from `/api/roles` like create flow) + Switch/Checkbox for active. Email is shown read-only. Submitting calls PATCH `/users/:id` and refreshes the table.
   - Acceptance: clicking Editar on any other user's row opens the Dialog pre-filled with current values; submitting after changing role updates the badge in the table; submitting after toggling active flips the row's status badge.

3. **Delete-user endpoint**: Backend exposes DELETE `/users/:id` that transfers scenarios then hard-deletes the user.
   - Current: `UsersService.remove()` exists but no controller route; DB has ON DELETE CASCADE so naive deletion would wipe scenarios.
   - Target: `DELETE /users/:id` requires `can_manage_users`. In a single DB transaction: (a) `UPDATE scenarios SET user_id = <caller.id>, name = scenarios.name || ' - ' || COALESCE(victim.name, 'usuario borrado') WHERE user_id = :id`, then (b) `DELETE FROM users WHERE id = :id`. Returns 204. 404 if user not found.
   - Acceptance: DELETE on a user with 2 scenarios named "Promo" and "Black Friday" reassigns both to the caller and renames them to "Promo - {victim.name}" / "Black Friday - {victim.name}" (or " - usuario borrado" when `victim.name IS NULL`); the user row no longer exists; the transaction is atomic (if scenario rename fails, user is not deleted).

4. **Delete-user UI with preview**: "Borrar" button opens an `AlertDialog` that previews exactly what will happen.
   - Current: no delete button anywhere.
   - Target: `<DeleteUserAlertDialog>` shows the victim's email + name + a sentence "Sus N escenarios serán transferidos a tu cuenta con el sufijo ' - {name}'." (where N is fetched or pre-computed via the existing user payload's scenario count, OR via a count query before opening). Destructive button "Borrar usuario" calls DELETE `/users/:id` and shows toast on success.
   - Acceptance: clicking Borrar on a user who owns ≥1 scenario shows the count and the suffix string; clicking on a user with 0 scenarios shows "Sus 0 escenarios serán transferidos…" (or simpler phrasing — the dialog still confirms); confirming triggers DELETE; canceling does nothing.

5. **Self-lockout — cannot edit own role**: Backend rejects PATCH that would change the caller's own role.
   - Current: no such guard. An admin could PATCH themselves into a roleless or non-admin role and lose `can_manage_users`.
   - Target: if `req.id === param.id` AND `dto.roleId` is provided AND `dto.roleId !== victim.currentRoleId`, throw `BadRequestException('No puedes cambiar tu propio rol')`. Editing your own `name` and `isActive` is also blocked for `isActive`, but editing own `name` is allowed.
   - Acceptance: admin calling PATCH `/users/<own-id>` with `{ roleId: <different> }` → 400; with `{ name: "New name" }` → 200; with `{ isActive: false }` → 400 with "No puedes desactivar tu propia cuenta" (existing message reused/extended).

6. **Self-lockout — cannot delete self**: Backend rejects DELETE on the caller's own id.
   - Current: no DELETE route exists; rule does not exist.
   - Target: if `req.id === param.id`, throw `BadRequestException('No puedes borrarte a vos mismo')` before any DB work.
   - Acceptance: DELETE `/users/<own-id>` returns 400 with message; victim user is not deleted; no scenarios are transferred.

7. **Self-lockout — cannot leave system without active admin**: Backend rejects any change that would leave 0 active users with `can_manage_users = true`.
   - Current: no such guard. An admin could deactivate or role-demote the only OTHER admin and then lock themselves out via a separate session.
   - Target: before persisting any PATCH or DELETE that would (a) deactivate a user with `can_manage_users`, (b) move a user with `can_manage_users` to a role without it, or (c) delete a user with `can_manage_users`, count post-change active users with `can_manage_users`. If count would be 0, throw `BadRequestException('No se puede dejar el sistema sin administradores activos')`. The count uses `users.isActive = true AND role.can_manage_users = true`.
   - Acceptance: with exactly two admins A and B, both active: PATCH `B` to deactivate → 200; PATCH `B` to role-without-can_manage_users → 200; then PATCH `A` (current state: only `A` is admin) to deactivate → 400; DELETE `A` → 400; PATCH `A` with role-without-can_manage_users → 400.

## Boundaries

**In scope:**
- New backend endpoints: PATCH `/users/:id`, DELETE `/users/:id`
- New backend DTO: `UpdateUserDto` (name?, roleId?, isActive?)
- New backend logic: scenario transfer with name-suffix rename inside a transaction (`UsersService.remove(victimId, callerId)` signature change)
- New backend guards: self-role-edit, self-delete, last-active-admin
- Frontend `<EditUserDialog>` replacing the toggle-status button (toggle becomes part of the Dialog)
- Frontend `<DeleteUserAlertDialog>` with preview
- Updated `UsersClient.tsx` table actions column: Editar + Borrar buttons (Editar disabled or hidden on own row only for role; Borrar disabled on own row)
- Permission gating: both endpoints require `can_manage_users`
- Unit/spec tests for new endpoints and guards (extending `users.controller.spec.ts` / `users.service.spec.ts`)

**Out of scope:**
- Email change — email is the OAuth identity / whitelist key; changing it would break Google sign-in matching
- pictureUrl / googleId edits — sourced from Google, overwritten on next sign-in (`updateGoogleProfile`)
- Audit log of who-deleted-whom — no audit infrastructure exists project-wide
- Soft delete on users — the user explicitly chose hard delete (`delete-user.md` todo says "hard delete está bien")
- Restoring deleted users / undo — out of scope; requires soft delete to be meaningful
- Editing roles or role permissions — Phase 12.1 already covers `/roles` page
- Bulk operations (delete multiple, change role for many) — not requested
- New `original_owner_email` column on scenarios — the user chose the simpler name-suffix approach over adding a column
- UX for the deleted-user's scenarios beyond the rename — admin manually edits/deletes from `/escenarios` if desired (the rename is the only provenance hook)
- Migration to clean up the existing CASCADE — the FK stays as ON DELETE CASCADE; the transactional `UPDATE scenarios SET user_id = caller` runs BEFORE the DELETE, so cascade never fires in the new flow (defense in depth — if anything bypasses the service path, cascade is still a safety net)

## Constraints

- The scenario-transfer + user-delete operation MUST run inside a single TypeORM transaction (`QueryRunner` or `@Transaction`). If the UPDATE fails (e.g., name length overflow on a 200-char column), the DELETE must NOT execute. Pattern already used in `scenarios.service.ts` full-replace strategy.
- The "name" column on scenarios is `varchar(200)`. The suffix " - {name}" can push the result over 200 chars. The implementation MUST guard against overflow: either truncate the rename result to 200 chars before UPDATE, or reject the delete with a clear message if any of the user's scenarios would exceed the column limit. (The simpler choice — truncate — is acceptable; provenance is informational, not load-bearing.)
- The "last active admin" check counts users where `isActive = true AND role.can_manage_users = true`. The query must be a `SELECT COUNT(*)` after the WRITE in the same transaction, OR the check must be performed BEFORE the write using a calculated "after" state. Either is acceptable as long as it is inside the transaction.
- Frontend buttons on the caller's own row: "Editar" stays visible but the role Select is disabled when editing own user; "Borrar" is hidden or disabled with tooltip "No puedes borrarte a vos mismo".
- All new code follows project conventions: TS strict, no `any`, explicit return types, single quotes, `@RequirePermission('can_manage_users')` on every new endpoint.
- Backend Spanish error messages match existing style (`No puedes desactivar tu propia cuenta`, etc.).

## Acceptance Criteria

- [ ] PATCH `/users/:id` exists with `UpdateUserDto`, requires `can_manage_users`, returns 200 + updated user
- [ ] PATCH with unknown `roleId` returns 404 ("Rol no encontrado")
- [ ] PATCH on own id with `{ roleId }` (different from current) returns 400 ("No puedes cambiar tu propio rol")
- [ ] PATCH on own id with `{ name }` only returns 200
- [ ] PATCH that would deactivate or demote the last active admin returns 400 ("No se puede dejar el sistema sin administradores activos")
- [ ] DELETE `/users/:id` exists, requires `can_manage_users`, returns 204
- [ ] DELETE on own id returns 400 ("No puedes borrarte a vos mismo")
- [ ] DELETE that would leave 0 active admins returns 400
- [ ] DELETE transfers ALL scenarios of the victim to the caller in a transaction
- [ ] Transferred scenarios are renamed to `<old name> - <victim.name>` (or `<old name> - usuario borrado` when victim.name is null)
- [ ] Scenario rename result is truncated to ≤200 chars when needed
- [ ] If scenario rename fails inside the transaction, user is NOT deleted
- [ ] `/usuarios` page shows "Editar" button per row that opens `<EditUserDialog>`
- [ ] EditUserDialog has Name input, Role select, Active toggle; Email shown read-only
- [ ] EditUserDialog on own row disables the Role select and the Active toggle
- [ ] `/usuarios` page shows "Borrar" button per row (hidden/disabled on own row)
- [ ] Clicking Borrar opens AlertDialog showing victim email + scenario count + rename preview string
- [ ] Confirming Borrar calls DELETE and refreshes the table; canceling does nothing
- [ ] `users.controller.spec.ts` covers: PATCH happy path, PATCH self-role-edit rejection, PATCH last-admin rejection, DELETE happy path, DELETE self rejection, DELETE last-admin rejection
- [ ] `users.service.spec.ts` covers the transactional rename+delete and the rollback on rename failure
