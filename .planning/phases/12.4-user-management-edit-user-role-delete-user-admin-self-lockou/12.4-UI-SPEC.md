## Color

Project uses 60/30/10 via the existing OKLCH design tokens declared in `globals.css`. This phase introduces no new colors.

| Role | Token | Where in this phase |
|------|-------|---------------------|
| Dominant (60%) | `--background` (`oklch(0.99 0.007 68)` light / `oklch(0.198 0.011 39)` dark) | Page background, table rows, Dialog/AlertDialog `bg-background` |
| Secondary (30%) | `--card`, `--secondary`, `--muted` | Disabled fields shown via `disabled` attribute → opacity 0.5 (default), NOT a new bg. Tooltip uses `bg-foreground text-background` (inverted — existing primitive default) |
| Accent (10%) | `--primary` (`oklch(0.462 0.137 32)` — burnt sienna/leather brown) | Primary actions only: `Guardar` button (`variant='default'`), `Crear usuario` button (existing) |
| Destructive | `--destructive` (`oklch(0.577 0.215 27)` red) | "Borrar" button in actions column (`variant='destructive'` with leading `Trash2` icon at `text-destructive` color) AND `AlertDialogAction` confirm button (matches existing `ScenarioListClient.tsx:184-189` pattern: `className='bg-destructive text-destructive-foreground hover:bg-destructive/90'`) |

**Accent reserved for:**
1. Primary form submit (`Guardar` button — `variant='default'` = `bg-primary`)
2. (No other accent uses in this phase — table badges already use `default`/`secondary`/`outline`/`destructive` from existing UsersClient, unchanged)

**Destructive reserved for:**
1. `Borrar` button in actions column (table row)
2. `AlertDialogAction` "Borrar usuario" inside DeleteUserAlertDialog
3. Inline `text-destructive` for zod field errors (existing pattern from UsersClient form)

**Read-only Email field visual treatment:**
- `<Input>` with `readOnly` (NOT `disabled`) + `className='bg-muted text-muted-foreground cursor-not-allowed'`
- Rationale: `disabled` removes focusability and reduces opacity globally; `readOnly` preserves the field as part of the form, keeps tab order, communicates "this exists but cannot be edited", and matches accessibility best practice for non-mutable form fields. Visual cue is the `bg-muted` (`var(--muted)`) + muted foreground combo.

**Disabled vs read-only (own-row controls per D-08):**
- Role `<Select>` → `disabled={isOwnRow || isSubmitting}` — Shadcn Select with `disabled` is the canonical way; renders at 50% opacity per existing button.tsx CVA pattern.
- Active `<Switch>` → `disabled={isOwnRow || isSubmitting}` — radix Switch supports `disabled` (`cursor-not-allowed` + opacity 0.5 from line 20 of switch.tsx).

## Interaction Contracts

### EditUserDialog — Focus & Keyboard

| Event | Behavior |
|-------|----------|
| Open dialog | Radix auto-focus moves to first focusable interactive element. Email is `readOnly` (NOT in tab cycle as a focus target the way a button is, but still tabbable). Tab order: `Email (readOnly) → Name (input) → Role (Select trigger) → Active (Switch) → Cancelar → Guardar → Close X`. **Initial focus target:** explicit `autoFocus` on the Name `<Input>` (first editable field). |
| Tab / Shift+Tab | Cycles within dialog (Radix focus trap default). Does NOT escape to page. |
| `Enter` in any input | Submits the form (RHF `handleSubmit` wired to `<form onSubmit>` — same pattern as UsersClient.tsx:216). |
| `Escape` | Closes dialog (Radix Dialog default). Equivalent to clicking Cancelar. |
| Click overlay | Closes dialog (Radix Dialog default — Dialog is NOT AlertDialog). |
| Click Cancelar | Calls `onOpenChange(false)`. No PATCH fired. Form state cleared on next open via `reset()` on `useEffect([user])`. |
| Click Guardar | Calls RHF `handleSubmit(onSubmit)`. PATCH fired. See State Matrix above. |

### DeleteUserAlertDialog — Focus & Keyboard

| Event | Behavior |
|-------|----------|
| Open dialog | Radix AlertDialog auto-focuses `AlertDialogCancel` (the safe default — verified in Radix docs and existing ScenarioListClient/RolesClient behavior). |
| Tab / Shift+Tab | Cycles between Cancelar and Borrar usuario (focus trap). |
| `Enter` on Cancelar | Closes (no DELETE). |
| `Enter` on Borrar usuario | Fires DELETE. |
| `Escape` | Closes dialog. Equivalent to Cancelar. |
| Click overlay | **Does NOT close** (AlertDialog modal-blocking — Radix default). User MUST click a button. |

### UsersClient — Click

| Event | Behavior |
|-------|----------|
| Click `Editar` (any row) | `setEditTarget(user)` → opens EditUserDialog. |
| Click `Borrar` (other-user row) | `setDeleteTarget(user)` → opens DeleteUserAlertDialog. |
| Click `Editar` (own row) | Same as other — opens EditUserDialog. Dialog renders own-row UX (Role/Active disabled). |
| Hover disabled Role Select / disabled Switch (own row) | Tooltip opens with delay 0ms (project default in `tooltip.tsx:9`). |

## Implementation Notes for Executor

### Switch + RHF wiring (P12 pitfall mitigation)

`<Switch>` uses `onCheckedChange`, not `onChange`. Wire via watch/setValue (matches existing pattern in `RolesClient.tsx:459-466` and `UsersClient.tsx:252-256` for Select):

```tsx
<Switch
  id='user-active'
  checked={watch('isActive')}
  onCheckedChange={(value) => setValue('isActive', value, { shouldDirty: true })}
  disabled={isOwnRow || isSubmitting}
/>
```

### Tooltip wrapping a disabled control

Radix Tooltip Trigger does NOT fire on a `disabled` button (browser does not emit pointer events). Wrap the disabled control in a `<span>` (and `<span>` becomes the trigger), OR use Radix `<Tooltip.Trigger asChild>` on a wrapper. Existing project has no precedent for this — adopt the wrapper `<span tabIndex={0}>` pattern:

```tsx
<TooltipProvider>
  <Tooltip>
    <TooltipTrigger asChild>
      <span tabIndex={0}>
        <Switch ... disabled />
      </span>
    </TooltipTrigger>
    <TooltipContent>No puedes cambiar tu propio estado</TooltipContent>
  </Tooltip>
</TooltipProvider>
```

The `<TooltipProvider>` wrapper can sit at the dialog content level (not the global app level — this phase scope only).

### Form reset on open

When `editTarget` changes (re-open with different user), call `reset({ name, roleId, isActive })` inside `useEffect([editTarget])` to repopulate. Matches the pattern in `RolesClient.tsx:173-191` (`openEditDialog`).

### Submission shape

```ts
const dto: Record<string, unknown> = {};
if (data.name !== user.name) dto.name = data.name;
if (data.roleId !== user.role.id) dto.roleId = data.roleId;
if (data.isActive !== user.isActive) dto.isActive = data.isActive;

// Only send changed fields. Backend treats omitted fields as no-change (UpdateUserDto all-optional per Finding 4 in RESEARCH.md).
await apiClientFetch(`/api/users/${user.id}`, token, {
  method: 'PATCH',
  body: JSON.stringify(dto),
});
```

If `dto` ends up empty (no changes), short-circuit with `toast.info('Sin cambios')` or just close — implementer's call. Recommendation: still call PATCH with empty body so backend can validate auth; backend will return 200 + unchanged user. Avoid the extra UI branch.

### Zod schema for edit form

```ts
const editUserSchema = z.object({
  name: z.string().max(255).optional().or(z.literal('')),
  roleId: z.string().uuid('Selecciona un rol'),
  isActive: z.boolean(),
});
```

Name is optional + accepts empty string. Role MUST be a UUID (server validates existence). Active is required boolean.
