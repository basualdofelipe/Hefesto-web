## Integration Note for Plan 06

Plan 06 (UsersClient integration) will:
- Import `DeleteUserAlertDialog` from `'./DeleteUserAlertDialog'`
- Manage `deleteTarget` state (`UserRow | null`)
- **Crucially**: hide the row-level Borrar button when `user.id === currentUserId` (D-09) — this dialog assumes the `user` prop never corresponds to the caller (defense-in-depth: backend also has self-delete guard on `UsersService.remove`)
- Pass `onSuccess={() => { setDeleteTarget(null); router.refresh(); }}` to clear the target and re-fetch the RSC tree

The dialog is fully decoupled — no shared state with the parent, no internal `currentUserId` check, no router.refresh inside the component.

## Decisions Made
- **className override over variant='destructive'** — RolesClient and ScenarioListClient both use this pattern. AlertDialogAction wraps a Shadcn `<Button>` via `asChild`; both styling approaches work, but the project convention is the className override for destructive AlertDialog confirms.
- **Dialog stays open on error** — gives user a chance to read the backend Spanish message and either cancel or retry. Closing-on-error would force a re-open + re-render and lose context.
- **`fireEvent.click` for the action button in tests** — same Radix `body { pointer-events: none }` workaround documented in Plan 03 SUMMARY. `userEvent.click` is still used for Cancelar (no submission gate).
- **Title "Eliminar usuario" vs button "Borrar usuario"** — UI-SPEC mandates this intentional verb mismatch: title follows project-wide "Eliminar X" pattern (RolesClient, ScenarioListClient); button matches the row trigger "Borrar".
