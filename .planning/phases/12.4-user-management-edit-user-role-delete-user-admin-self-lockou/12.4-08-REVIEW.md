# Phase 12.4-08: Code Review Report (Gap Closure)

**Reviewed:** 2026-05-26
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Gap-closure review of the two UAT blocker fixes plus the new real-Postgres e2e
spec. Both blocker fixes are individually correct:

- **BLOCKER 1** — Dropping `.setLock('pessimistic_write')` from
  `countOtherActiveAdmins` is the right call: PostgreSQL genuinely rejects
  `FOR UPDATE` with aggregates (`SELECT COUNT(...) ... FOR UPDATE` →
  `25006`/`0A000` class error). The COUNT itself was the illegal part, not the
  lock placement, so removing the lock fixes the crash.
- **BLOCKER 2** — `updated_at` → `updatedAt` in `.set()` is correct: TypeORM's
  `UpdateQueryBuilder.set()` resolves keys by **entity property name**, and
  `BaseEntity` declares `@UpdateDateColumn({ name: 'updated_at' })` as property
  `updatedAt`. The DB column is `updated_at`, the property is `updatedAt`.

However, the **concurrency-safety reasoning behind BLOCKER 1 has a real gap**:
once the row lock is removed from the aggregate, the anti-race guarantee leans
entirely on SERIALIZABLE isolation — and the codebase has **no handling for the
`serialization_failure` (40001) that SERIALIZABLE will raise** under exactly the
concurrent demote/delete scenario the guard is meant to protect. That aborted
transaction surfaces as an opaque HTTP 500 through the global filter. This is
the headline finding (WR-01).

Secondary findings concern e2e assertion strength (two tests would still pass if
a fix were reverted, defeating their stated purpose) and shared-DB teardown
robustness.

## Warnings

### WR-01: SERIALIZABLE `serialization_failure` (40001) surfaces as an ungraceful HTTP 500

**File:** `nemea-back/src/users/users.service.ts:228-233` (and `:304-309`)
**Issue:**
The BLOCKER 1 fix removes the pessimistic row lock and explicitly delegates the
anti-race guarantee to the enclosing SERIALIZABLE transaction (see comment at
`:98-106`). That is sound in principle — but SERIALIZABLE achieves serialization
by *aborting* one of two conflicting transactions with SQLSTATE `40001`
(`serialization_failure`), not by blocking. When two admins concurrently demote
or delete the penultimate admin, the loser's transaction is rolled back by
Postgres and the driver throws a `QueryFailedError`.

The `catch` block here only does `rollbackTransaction()` + `throw error` — it
re-throws the raw `QueryFailedError`. That is **not** an `HttpException`, so
`HttpExceptionFilter` (`nemea-back/src/common/filters/http-exception.filter.ts:42-51`)
maps it to a generic `500 Internal Server Error`. So the exact concurrency case
the guard exists to handle produces a confusing 500 for a user who did nothing
wrong, instead of either succeeding on retry or returning a clean 4xx.

There is no retry loop and no 40001→409 mapping anywhere in the service or the
filter. The fix comment claims SERIALIZABLE "already serializes concurrent
demote/deactivate/delete requests ... no pessimistic_write lock on the COUNT is
needed to guarantee safety" — the *data-integrity* guarantee holds, but the
*graceful-degradation* guarantee does not.

Note: the victim re-fetch (`:144-149`, `:248-253`) still uses
`pessimistic_write` on a non-aggregate `getOne()`, which is legal and does
serialize on the victim row. But the last-admin COUNT reads *other* admin rows
that are not covered by that single-row lock, so SERIALIZABLE remains the only
backstop for the COUNT — and 40001 remains reachable.

**Fix:** Detect the serialization failure and map it to a retry or a clean
status code. Minimal version (mirrors the 23505 handling already in `create()`):
```ts
} catch (error) {
  await queryRunner.rollbackTransaction();
  if (
    error instanceof QueryFailedError &&
    (error as QueryFailedError & { code?: string }).code === '40001'
  ) {
    throw new ConflictException(
      'Operación concurrente detectada, reintentá la acción',
    );
  }
  throw error;
} finally {
  await queryRunner.release();
}
```
A bounded retry loop (e.g. 3 attempts on 40001) is the more user-friendly option
if silent success on retry is acceptable. Either way, do not let a raw 40001
escape as a 500.

### WR-02: e2e Test 1 assertion is a no-op and would not catch a reverted BLOCKER 1 fix

**File:** `nemea-back/test/users-admin-guard.e2e-spec.ts:173-179`
**Issue:**
```ts
await expect(
  usersService.update(secondAdmin.id, { roleId: editorRole.id }, seedAdmin.id),
).resolves.not.toThrow();
```
`.resolves` already unwraps the promise; chaining `.not.toThrow()` on the
resolved value is a Jest anti-pattern. `.toThrow()` is a matcher for *functions*,
not resolved values, so `resolves.not.toThrow()` does not assert "did not
reject" in a meaningful way — if the promise *rejects*, the `.resolves` chain
fails with a generic unwrap error, but the matcher itself adds nothing and the
intent ("resolved without throwing") is not actually expressed. More importantly,
the test makes **no positive assertion** about the outcome (no check that
`secondAdmin` was actually demoted, no role assertion). The spec's stated purpose
(line 30-32, 155-160) is "pre-fix → 500; post-fix → resolves OK", but if the fix
were reverted to `FOR UPDATE + COUNT`, the rejection would still need to be
surfaced by the `.resolves` chain — fragile and indirect.

**Fix:** Make a positive, behavior-anchored assertion:
```ts
const updated = await usersService.update(
  secondAdmin.id,
  { roleId: editorRole.id },
  seedAdmin.id,
);
expect(updated.role.id).toBe(editorRole.id);
expect(updated.role.canManageUsers).toBe(false);
```
Reverting the fix then throws a real `QueryFailedError` here and the test fails
loudly, which is the whole point of the e2e spec.

### WR-03: e2e Test 3 `updatedAt` assertion uses `>=`, so reverting the NOW() bump still passes

**File:** `nemea-back/test/users-admin-guard.e2e-spec.ts:282-284`
**Issue:**
```ts
expect(transferred!.updatedAt.getTime()).toBeGreaterThanOrEqual(
  originalUpdatedAt.getTime(),
);
```
The test's own header (line 38) claims it verifies "updatedAt bumped". But `>=`
is satisfied even when `updatedAt` is *unchanged* (equal). If a future regression
dropped the `updatedAt: () => 'NOW()'` line from `transferOwnership` entirely,
the transferred row would keep its original `updatedAt` and this assertion would
still pass — the test does not actually defend the behavior it documents. The
10ms `setTimeout` (line 252) reduces but does not eliminate this (a save that
does not touch `updatedAt` leaves it at `originalUpdatedAt`, so equality holds
and `>=` passes).

**Fix:** Assert a strict increase. With the 10ms pre-delay and `timestamptz`
microsecond resolution, `>` is safe:
```ts
expect(transferred!.updatedAt.getTime()).toBeGreaterThan(
  originalUpdatedAt.getTime(),
);
```
This makes the test fail if the `NOW()` bump is removed, fulfilling its stated
guarantee.

### WR-04: e2e teardown leaves the seed admin deactivated if the suite crashes mid-test

**File:** `nemea-back/test/users-admin-guard.e2e-spec.ts:108-122, 194-197`
**Issue:**
Test 2 mutates **shared, migration-seeded** state: it sets the seed admin
(`basualdofelipe@gmail.com`) to `is_active = false` (line 194-197) and relies on
`afterEach` (line 117-121) to restore it. `afterEach` runs even on test failure,
so the normal path is covered. But if the *process* dies between the
deactivation and `afterEach` (uncaught error in the suite, timeout kill, CI
SIGTERM, a throw inside `beforeAll` of a later run), the shared test DB is left
with the real admin deactivated. Because this is a long-lived shared DB on port
5433 (not an ephemeral container per run, per the file header), the next run's
`beforeAll` resolves `adminRole`/`editorRole` fine but any test assuming the seed
admin is active starts from a corrupted baseline. There is no `beforeEach`/
`beforeAll` reconciliation that *re-activates* the seed admin before a run; the
restore is only in `afterEach`.

**Fix:** Add a defensive reactivation in `beforeEach` (or at the top of
`beforeAll`) so every run starts from a known-good baseline regardless of how the
previous run terminated:
```ts
beforeEach(async () => {
  await dataSource.query(
    `UPDATE users SET is_active = true WHERE email = $1`,
    [SEED_ADMIN_EMAIL],
  );
});
```
This makes the suite self-healing against a crashed predecessor run.

### WR-05: e2e Test 2 issues the same failing `update()` twice, the second time after `afterEach` would not have run

**File:** `nemea-back/test/users-admin-guard.e2e-spec.ts:223-232`
**Issue:**
The test calls `usersService.update(lastAdmin.id, { isActive: false }, callerEditor.id)`
twice in a row (once asserting `BadRequestException`, once asserting the message
string). Both calls happen inside the *same* test, so `afterEach` has not run
between them — the seed admin is still deactivated and `lastAdmin` still the sole
active admin, so the second call does re-fire the guard correctly. That part is
fine. The smell is the duplication: two full transactional round-trips against a
real DB to assert two facets of one rejection. If the first call's rejection
were ever to mutate state (it does not today, since the guard throws before any
write — but `remove()`/`update()` open a SERIALIZABLE transaction and roll back),
the second assertion would be testing a subtly different precondition. Prefer a
single call captured once.

**Fix:** Assert both facets against a single invocation:
```ts
await expect(
  usersService.update(lastAdmin.id, { isActive: false }, callerEditor.id),
).rejects.toThrow(
  new BadRequestException(
    'No se puede dejar el sistema sin administradores activos',
  ),
);
```
`rejects.toThrow(new BadRequestException(msg))` checks both the type and the
message in one round-trip.

### IN-01: Misleading / self-contradicting comment in e2e Test 1

**File:** `nemea-back/test/users-admin-guard.e2e-spec.ts:162-170`
**Issue:** The comment block thinks out loud and contradicts itself ("we use
secondAdmin as the caller ... but actually we want to demote secondAdmin"),
while the code uses `seedAdmin.id` as caller and `secondAdmin.id` as victim. The
narration does not match the code and is confusing to a future reader.
**Fix:** Replace with a one-line statement of intent: "Demote the test admin
(victim) with the seed admin as caller; both are active so the last-admin guard
must NOT fire — this exercises countOtherActiveAdmins against real SQL."

### IN-02: `countOtherActiveAdmins` JSDoc overstates the safety guarantee

**File:** `nemea-back/src/users/users.service.ts:98-106`
**Issue:** The comment asserts SERIALIZABLE "already serializes concurrent
demote/deactivate/delete requests ... no pessimistic_write lock on the COUNT is
needed to guarantee safety." This is true for *correctness* but omits that
SERIALIZABLE enforces it by *aborting* a transaction with 40001 (see WR-01),
which the code does not handle gracefully. A reader trusting this comment would
not anticipate the 500.
**Fix:** Append a sentence: "Note: SERIALIZABLE enforces this by raising
serialization_failure (40001) on the losing txn; callers must map/retry that —
see error handling in update()/remove()."

### IN-03: Repeated queryRunner boilerplate across `update()` and `remove()`

**File:** `nemea-back/src/users/users.service.ts:138-143, 242-247`
**Issue:** The connect → startTransaction('SERIALIZABLE') → try/catch-rollback/
finally-release scaffolding is duplicated verbatim in `update()` and `remove()`
(and a non-SERIALIZABLE variant lives in `scenarios.service.ts:221-249`). When
WR-01's 40001 handling is added, it will need to be copied into both blocks,
inviting drift. Consider a small `runInSerializableTxn<T>(fn)` helper that owns
the lifecycle (and the 40001 mapping/retry) in one place.
**Fix:** Extract a private helper that takes a callback receiving the
`EntityManager`/`QueryRunner` and centralizes connect/transaction/rollback/
release plus serialization-failure handling.
