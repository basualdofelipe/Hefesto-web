# Phase 12.4 Plan 02: ScenariosService.transferOwnership Summary

**One-liner:** Adds a cross-module bulk-UPDATE method that moves a victim's scenarios to a new owner (with a name-suffix marking provenance) inside a caller-owned transaction — wiring the back half of the user-delete cascade per D-01/D-02.

## What Was Built

### Service — `nemea-back/src/scenarios/scenarios.service.ts` (MODIFIED)

**Import extension:** `import { Not, QueryRunner, Repository } from 'typeorm';` (added `QueryRunner`).

**New method (placed at end of class, under section comment "Cross-module: transfer scenarios to a new owner"):**

```ts
async transferOwnership(
  victimId: string,
  newOwnerId: string,
  suffix: string,
  qr: QueryRunner,
): Promise<void> {
  await qr.manager
    .createQueryBuilder()
    .update(Scenario)
    .set({
      name: () => "LEFT(COALESCE(name, '') || :suffix, 200)",
      user: { id: newOwnerId },
    })
    .where('user_id = :victimId', { victimId })
    .setParameter('suffix', suffix)
    .execute();
}
```

**Key properties of the method:**

| Property | Value |
|----------|-------|
| Transaction lifecycle | **Caller-owned** — no `qr.connect`/`commit`/`rollback`/`release` calls in this method. Errors propagate so the caller (Plan 05 `UsersService.remove`) can rollback. |
| SQL injection safety | `setParameter('suffix', suffix)` binds via the TypeORM parameter binder. The arrow-function form `() => "LEFT(...)"` is the only TypeORM-supported way to mix raw SQL with parametrization (issue #1580). Plain `set({ name: ':suffix' })` would treat it as a literal string. |
| Truncation safety | `LEFT(COALESCE(name, '') || :suffix, 200)` keeps the result within the `varchar(200)` limit of `scenarios.name`. `COALESCE` defends against `NULL` even though `scenarios.name` is `nullable: false` in the schema. |
| FK update | `user: { id: newOwnerId }` — TypeORM extracts the `id` from the object literal to compose the `user_id` column write. Object-form coexists with the arrow-function `name` without tsc/runtime issue (fallback `user: () => ':newOwnerId'` not needed). |
| Bulk semantics | A single `UPDATE … WHERE user_id = :victimId` — touches `affected: N` rows atomically. No-op when `affected: 0`. |

## Decisions Made

- **Kept `user: { id: newOwnerId }` object-form** in `.set({...})` — the plan offered a fallback to `user: () => ':newOwnerId'` + extra `setParameter('newOwnerId', newOwnerId)` if TypeORM rejected the mix. Object-form works clean: tsc passes and the full QueryBuilder chain runs without runtime error in the mocked test. Fallback path stays documented in PLAN.md for future maintainers.
- **`beforeEach` inside the `transferOwnership` describe** resets `mockReturnThis()` after each test — because the outer suite's `jest.clearAllMocks()` wipes the return-value behavior on chain methods (turning them into `mockFn()` returning `undefined`). Without this reset, the second test would fail at the first chain hop. Documented inline.
- **Did NOT assert literal SQL string** in the spec (per plan + RESEARCH P) — assertion target is `name: expect.any(Function)`. Treating the SQL string as a contract would convert this into an implementation-detail test that breaks every time we reformat the raw SQL.

## Called By

This method is invoked by **Plan 05's `UsersService.remove`** inside its `QueryRunner` transaction (sequence: open tx → `transferOwnership(victim, caller, suffix, qr)` → `qr.manager.delete(User, victim)` → commit). The order is deliberate: the UPDATE moves scenario rows OFF `user_id = victim` before the DELETE looks for them, so the `ON DELETE CASCADE` constraint never fires (defense-in-depth net stays intact for direct-SQL bypass scenarios). See RESEARCH Finding 10.
