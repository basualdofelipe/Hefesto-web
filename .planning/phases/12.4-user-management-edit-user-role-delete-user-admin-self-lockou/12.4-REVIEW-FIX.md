### CR-A1: Last-active-admin guard TOCTOU + CR-A2: UsersService.update no usaba transacción

**Files modified:** `nemea-back/src/users/users.service.ts`, `nemea-back/src/users/users.service.spec.ts`, `nemea-back/src/users/dto/update-user.dto.ts`
**Commit:** `a44bc40`
**Applied fix:** Wrapped `UsersService.update()` and `UsersService.remove()` in `SERIALIZABLE` `QueryRunner` transactions with `pessimistic_write` locks on the victim row and on the admin rows the COUNT reads. Extracted a private `countOtherActiveAdmins(manager, excludeId)` helper that runs on `queryRunner.manager` so the read and the subsequent save/delete share the same transaction snapshot. This closes the TOCTOU window where two concurrent demote/deactivate/delete requests on the penultimate admin could both pass the last-admin guard and leave the system with zero active admins (REQ #7).

The commit bundles five refactor-coupled fixes that touch the same file regions:
- **WR-A1**: self-deactivate guard is now idempotent — `victim.isActive === true` precondition.
- **WR-A2**: explicit service guard rejects `dto.roleId === null` with a clear 400; DTO mirrors the `name` pattern with `@ValidateIf((_, v) => v !== null)`.
- **WR-A3**: `remove()` does findOne BEFORE the self-delete check, so a caller whose own row was concurrently deleted gets a 404 instead of "No puedes borrarte a vos mismo".
- **WR-A5**: `findActiveByEmail` "deactivated user" test now asserts the where clause includes `isActive: true` (previous version was a tautology).
- **WR-A6**: new rollback test for `update()` — if `save()` fails, the transaction rolls back so the intermediate role lookup never persists.
- **IN-A1**: only mutates `victim.role` when `roleId` actually changed (avoids spurious updated_at bumps).

### WR-A4: Race condition en UsersController.create — email uniqueness fuera de transacción

**Files modified:** `nemea-back/src/users/users.service.ts`, `nemea-back/src/users/users.controller.ts`, `nemea-back/src/users/users.service.spec.ts`, `nemea-back/src/users/users.controller.spec.ts`
**Commit:** `82157b0`

Pattern mirrors the existing `products.service.ts:149-159` convention. Added service tests for both `23505 -> 409` and `23503 (FK) -> rethrown as-is` paths; controller test now asserts no pre-check and propagation of `ConflictException` from the service.

### WR-A7: transferOwnership colisión por truncamiento + WR-A8: bulk UPDATE no bumpea updated_at

**Files modified:** `nemea-back/src/scenarios/scenarios.service.ts`, `nemea-back/src/scenarios/scenarios.service.spec.ts`
**Commit:** `1becac2`
**Applied fix:**
- **WR-A7**: `transferOwnership` now truncates the base name to `(200 - suffix.length)` BEFORE concatenating the suffix, so the discriminator (` - <name> #<shortId>`) is never cut by the final varchar(200) limit. Includes a defensive throw when `suffix.length >= 200`. New SQL: `LEFT(COALESCE(name, ''), ${baseMax}) || :suffix`.
- **WR-A8**: `.set()` now includes `updated_at: () => 'NOW()'` so the bulk UPDATE bumps the timestamp. TypeORM's `@UpdateDateColumn` only fires through `repository.save` / `manager.save` — a raw `QueryBuilder.update().execute()` skips entity subscribers. Without this, transferred scenarios kept their old `updatedAt` and stayed buried in the new owner's list (which sorts by `updatedAt DESC`), making the "we transferred this user's scenarios to you" flow invisible.

Tests: new assertion on `updated_at` key in `.set()`, new test verifying the raw SQL string emitted by the name-set arrow function, new defensive-throw test for suffix overflow.
