## Critical Issues

### CR-A1: Last-active-admin guard tiene ventana TOCTOU (race condition) — REQ #7 violado bajo concurrencia

**Files:**
- `nemea-back/src/users/users.service.ts:124-137` (`update`, last-admin guard)
- `nemea-back/src/users/users.service.ts:168-181` (`remove`, last-admin guard — antes de `createQueryRunner`)

**Issue:**

El guard que defiende la propiedad central de la fase ("no se puede dejar el sistema sin administradores activos", REQ #7) se implementa como una secuencia **lectura → decisión → escritura** que NO está protegida por ninguna transacción ni lock.

En `remove`:

```ts
// users.service.ts:168-181
if (victim.isActive && (victim.role?.canManageUsers ?? false)) {
  const otherActiveAdmins = await this.usersRepository
    .createQueryBuilder('u')
    // ... COUNT ...
    .getCount();
  if (otherActiveAdmins === 0) {
    throw new BadRequestException('No se puede dejar el sistema sin administradores activos');
  }
}

// LÍNEAS 186-216: recién acá empieza la transacción del DELETE,
// usando un QueryRunner separado, sin compartir snapshot ni lock con el COUNT.
const queryRunner = this.usersRepository.manager.connection.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction();
// ...
await queryRunner.manager.delete(User, id);
```

En `update` la situación es peor: el COUNT se hace sin transacción y el `save()` siguiente tampoco entra en una transacción (línea 153).

**Escenario reproducible con dos admins A y B (ambos activos):**

1. T0: Admin A abre `/usuarios` y clickea Borrar B → request R1 entra al backend.
2. T0+ε: Admin B abre `/usuarios` desde otra sesión y clickea Borrar A → request R2 entra al backend.
3. R1 ejecuta `COUNT(*) WHERE isActive AND canManageUsers AND id != B.id` → ve a A → 1.
4. R2 ejecuta `COUNT(*) WHERE isActive AND canManageUsers AND id != A.id` → ve a B → 1.
5. Ambos pasan el guard.
6. R1 commitea: borra a B.
7. R2 commitea: borra a A.
8. **Estado final: 0 admins activos. Sistema lockeado. Necesita reset manual de DB.**

El mismo problema aplica a:
- DELETE A + PATCH `B: isActive=false` concurrentes.
- PATCH `A: roleId=NON_ADMIN` + PATCH `B: roleId=NON_ADMIN` concurrentes.
- DELETE A + PATCH `B: roleId=NON_ADMIN` concurrentes.

El SPEC en `Constraints` (línea 90) **explícitamente requiere** que esta verificación se haga "inside the transaction":

> "The query must be a SELECT COUNT(*) after the WRITE in the same transaction, OR the check must be performed BEFORE the write using a calculated 'after' state. Either is acceptable as long as it is inside the transaction."

La implementación actual **viola el constraint del SPEC** y abre la ventana TOCTOU.

**Fix:**

Mover el COUNT dentro de la transacción y usar isolation level `SERIALIZABLE` o `REPEATABLE READ`, o (más simple) tomar un `SELECT ... FOR UPDATE` lock pesimista sobre las filas de admin antes del COUNT:

```ts
// remove(): mover el guard dentro del try del queryRunner
const queryRunner = this.usersRepository.manager.connection.createQueryRunner();
await queryRunner.connect();
await queryRunner.startTransaction('SERIALIZABLE'); // PG retry-on-conflict semantics

try {
  // Re-fetch victim inside the transaction with a row lock
  const victimLocked = await queryRunner.manager
    .createQueryBuilder(User, 'u')
    .innerJoinAndSelect('u.role', 'r')
    .setLock('pessimistic_write')
    .where('u.id = :id', { id })
    .getOne();

  if (!victimLocked) throw new NotFoundException('Usuario no encontrado');

  if (victimLocked.isActive && (victimLocked.role?.canManageUsers ?? false)) {
    const otherActiveAdmins = await queryRunner.manager
      .createQueryBuilder(User, 'u')
      .innerJoin('u.role', 'r')
      .setLock('pessimistic_write') // lock all admin rows so concurrent tx blocks
      .where('u.isActive = :active', { active: true })
      .andWhere('r.canManageUsers = :flag', { flag: true })
      .andWhere('u.id != :id', { id })
      .getCount();
    if (otherActiveAdmins === 0) {
      throw new BadRequestException('No se puede dejar el sistema sin administradores activos');
    }
  }

  await this.scenariosService.transferOwnership(id, callerId, suffix, queryRunner);
  await queryRunner.manager.delete(User, id);
  await queryRunner.commitTransaction();
} catch (error) {
  await queryRunner.rollbackTransaction();
  throw error;
} finally {
  await queryRunner.release();
}
```

Para `update`: envolver toda la operación (find + role lookup + count + save) en un QueryRunner con la misma estrategia.

**Test cobertura:** sumar test de integración con dos requests concurrentes y `setTimeout`-delays artificiales, o al menos un test unitario que verifique que el COUNT está dentro de la misma transacción que el save/delete.

### CR-A2: `UsersService.update` no usa transacción — read+write desacoplados

**File:** `nemea-back/src/users/users.service.ts:75-154`

**Issue:**

A diferencia de `remove` (que sí abre un `QueryRunner` para la operación delete+transfer), el método `update` **no usa transacción alguna**:

1. Línea 80: `findOne({ where: { id } })` — lectura suelta.
2. Línea 108: `roleRepository.findOne(...)` — lectura suelta.
3. Línea 124-131: COUNT de admins — lectura suelta.
4. Línea 153: `usersRepository.save(victim)` — escritura suelta.

Entre los pasos (3) y (4) puede ejecutarse cualquier otro request — incluso un DELETE concurrente sobre el mismo usuario. Resultado:

- **Lost update** clásico: dos PATCH concurrentes pueden pisar campos del otro (último gana).
- **Phantom-row delete**: PATCH puede ejecutar `save` sobre un user que fue borrado en T+ε; con TypeORM/PostgreSQL un `save` con id existente hace `UPDATE WHERE id=X` que afecta 0 rows (silencioso), o un `INSERT` si TypeORM no detecta el id existente — comportamiento dependiente de la versión/config.
- **TOCTOU para el guard de admin** (descripto en CR-A1).
- **Inconsistencia role lookup vs save**: el `roleRepository.findOne` lee el rol en T0, pero el rol puede haber sido borrado/modificado antes del `save` en T+ε. `Role.isSystem` boundary no se chequea acá; un admin podría asignar un rol que está siendo borrado concurrentemente.

**Fix:**

Envolver `update` en un `QueryRunner` siguiendo el mismo patrón de `remove`:

```ts
async update(id: string, dto: UpdateUserDto, callerId: string): Promise<User> {
  const queryRunner = this.usersRepository.manager.connection.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction('SERIALIZABLE');

  try {
    const victim = await queryRunner.manager
      .createQueryBuilder(User, 'u')
      .innerJoinAndSelect('u.role', 'r')
      .setLock('pessimistic_write')
      .where('u.id = :id', { id })
      .getOne();

    if (!victim) throw new NotFoundException('Usuario no encontrado');

    // ... resto de los guards, pero usando queryRunner.manager
    // ... y countQB también con setLock('pessimistic_write')

    const saved = await queryRunner.manager.save(User, victim);
    await queryRunner.commitTransaction();
    return saved;
  } catch (error) {
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
}
```

Esto cierra simultáneamente CR-A1 y CR-A2.

### WR-A2: `update` self-role-edit guard acepta `roleId: null` y dispara mensaje engañoso

**File:** `nemea-back/src/users/users.service.ts:92-98`

**Issue:** La condición `dto.roleId !== undefined && dto.roleId !== victimRoleId` es true cuando `dto.roleId === null` (porque null ≠ undefined y null ≠ uuid). Pero `@IsUUID()` sobre el DTO debería rechazar null. Confirmar comportamiento real:

- `class-validator` `@IsOptional()` skipea cuando value es null o undefined.
- Por ende `{ roleId: null }` pasa la validación.
- En el service, `dto.roleId !== victimRoleId` → throws "No puedes cambiar tu propio rol" en el caso self-edit.
- Para non-self, intenta `roleRepository.findOne({ where: { id: null } })` → comportamiento TypeORM impredecible (puede retornar el primer row, retornar null, o tirar error).

**Fix:** explicitar el rechazo de null en el DTO:

```ts
@ApiProperty({ required: false, description: 'UUID del rol a asignar' })
@IsUUID()
@IsOptional()
@IsNotEmpty({ message: 'roleId no puede ser vacío o null' }) // o ValidateIf((_, v) => v !== null) + IsString
roleId?: string;
```

O bien, en el service, agregar guard: `if (dto.roleId === null) throw new BadRequestException('roleId no puede ser null')`.

### WR-A3: `remove` chequea self-delete ANTES de validar existencia del victim — mensaje confuso

**File:** `nemea-back/src/users/users.service.ts:157-160`

```ts
if (id === callerId) {
  throw new BadRequestException('No puedes borrarte a vos mismo');
}
const victim = await this.usersRepository.findOne({ where: { id } });
if (!victim) throw new NotFoundException('Usuario no encontrado');
```

**Issue:** Si un admin (caller) tiene una JWT válida (id `X`) pero el row de su user fue borrado de la BD por otro admin (escenario raro pero posible con multi-sesión), un intento de borrar a SU PROPIO id devuelve "No puedes borrarte a vos mismo" en lugar de 404. **Fix:** invertir el orden — primero validar existencia, después self-check:

```ts
const victim = await this.usersRepository.findOne({ where: { id } });

if (id === callerId) throw new BadRequestException('No puedes borrarte a vos mismo');
```

Sí, esto agrega una query extra cuando el caller intenta borrarse a sí mismo, pero da el error correcto y el orden semánticamente coincide con `update` (que primero hace findOne).

### WR-A4: Race condition en `UsersController.create` — email uniqueness check fuera de transacción

**File:** `nemea-back/src/users/users.controller.ts:59-67`

```ts
async create(@Body() dto: CreateUserDto): Promise<User> {
  const existing = await this.usersService.findByEmail(dto.email);
  if (existing) throw new ConflictException('Email ya registrado');
  return this.usersService.create(dto);
}
```

**Issue:** Patrón TOCTOU clásico: dos requests concurrentes con el mismo email ambos pasan `findByEmail`, ambos llaman `create`, y el segundo recibe un 500 (DB unique constraint violation) en lugar del 409 esperado. La cobertura del DB `@Unique(['email'])` evita data corruption pero el código del controller pretende dar el 409 y falla bajo concurrencia.

Adicionalmente, la lógica de "chequear duplicado antes de insertar" idiomáticamente debería vivir en el service (cerca del resto de la lógica de negocio), no en el controller. Hoy esto está partido: el controller hace el chequeo, y la BD también.

**Fix:** mover el chequeo al service y envolverlo en una transacción, o (más simple) capturar el error de unique constraint en el service y re-lanzar como `ConflictException`:

```ts
// users.service.ts
async create(dto: CreateUserDto): Promise<User> {
  // ... role validation ...
  const user = this.usersRepository.create({ ... });
  try {
    return await this.usersRepository.save(user);
  } catch (err) {
    if (isUniqueViolation(err, 'email')) {
      throw new ConflictException('Email ya registrado');
    }
    throw err;
  }
}
```

Y eliminar el `findByEmail` del controller.

### WR-A7: `transferOwnership` no protege contra colisiones de truncamiento que sobreviven al UUID-suffix mitigation

**File:** `nemea-back/src/scenarios/scenarios.service.ts:391-400`

**Issue:** WR-01 del review anterior se cerró agregando `#${shortId}` al suffix. Esto mitiga las colisiones de NOMBRE base. Pero el truncamiento `LEFT(name || :suffix, 200)` se aplica al RESULTADO. Si el `name` original es 195 chars y el suffix son 18 chars, el resultado se trunca a 200 → corta el suffix por la mitad, incluyendo posiblemente el hash UUID. Dos scenarios distintos con nombres largos cuyos prefijos coinciden tras truncar quedan idénticos.

Ejemplo:

- Scenario 1: `"Plan migración inversores 2027 Q1 con datos consolidados desde Tiendanube hub..."` (195 chars) + suffix `" - Maria #ab12cd34"` → `"Plan migración inversores 2027 Q1 con datos consolidados desde Tiendanube hub... - Maria #"` (cortado al char 200, perdiendo `ab12cd34`).
- Scenario 2: idem prefijo, distinto sufijo en chars 196-200 del original → tras concatenar y truncar, colisiona.

Probabilidad real: baja (necesita nombres específicamente largos). Pero el invariante de unicidad se rompe.

**Fix:** truncar el `name` original ANTES de concatenar, para garantizar que el suffix sobrevive intacto:

```ts
.set({
  // Truncate base name to (200 - len(suffix)) so the discriminator never gets cut.
  // suffix.length is known at JS time so we can interpolate it safely.
  name: () => `LEFT(COALESCE(name, ''), ${200 - suffix.length}) || :suffix`,
  user: { id: newOwnerId },
})
```

Verificar overflow si `suffix.length > 200` (defensive throw en JS antes del UPDATE).

Alternativa: agregar la `@Unique(['user', 'name'])` constraint a la entity y dejar que la BD enforcee. Si la unicidad importa, hay que enforcearla en el schema.
