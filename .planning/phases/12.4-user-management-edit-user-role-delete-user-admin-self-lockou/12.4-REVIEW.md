# Fase 12.4 — Code Review Report

**Profundidad:** standard

## Summary

Se revisaron los archivos de la fase 12.4 (user management edit + delete con guards de self-lockout y transferencia atómica de scenarios). El diseño general es sólido: los tres guards (self-role-edit, self-deactivate, self-delete) están bien implementados, el guard de last-admin se calcula correctamente, y la transacción `QueryRunner` envuelve transferOwnership + delete con orden correcto (transfer antes que delete, para evitar que `ON DELETE CASCADE` borre scenarios). El uso de `setParameter('suffix', ...)` en TypeORM también es la forma correcta de evitar SQL injection.

Sin embargo, hay 2 issues BLOCKER y 7 WARNING que merecen atención antes de mergear:

- **CR-01**: `victim.role` accedido sin null-check en tres lugares del service — explota con `Cannot read of null` si llega un usuario sin rol (caso edge pero permitido por el create code path).
- **CR-02**: el sufijo `usuario borrado` no se aplica cuando `victim.name === ''` (string vacío, no null), porque el código usa `??` en vez de chequear truthiness — produce sufijos visualmente vacíos (` - `) en los scenarios transferidos.

Las warnings cubren issues de robustez (nombres concatenados que pueden romper la unicidad latente de `(user_id, name)`, validación faltante en DTO, comportamiento de `apiClientFetch` con 204, etc.) que conviene resolver antes de promocionar a prod.

## Critical Issues

### CR-01: `victim.role.id` / `victim.role.canManageUsers` acceden sin null-check

**Files:**
- `nemea-back/src/users/users.service.ts:89` (`update`, self-role-edit guard)
- `nemea-back/src/users/users.service.ts:101` (`update`, role-changed check)
- `nemea-back/src/users/users.service.ts:114` (`update`, `wasAdminActive`)
- `nemea-back/src/users/users.service.ts:156` (`remove`, last-admin guard)

**Issue:** El código asume que `victim.role` siempre está definido. Sin embargo:

1. `User.role` es `@ManyToOne(() => Role, { eager: true })` sin `nullable: false` explícito en el decorador, por lo que TypeORM crea la columna `role_id` como **nullable** por default.
2. `UsersService.create()` (línea 56-61) crea el user con `role: dto.roleId ? ({ id: dto.roleId } as Role) : undefined` — admite construir un user sin rol asignado.
3. La migration `1773000000000-CreateRolesAndMigrateUsers` debería verificarse para confirmar la nullability real en BD, pero el contrato del código permite el caso.

Si un usuario sin rol entra al endpoint `PATCH /users/:id` o `DELETE /users/:id`, el service tira `TypeError: Cannot read properties of null (reading 'id')` (500 al cliente, no el 400/404 esperado).

**Fix:**

```ts
// users.service.ts:88
const victimRoleId = victim.role?.id ?? null;
if (
  id === callerId &&
  dto.roleId !== undefined &&
  dto.roleId !== victimRoleId
) {
  throw new BadRequestException('No puedes cambiar tu propio rol');
}

// users.service.ts:101 — resolve newRole
let newRole = victim.role ?? null;
if (dto.roleId !== undefined && dto.roleId !== victimRoleId) {
  const found = await this.roleRepository.findOne({ where: { id: dto.roleId } });
  if (!found) throw new NotFoundException('Rol no encontrado');
  newRole = found;
}

// users.service.ts:113-114 — last-admin
const willBeAdmin = newRole?.canManageUsers ?? false;
const wasAdminActive =
  victim.isActive && (victim.role?.canManageUsers ?? false);

// users.service.ts:156 — remove
if (victim.isActive && (victim.role?.canManageUsers ?? false)) {
  // ...
}
```

Alternativamente, agregar un guard fast-fail al inicio de `update`/`remove`: `if (!victim.role) throw new InternalServerErrorException('Usuario sin rol asignado')`. La opción de null-coalesce es más defensiva.

### CR-02: Sufijo del delete pierde el nombre cuando `victim.name === ''`

**File:** `nemea-back/src/users/users.service.ts:180`

```ts
const suffix = ` - ${victim.name ?? 'usuario borrado'}`;
```

**Issue:** El operador `??` solo dispara el fallback cuando `victim.name` es `null` o `undefined`. Pero `User.name` es `varchar(255) nullable`, y nada en el flujo lo normaliza a `null` cuando se vacía. El frontend `EditUserDialog` (línea 115) envía `nextName = ''` cuando el admin borra el campo:

```ts
const nextName = data.name ?? '';
if (nextName !== (user.name ?? '')) {
  dto.name = nextName; // '' se envía al backend
}
```

Y `UpdateUserDto.name` es `@IsString @IsOptional` (acepta `''`). El backend lo guarda tal cual (`victim.name = ''` en línea 134), porque `dto.name !== undefined` es true. Resultado: cuando un admin posteriormente borra ese usuario, el sufijo se vuelve `' - '` (string vacío visible), no `' - usuario borrado'`.

Esto rompe la propiedad principal del contrato del UI-SPEC: que los scenarios transferidos sean visualmente identificables como provenientes de un usuario borrado. El test del front (`DeleteUserAlertDialog.test.tsx:70-80`) cubre el caso `name: null` pero **no el caso `name: ''`**.

**Fix opción A** (backend, robustez defensiva):

```ts
// users.service.ts:180
const trimmedName = victim.name?.trim();
const suffix = ` - ${trimmedName ? trimmedName : 'usuario borrado'}`;
```

**Fix opción B** (frontend, normalización):

```ts
// EditUserDialog.tsx:115-118
const nextName = data.name?.trim() ?? '';
if (nextName !== (user.name ?? '')) {
  dto.name = nextName === '' ? null : nextName;
}
```

**Fix opción C** (recomendado, ambos): normalizar tanto en el backend (defensa-en-profundidad) como en el frontend (cleanup). Sumar un test en `users.service.spec.ts` que cubra `victim.name = ''` y `victim.name = '   '`.

## Warnings

### WR-01: `transferOwnership` puede crear nombres duplicados sin constraint UNIQUE

**File:** `nemea-back/src/scenarios/scenarios.service.ts:385-401`

**Issue:** El bulk UPDATE renombra los scenarios del víctim agregando el sufijo, pero no chequea colisiones contra los scenarios existentes del caller. Verifiqué `1772900000000-CreateScenariosTables.ts` y la entidad `Scenario`: **no existe constraint UNIQUE en `(user_id, name)`**, solo se valida en código en `create()` y `update()`.

Escenario realista:

1. Caller (admin "Juan") ya borró previamente al user "Maria" → recibió "Aumento Enero 2027 - Maria".
2. Ahora va a borrar al user "Maria 2" (nombre repetido) → el bulk UPDATE intenta renombrar "Aumento Enero 2027" → "Aumento Enero 2027 - Maria".
3. **Resultado:** caller queda con dos scenarios distintos llamados "Aumento Enero 2027 - Maria". El próximo `create` o `update` en la UI va a tirar `ConflictException` aún cuando el user no esté creando nada.

Adicionalmente: la truncación `LEFT(name || suffix, 200)` puede dejar a múltiples scenarios con el mismo nombre cortado (ej.: dos nombres de 198 chars que difieren al final pasan a colisionar tras la concatenación + truncate).

**Fix sugerido:** Posibles opciones:

- (a) Agregar uniqueness check en transferOwnership: leer scenarios del caller con el nombre proyectado, y agregar contador `" - Juan (2)"` si colisiona. Caro pero correcto.
- (b) Agregar el `id` UUID corto al sufijo (`" - Juan #a1b2"`) — barato, garantiza unicidad práctica.
- (c) Documentar y aceptar que el invariante `(user_id, name)` es soft. Si se elige esta opción, hay que **eliminar el `ConflictException` de `create`/`update`** o renombrar el error para que no aparezca como "duplicado" cuando el duplicado fue creado por una transferencia.

### WR-02: `apiClientFetch` parsea JSON del response 204 No Content

**File:** `nemea-front/src/app/(app)/usuarios/DeleteUserAlertDialog.tsx:54` (consumer) y `nemea-front/src/lib/api-client.ts:38` (root cause)

**Issue:** El endpoint `DELETE /users/:id` está marcado con `@HttpCode(204)`. Cuando el backend responde 204 sin body, `apiClientFetch` ejecuta `return res.json()` (línea 38) que lanza `SyntaxError: Unexpected end of JSON input` (rejecta la promise). El `try/catch` en `handleDelete` (línea 53-67) captura ese error como un `Error` y muestra `toast.error('Unexpected end of JSON input')` — el delete **funcionó en el backend** pero el usuario ve un error y `onSuccess()` nunca se llama (no se cierra el dialog, no se refresca la tabla).

El test (`DeleteUserAlertDialog.test.tsx:46-50`) mockea `fetch` con `json: () => Promise.resolve({})`, por lo que el bug está oculto en CI.

**Nota:** Este bug pre-existe (el patrón se usa en `scenarios`, `catalogs`, `roles` con `@HttpCode(204)` también), pero esta fase es la primera en introducir un endpoint **nuevo** que usa explícitamente `@HttpCode(204)` en este sub-proyecto. Vale la pena resolverlo en `apiClientFetch` para que sea robusto:

**Fix:**

```ts
// api-client.ts:38
if (res.status === 204 || res.headers.get('content-length') === '0') {
  return undefined as T;
}
return res.json() as Promise<T>;
```

Sumar test del fetch real (no mockeado) o test que mockee con `Response` real (`new Response(null, { status: 204 })`).

### WR-03: `isOwnRow` se computa con `editTarget?.id === currentUserId` — falsos positivos cuando ambos son undefined

**File:** `nemea-front/src/app/(app)/usuarios/UsersClient.tsx:297`

```tsx
isOwnRow={editTarget?.id === currentUserId}
```

**Issue:** Cuando `editTarget === null` (dialog cerrado) y `session?.user?.id` es `undefined` (sesión cargando, expirada, o degradada), la comparación es `undefined === undefined` → `true`. Mientras el dialog esté cerrado (`open={editTarget !== null}` → `false`), no afecta visualmente. Pero el caso problemático es: **si `currentUserId` es `undefined` para un user legítimamente autenticado** (race condition de NextAuth al montar), `editTarget?.id === currentUserId` queda en `false` para cualquier `editTarget` con id, y el guard `isOwnRow` no protege. El usuario podría editar su propio rol/active porque el front no lo bloquea (el backend SÍ lo bloquea — defensa en profundidad — pero la UX queda inconsistente).

**Fix:**

```tsx
isOwnRow={Boolean(currentUserId) && editTarget?.id === currentUserId}
```

O bien, hacer un guard explícito en `EditUserDialog`: si `currentUserId` no está disponible, mostrar un toast de error y cerrar el dialog.

### WR-04: `EditUserDialog` permite enviar PATCH vacío (no-op request)

**File:** `nemea-front/src/app/(app)/usuarios/EditUserDialog.tsx:111-140`

**Issue:** El `onSubmit` construye `dto` solo con campos cambiados. Si el user abre el dialog y hace click en "Guardar" sin tocar nada, `dto = {}` y se envía `PATCH /users/:id` con body `{}`. El backend acepta y hace el flujo entero (findOne + save). Esto es:

- Inocuo funcionalmente, pero **dispara request innecesario** y **muestra `toast.success('Usuario actualizado')` aunque nada haya cambiado** (UX engañosa).
- Si en el medio cambia el rol del user (otro admin), el save trae un side effect inesperado al persistir un objeto que el cliente ya tenía cacheado (`save(victim)` re-aplica `victim.role = newRole` que internamente apunta al rol viejo si el victim fue cargado antes).

**Fix:**

```ts
// EditUserDialog.tsx:126
if (Object.keys(dto).length === 0) {
  toast.info('No hay cambios para guardar');
  onOpenChange(false);
  return;
}

try {
  await apiClientFetch(...);
  // ...
}
```

### WR-05: Zod schema permite `roleId: ''` cuando hay roles pero solo valida en submit

**File:** `nemea-front/src/app/(app)/usuarios/EditUserDialog.tsx:48-52`

```ts
const editUserSchema = z.object({
  name: z.string().max(255).optional().or(z.literal('')),
  roleId: z.string().uuid('Selecciona un rol'),
  isActive: z.boolean(),
});
```

**Issue:** El initial values pone `roleId: user?.role.id ?? ''` (línea 79). Si por alguna razón `user.role.id` es undefined o `''`, el primer render del Select pasa `value=''` que el Radix Select interpreta como "ningún item seleccionado" — el SelectTrigger queda en estado `placeholder`. El user puede hacer click en "Guardar" antes de cargar el rol válido. El zod resolver tira `roleId: 'Selecciona un rol'` y bloquea, pero **el error se muestra después de un click frustrante**. Más serio: si llegara a inyectarse un `roleId` no-UUID por algún path (mock, malformed prop), el resolver se cae en runtime sin un fallback claro.

**Fix:** Agregar guard al inicio del `onSubmit` o al `formValues`:

```ts
const formValues = useMemo<EditUserFormData>(
  () => ({
    name: user?.name ?? '',
    roleId: user?.role?.id ?? '',
    isActive: user?.isActive ?? true,
  }),
  [user],
);
```

Y validar al inicio del submit (defensive): `if (!data.roleId) { toast.error('Falta el rol'); return; }`.

(Nota: `user?.role.id` también explota si `user.role` es null/undefined — mismo patrón que CR-01.)

### WR-06: `UpdateUserDto.name` permite `null` semánticamente pero el tipo es `string`

**File:** `nemea-back/src/users/dto/update-user.dto.ts:7-8`

```ts
@IsString()
@IsOptional()
name?: string;
```

**Issue:** El DTO no permite `null` (sólo `string` u omitido). Pero la entidad `User.name` es nullable. El frontend siempre envía string (puede ser `''`), nunca `null`. Resultado: **es imposible volver a poner `null` en `name` desde el endpoint de update**. Sólo el flujo `updateGoogleProfile` puede setear null.

Si se quiere soportar "limpiar el nombre y dejarlo como null" (recomendado para que `victim.name ?? 'usuario borrado'` funcione, ver CR-02), el DTO debería aceptarlo:

```ts
@ApiProperty({ required: false, description: 'Nombre del usuario (null para limpiar)', nullable: true })
@IsString()
@ValidateIf((_, value) => value !== null)
@IsOptional()
name?: string | null;
```

Y `users.service.ts:133` ya cubre `dto.name !== undefined` (acepta null correctamente).

### WR-07: Test de `transferOwnership` "no-op" no valida que NO se haga UPDATE cuando victim no tiene scenarios

**File:** `nemea-back/src/scenarios/scenarios.service.spec.ts:360-373`

**Issue:** El test "es no-op cuando victim no tiene scenarios" mockea `execute` para devolver `{ affected: 0 }` y verifica que `execute` se llamó. Pero la implementación **siempre** ejecuta el UPDATE bulk, sin importar si el víctim tiene scenarios o no. Esto significa: el test no cubre un real no-op (no se hace query). Si en el futuro alguien optimiza para evitar el UPDATE cuando no hay scenarios, este test pasaría con la implementación opuesta.

Más importante: la implementación actual hace un UPDATE incluso cuando no hay filas, lo cual es funcionalmente correcto (no-op a nivel BD) pero el test no diferencia.

**Fix:** Eliminar el test ambiguo o renombrarlo a "ejecuta UPDATE incluso cuando victim no tiene scenarios (idempotent)". Si la intención era validar "no falla cuando affected=0", el nombre debe reflejarlo.

### IN-02: Comentario en `scenarios.service.ts:176-177` sobre `canManageUsers` vs `canManageScenarios` está bien documentado pero el endpoint usa otro permiso

**File:** `nemea-back/src/scenarios/scenarios.service.ts:174-178` y `nemea-back/src/scenarios/scenarios.controller.ts:97`

**Issue:** El comentario aclara que `canManageUsers` se usa para el admin bypass del delete de scenarios. Pero el endpoint `DELETE /scenarios/:id` requiere `can_manage_scenarios` (no `can_manage_users`). Esto significa que un admin con `can_manage_users=true` pero `can_manage_scenarios=false` **no puede llegar al endpoint** — su bypass es teórico, nunca dispara. La documentación es engañosa.

Las opciones son: (a) cambiar el `@RequirePermission` del endpoint a un guard custom que acepte ambos; (b) clarificar el comentario indicando que en la práctica el bypass solo se activa para users con ambos permisos.

### IN-03: `users.controller.ts` reusa el chequeo de email duplicado en POST pero `update` no

**File:** `nemea-back/src/users/users.controller.ts:59-66`

**Issue:** `POST /users` hace `findByEmail` antes de `create` para tirar `ConflictException`. Si en el futuro `UpdateUserDto` agrega `email`, hay que replicar este patrón. Comentario menor: idealmente el chequeo de unicidad debería estar en el service (no en el controller) para mantenerlo cerca de la lógica de negocio. Hoy está duplicado: `users.controller.ts` chequea, y la BD también lo previene con `@Unique(['email'])` en `User.entity.ts`. Si el controller se omite (por mistake en otra fase), el error vendría como 500 (constraint violation) en vez de 409.

### IN-05: `useMemo` para `formValues` con dependencia `[user]` puede causar reset extra en RHF

**File:** `nemea-front/src/app/(app)/usuarios/EditUserDialog.tsx:76-105`

**Issue:** `formValues` se re-computa en cada render cuando `user` cambia de referencia (mismo contenido, distinta ref si `users` se refresca con `router.refresh()`). Combinado con `values: formValues` en RHF + `useEffect → reset(formValues)`, esto puede causar resets dobles. El comment del código reconoce el problema ("Defensive: ensure form resets..."), pero la solución más limpia es usar `defaultValues` + key prop:

```tsx
<EditUserDialog key={user?.id ?? 'none'} ... />
```

Esto fuerza remount cuando cambia el user target y elimina la necesidad del `useEffect` + `useMemo` workaround.
