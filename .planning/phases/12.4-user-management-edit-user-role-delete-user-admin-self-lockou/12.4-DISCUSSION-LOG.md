## Cross-Module Coupling for Scenario Rename

| Option | Description | Selected |
|--------|-------------|----------|
| Raw SQL UPDATE in QueryRunner | UsersService writes raw SQL inside its transaction. UsersModule independent. Worse type-safety. | |
| Inject `Repository<Scenario>` | UsersModule imports `TypeOrmModule.forFeature([..., Scenario])`. Type-safe but UsersService knows Scenario schema. | |
| Inject `ScenariosService` with new `transferOwnership` method | New dedicated service method, bulk UPDATE via QueryBuilder, accepts QueryRunner. Domain boundary respected. | ✓ |

**User's choice:** Inject `ScenariosService` with new `transferOwnership(victimId, newOwnerId, suffix, qr)` method using bulk UPDATE via QueryBuilder.
**Notes:** User asked whether existing `ScenariosService.update()` could be reused — confirmed unsuitable (per-id, user-scoped, no QueryRunner). User explicitly framed the decision around portfolio quality: "esta app es pensada para ser un portafolio, no quiero chanchadas". Recommended Option 3 with parameter binding (no string interpolation) for safety against SQL injection on the suffix.

| Option | Description | Selected |
