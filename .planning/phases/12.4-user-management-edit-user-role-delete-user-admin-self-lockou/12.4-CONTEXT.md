# Phase 12.4: User management ‚Äî edit user, delete user, admin self-lockout - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Admin can edit any user's name, role, and active status via a Dialog on `/usuarios`, and hard-delete a user (transferring their scenarios to the deleting admin with a name suffix marking provenance). Backend enforces three self-lockout guards: no editing own role, no deleting self, no leaving the system without active admins. The current `PATCH /users/:id/toggle-status` endpoint is retired in favor of a single `PATCH /users/:id`.

</domain>

<decisions>
## Implementation Decisions

### Cross-Module Coupling for Scenario Rename
- **D-01:** Add a new method `transferOwnership(victimId: string, newOwnerId: string, suffix: string, qr: QueryRunner): Promise<void>` to `ScenariosService`. It does a single bulk `UPDATE` via QueryBuilder, joining the caller's transaction. UsersModule imports ScenariosModule (no circular dep ‚Äî ScenariosModule does not import UsersModule today).
- **D-02:** The bulk UPDATE uses parameter binding for the suffix to avoid SQL injection: `LEFT(name || :suffix, 200)` with `setParameter('suffix', suffixString)`. Domain encapsulation: scenario mutation logic stays in `ScenariosService`, not leaked into `UsersService`.
- **D-03:** UsersModule update ‚Äî `imports: [TypeOrmModule.forFeature([User, Role]), ScenariosModule]`. No direct `Repository<Scenario>` injection in UsersModule.

### Self-Lockout Guards: Service-Level
- **D-04:** All three self-lockout rules live in `UsersService`, not in the controller. Methods accept `callerId` as an explicit argument: `update(id, dto, callerId)` and `remove(id, callerId)`. Controllers pass `@CurrentUser('id') callerId` and delegate.
- **D-05:** For consistency, the existing `toggle-status` self-check (currently inline in the controller at `users.controller.ts:86`) becomes irrelevant because **the toggle-status endpoint is deleted entirely** (see D-15). The self-deactivation guard lives only in `UsersService.update` via the `isActive` path.
- **D-06:** The "no admin lockout" guard is computed as a `SELECT COUNT(*)` inside the transaction, after the candidate write would be applied. Concretely: in `update`, if the change would (a) deactivate a user with `can_manage_users`, or (b) move a user with `can_manage_users` to a role without it ‚Äî compute `SELECT COUNT(*) FROM users u JOIN roles r ON u.role_id = r.id WHERE u.is_active = true AND r.can_manage_users = true AND u.id != :victimId` plus +1 if the change would leave the victim qualifying. If 0, throw `BadRequestException('No se puede dejar el sistema sin administradores activos')`. In `remove`, simpler: same count excluding the victim. The check happens inside the transaction so it sees the latest committed state plus pending writes.

### Delete Preview UX
- **D-07:** No scenario counter in the AlertDialog. Copy: "Sus escenarios ser√°n transferidos a tu cuenta con el sufijo ' - {name}'. Esta acci√≥n no se puede deshacer." Avoids the extra query and the entity-level OneToMany declaration. SPEC text adjusted: the count phrase in requirement 4 becomes generic.

### Self-Action UI in Own Row
- **D-08:** Editar button visible in own row. When opened on own row, the EditUserDialog renders the Role `Select` and the Active `Switch` as `disabled` with tooltip "No puedes cambiar tu propio rol/estado". Only the Name `Input` is editable.
- **D-09:** Borrar button hidden in own row, replaced by the text "(tu cuenta)" ‚Äî same pattern as the existing toggle-status behavior on `users.controller.ts` (which is being removed; the text label stays in the UI).

### Test Coverage
- **D-10:** Backend tests required:
  - Extend `users.service.spec.ts`: `update()` happy paths (name-only, role-only, isActive-only, combined); each of the 3 self-lockout guards; transactional rollback when `transferOwnership` fails.
  - Extend `users.controller.spec.ts`: PATCH endpoint (200 / 400 / 404 paths); DELETE endpoint (204 / 400 / 404); permission gating reflected (covered by the existing pattern).
  - Add cases to `scenarios.service.spec.ts` for `transferOwnership`: bulk UPDATE applies the suffix, truncates to 200 chars when needed, no-op when victim has 0 scenarios.
- **D-11:** Frontend tests required (first non-scaffold frontend tests in the repo):
  - `EditUserDialog.test.tsx`: form renders with prefilled values; Name editable / Role+Active disabled on own row; submit calls PATCH and triggers refresh; error handling shows toast.
  - `DeleteUserAlertDialog.test.tsx`: dialog renders with rename preview copy; Confirm calls DELETE; Cancel closes without firing.
  - Setup: extend existing jest.config.js + jest.setup.ts (already configured per `.planning/codebase/TESTING.md`). Use React Testing Library + `next-auth/react` mock (`useSession` returns a fixed session).

### Toggle-Status Endpoint Cleanup
- **D-12:** `PATCH /users/:id/toggle-status` and `UsersService.activate()` / `UsersService.deactivate()` are deleted. Replaced by `PATCH /users/:id { isActive }`.
- **D-13:** `UsersClient.tsx` updated to call `PATCH /users/:id` for activation/deactivation, no longer hitting `/toggle-status`.
- **D-14:** Existing tests targeting `toggle-status` in `users.controller.spec.ts` are deleted; replaced by `PATCH /users/:id` cases.
- **D-15:** OpenAPI/Swagger doc reflects the removal automatically (the endpoint is removed from the controller).

### Suffix Chain
- **D-16:** The rename does not parse or strip existing suffixes. Chain grows naturally: `Promo - Nacho - Ana` after two cascade deletes. SQL truncates to 200 chars via `LEFT(...)`. If a chain hits the cap, the rightmost characters of the most-recent victim name are clipped ‚Äî acceptable because provenance is informational, not load-bearing. Admin can manually clean up names from `/escenarios` if desired.

### EditUserDialog Form Layout
- **D-17:** Single vertical form, no tabs, no separated destructive action. Field order top-to-bottom:
  1. Email (read-only, gray-muted) ‚Äî shown for context, not editable
  2. Name (Input, optional ‚Äî can be cleared)
  3. Role (Select, fetched from `/api/roles` as in the create flow ‚Äî reuse the same `RoleOption[]` prop already passed to `UsersClient`)
  4. Active (Switch with label "Activo")
- **D-18:** Single Save button persists all three changes in one `PATCH /users/:id`. Footer: `Cancelar` + `Guardar`. Patron consistente con CreateScenarioDialog y RoleDialog.
