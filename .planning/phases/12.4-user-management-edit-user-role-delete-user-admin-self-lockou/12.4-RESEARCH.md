### Finding 1 — TypeORM QueryRunner cross-service joining

**Confidence:** HIGH — patrón espejo verificado en `scenarios.service.ts:220-249`.

**Contrato del patrón QueryRunner cross-service** (D-01):

- **El caller** (`UsersService.remove`) es quien crea + connect + startTransaction + commit/rollback + release.
- **El callee** (`ScenariosService.transferOwnership`) recibe el `qr: QueryRunner` y ejecuta sus queries vía `qr.manager` — **nunca** llama a `commit`, `rollback`, ni `release`. Si tira, propaga el error y el caller hace rollback.

**Esqueleto de `UsersService.remove` (patrón a seguir):**

```typescript
// [VERIFIED: nemea-back/src/scenarios/scenarios.service.ts:220-249 — patrón espejo]
async remove(id: string, callerId: string): Promise<void> {
  // Pre-checks fuera de la transacción (404 y self-delete guard) — fast-fail
  if (id === callerId) {
    throw new BadRequestException('No puedes borrarte a vos mismo');
  }
  const victim = await this.usersRepository.findOne({
    where: { id },
    relations: ['role'],
  });
  if (!victim) {
    throw new NotFoundException('Usuario no encontrado');
  }

  // Last-admin guard (puede ir aquí o dentro de la tx — ver Finding 5)
  if (victim.isActive && victim.role.canManageUsers) {
    const remainingAdmins = await this.usersRepository
      .createQueryBuilder('u')
      .innerJoin('u.role', 'r')
      .where('u.isActive = :active', { active: true })
      .andWhere('r.canManageUsers = :flag', { flag: true })
      .andWhere('u.id != :id', { id })
      .getCount();
    if (remainingAdmins === 0) {
      throw new BadRequestException(
        'No se puede dejar el sistema sin administradores activos',
      );
    }
  }

  const queryRunner =
    this.usersRepository.manager.connection.createQueryRunner();
  await queryRunner.connect();
  await queryRunner.startTransaction();

  try {
    const suffix = ` - ${victim.name ?? 'usuario borrado'}`;
    await this.scenariosService.transferOwnership(
      id,
      callerId,
      suffix,
      queryRunner,
    );
    await queryRunner.manager.delete(User, id);
    await queryRunner.commitTransaction();
  } catch (error) {
    await queryRunner.rollbackTransaction();
    throw error;
  } finally {
    await queryRunner.release();
  }
}
```

**Pitfall — la fuente de la connection:** En el espejo (`scenarios.service.ts:222`) la connection se obtiene vía `this.scenarioRepo.manager.connection.createQueryRunner()`. En UsersService usar la misma vía: `this.usersRepository.manager.connection.createQueryRunner()`. Ambos repositorios apuntan al mismo `DataSource`, así que el QueryRunner es compatible con cualquier entidad — esto es lo que permite el cross-module join.

### Finding 2 — Bulk UPDATE via QueryBuilder con parameter binding

**Confidence:** HIGH — confirmado vía docs TypeORM (typeorm.io + GitHub issues #881, #1580).

**Sintaxis canónica de `ScenariosService.transferOwnership`:**

```typescript
// [VERIFIED: typeorm.io/docs/query-builder/update-query-builder/ +
//  github.com/typeorm/typeorm/issues/1580 (arrow function + setParameter pattern)]
async transferOwnership(
  victimId: string,
  newOwnerId: string,
  suffix: string,
  qr: QueryRunner,
): Promise<void> {
  await qr.manager
    .createQueryBuilder()
    .update(Scenario)
    .set({
      name: () => 'LEFT(COALESCE(name, \'\') || :suffix, 200)',
      user: { id: newOwnerId },
    })
    .where('user_id = :victimId', { victimId })
    .setParameter('suffix', suffix)
    .execute();
}
```

**Reglas confirmadas:**

1. **Arrow function en `.set()`**: `() => 'expression'` permite SQL raw en el SET clause. Esto NO escapa el string — pero los placeholders `:name` dentro del string sí se vinculan vía `setParameter`. [VERIFIED: typeorm.io]
2. **`setParameter` (singular)** une un parámetro de a uno; **`setParameters({})`** une muchos en un objeto. Ambos funcionan dentro de raw expressions y dentro de `.where()`. Para un solo parámetro como `suffix`, `setParameter` es el match.
3. **Orden de `setParameter`**: puede ir antes o después de `.where(..., { victimId })` — el binding es resuelto al `.execute()`.
4. **PostgreSQL `||` para concat**: nativo. `COALESCE(name, '')` defiende contra `name NULL` (defensa adicional aunque el schema declare `nullable: false` para `scenarios.name` — ver entity).
5. **`LEFT(str, 200)`**: nativo en PostgreSQL. Trunca por la derecha cuando se excede `varchar(200)`.
6. **Cómo se updatea la FK `user_id`**: con `set({ user: { id: newOwnerId } })`, TypeORM extrae el `id` para componer la FK column. **Alternativa más explícita** si TypeORM no resuelve bien con el arrow function mezclado: `.set({ name: () => '...', userId: newOwnerId })` — pero `user_id` no existe como column property en la entity (es `@JoinColumn`). Si hay duda, usar la sintaxis raw también para el user_id:

```typescript
.set({
  name: () => 'LEFT(COALESCE(name, \'\') || :suffix, 200)',
  user: () => ':newOwnerId',
})
.setParameter('newOwnerId', newOwnerId)
```

**Pitfall confirmado:** el GitHub issue #1580 muestra que `.set({ value: ':value' }).setParameter('value', 2)` NO funciona (TypeORM lo trata como string literal). La única forma de mezclar parametrización con expresión SQL es vía `() => 'expr'`. Documentado.

**Pitfall — `LEFT()` cuenta bytes vs chars:** En PostgreSQL `LEFT()` cuenta **caracteres** (codepoints), no bytes. Si la columna es `varchar(200)` y tiene tildes/emojis, no hay problema — 200 chars siempre cabe en 200 codepoints. Sin riesgo.

### Finding 3 — `@CurrentUser('id')` decorator

**Confidence:** HIGH — verificado en código.

`nemea-back/src/auth/decorators/current-user.decorator.ts:11-25` ya soporta la firma con key parameter:

```typescript
// [VERIFIED: nemea-back/src/auth/decorators/current-user.decorator.ts]
export const CurrentUser = createParamDecorator(
  (data: keyof JwtUser | undefined, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest<Request>();
    const user = request.user as JwtUser;
    if (data) {
      return user[data]; // returns user['id'] when @CurrentUser('id') is used
    }
    return user;
  },
);
```

`JwtUser.id: string` (UUID del sub del JWT). Uso en los nuevos endpoints:

```typescript
@Patch(':id')
@RequirePermission('can_manage_users')
async update(
  @Param('id', ParseUUIDPipe) id: string,
  @Body() dto: UpdateUserDto,
  @CurrentUser('id') callerId: string,
): Promise<User> {
  return this.usersService.update(id, dto, callerId);
}

@Delete(':id')
@HttpCode(204)
@RequirePermission('can_manage_users')
async remove(
  @Param('id', ParseUUIDPipe) id: string,
  @CurrentUser('id') callerId: string,
): Promise<void> {
  return this.usersService.remove(id, callerId);
}
```

**Nota — return type del decorator:** la signature actual devuelve `JwtUser | JwtUser[keyof JwtUser]`. Cuando se llama con `'id'`, TypeScript no estrecha al tipo `string` automáticamente. **Mitigación:** anotar el parámetro: `@CurrentUser('id') callerId: string`. NestJS no chequea esto en runtime, pero la anotación de TS gana para el consumidor del service.

### Finding 5 — Last-admin count: visibilidad dentro de transacción

**Confidence:** HIGH (semántica PostgreSQL bien definida).

**Hecho confirmado:** PostgreSQL por defecto usa **READ COMMITTED**. Dentro de una misma transacción, todos los `SELECT` (incluyendo `COUNT(*)`) ven los `UPDATE`/`DELETE`/`INSERT` previos de la misma transacción — es el comportamiento "una transacción ve sus propios writes" garantizado por el estándar. Esto es exactamente lo que D-06 necesita. [CITED: postgresql.org/docs/current/transaction-iso.html]

**Patrón limpio (recomendado) — chequeo BEFORE write, sin race window:**

Aplicar la regla en lógica TypeScript, no en SQL:

```typescript
// Para UPDATE (deactivate o role-demote):
async update(id: string, dto: UpdateUserDto, callerId: string): Promise<User> {
  const victim = await this.usersRepository.findOne({
    where: { id },
    relations: ['role'],
  });
  if (!victim) throw new NotFoundException('Usuario no encontrado');

  // Guard: no editar propio rol
  if (
    id === callerId &&
    dto.roleId !== undefined &&
    dto.roleId !== victim.role.id
  ) {
    throw new BadRequestException('No puedes cambiar tu propio rol');
  }

  // Guard: no desactivarse a uno mismo
  if (id === callerId && dto.isActive === false) {
    throw new BadRequestException('No puedes desactivar tu propia cuenta');
  }

  // Resolver nuevo rol si cambia
  let newRole = victim.role;
  if (dto.roleId !== undefined && dto.roleId !== victim.role.id) {
    const found = await this.roleRepository.findOne({
      where: { id: dto.roleId },
    });
    if (!found) throw new NotFoundException('Rol no encontrado');
    newRole = found;
  }

  // Estado "after change" para chequear last-admin
  const willBeActive = dto.isActive ?? victim.isActive;
  const willBeAdmin = newRole.canManageUsers;
  const wasAdminActive = victim.isActive && victim.role.canManageUsers;
  const willNoLongerBeAdminActive = wasAdminActive && !(willBeActive && willBeAdmin);

  if (willNoLongerBeAdminActive) {
    const otherActiveAdmins = await this.usersRepository
      .createQueryBuilder('u')
      .innerJoin('u.role', 'r')
      .where('u.isActive = :active', { active: true })
      .andWhere('r.canManageUsers = :flag', { flag: true })
      .andWhere('u.id != :id', { id })
      .getCount();
    if (otherActiveAdmins === 0) {
      throw new BadRequestException(
        'No se puede dejar el sistema sin administradores activos',
      );
    }
  }

  // Aplicar cambios
  if (dto.name !== undefined) victim.name = dto.name;
  victim.role = newRole;
  if (dto.isActive !== undefined) victim.isActive = dto.isActive;

  return this.usersRepository.save(victim);
}
```

**Nota — concurrency:** En teoría hay race entre "count" y "save" si dos admins en paralelo se desactivan entre sí. Para nuestro escala (admin operativo manual), el riesgo es despreciable. **Defensa-en-profundidad opcional:** wrap UPDATE+COUNT en una tx con `SERIALIZABLE` o `SELECT ... FOR UPDATE` sobre la fila víctima. **No es necesario para el alcance de esta fase** — documentar como "considered, not implemented" en plan-checker. El SPEC acepta either pattern.

**Para `remove`:** versión simplificada (no hay re-cómputo):

```typescript
if (victim.isActive && victim.role.canManageUsers) {
  const otherActiveAdmins = await this.usersRepository
    .createQueryBuilder('u')
```

### Finding 10 — Cascade FK + order timing

**Confidence:** HIGH — semántica de FK constraints garantizada.

**Hecho:** La FK `scenarios.user_id` tiene `ON DELETE CASCADE` (migration `1772900000000-CreateScenariosTables.ts:24`). La cascada se dispara **solamente cuando el row referenciado en `users` desaparece**.

**En la transacción de remove:**

1. **UPDATE scenarios SET user_id = newOwner WHERE user_id = victim** → reescribe la FK. Los rows ahora apuntan a `newOwner`, no a `victim`.
2. **DELETE FROM users WHERE id = victim** → busca rows con `user_id = victim` para cascadear. Encuentra **cero** (porque el UPDATE ya los movió). DELETE procede sin cascadear.

**Riesgo de timing: cero.** Dentro de la misma transacción, el UPDATE es visible al DELETE (mismo READ COMMITTED self-view). La cascada NO se dispara contra el `victim` deleted. La defensa-en-profundidad del CASCADE queda como red de seguridad por si alguien ejecuta DELETE directo en SQL bypass del service path.

**Confirmación adicional:** PostgreSQL evalúa los CASCADE triggers al momento del DELETE, leyendo el estado actual de las tablas (que incluye el UPDATE previo). [CITED: postgresql.org/docs/current/ddl-constraints.html — referential actions]
