# Phase 12.4 Plan 05: UsersService Refactor Summary

**One-liner:** Refactors `UsersService` end-to-end to add `update(id, dto, callerId)` with three self-lockout guards (self-role-edit, self-deactivate, last-active-admin), refactors `remove(id, callerId)` to an atomic `QueryRunner` transaction (transfer scenarios → delete user → commit, with rollback on error), wires `ScenariosModule` into `UsersModule`, and deletes the legacy `activate()`/`deactivate()` methods — closing the open service contract that Plan 01 left dangling in `users.controller.ts`.

## What Was Built

### Module — `nemea-back/src/users/users.module.ts` (MODIFIED)

Added `ScenariosModule` to imports for cross-module DI:

```ts
imports: [TypeOrmModule.forFeature([User, Role]), ScenariosModule]
```

### Service — `nemea-back/src/users/users.service.ts` (MODIFIED)

**Constructor injection extended:** Added `private readonly scenariosService: ScenariosService` as the third constructor parameter (after the two repositories).

**Deleted methods:**
- `activate(id)` (~3 lines)
- `deactivate(id)` (~3 lines)
- Old `remove(id)` single-arg implementation

**New method — `update(id, dto, callerId): Promise<User>`** with 3 guards in order:

| Guard | Trigger | Spanish Message |
|-------|---------|-----------------|
| 1 — Self-role-edit | `id === callerId && dto.roleId !== victim.role.id` | `'No puedes cambiar tu propio rol'` |
| 2 — Self-deactivate | `id === callerId && dto.isActive === false` | `'No puedes desactivar tu propia cuenta'` |
| 3 — Last-active-admin | Change would leave 0 active users with `canManageUsers = true` | `'No se puede dejar el sistema sin administradores activos'` |

Guard 3 logic computes `willNoLongerBeAdminActive` from `wasAdminActive` (pre-change) and `(willBeActive && willBeAdmin)` (post-change). If `true`, runs a parametrized `createQueryBuilder` count excluding the victim. Throws if `count === 0`.

Field application is explicit (no `repo.merge`):
```ts
if (dto.name !== undefined) victim.name = dto.name;
victim.role = newRole;
if (dto.isActive !== undefined) victim.isActive = dto.isActive;
return this.usersRepository.save(victim);
```

**New method — `remove(id, callerId): Promise<void>`** with 2 fast-fail guards + atomic transaction:

1. Guard 1 (self-delete) — fast-fail BEFORE any DB I/O.
2. Find victim (404 if not found).
3. Guard 2 (last-admin) — fast-fail BEFORE opening QueryRunner.
4. Open QueryRunner: `connect() → startTransaction()`.
5. Try block:
   - Build `suffix = ' - ${victim.name ?? 'usuario borrado'}'`.
   - Call `scenariosService.transferOwnership(id, callerId, suffix, queryRunner)`.
   - Call `queryRunner.manager.delete(User, id)`.
   - `commitTransaction()`.
6. Catch: `rollbackTransaction() → throw error`.
7. Finally: `release()`.

**Critical ordering** (per RESEARCH Finding 10): `transferOwnership` executes BEFORE `manager.delete(User, id)` inside the same transaction. The bulk `UPDATE scenarios SET user_id = newOwner WHERE user_id = victim` moves all rows off the victim's FK reference before the `DELETE FROM users` looks for them. The `ON DELETE CASCADE` constraint never fires — defense-in-depth net stays intact for direct-SQL bypass scenarios.

## Decisions Made

- **Field-by-field assignment instead of `repo.merge(victim, dto)`** — even though global `ValidationPipe({ whitelist: true, forbidNonWhitelisted: true })` strips extras at the controller boundary, defense-in-depth (T-12.4-22 in PLAN threat model) calls for explicit field control inside the service. If the pipe is ever bypassed (e.g., misconfigured override, internal caller), no extra fields leak into the entity.
- **Last-admin guard computed BEFORE the write** (TypeScript logic on the post-change state) rather than inside a transaction with re-count. Per RESEARCH Finding 5: PostgreSQL READ COMMITTED isolation makes both patterns correct; the simpler pre-write check matches the existing `roles.service.ts` pattern and avoids opening a transaction just to validate.
- **Fast-fail guards before opening QueryRunner** — self-delete, 404, and last-admin all short-circuit in `remove` before `manager.connection.createQueryRunner()` is called. Avoids orphan transactions on the most common rejection paths.
- **`transferOwnership` ordered BEFORE `manager.delete`** in the try-block — per RESEARCH Finding 10, this ordering means the UPDATE has already moved scenario rows off the victim's FK reference by the time the DELETE runs, so the `ON DELETE CASCADE` constraint on `scenarios.user_id` never fires. The CASCADE stays as defense-in-depth for direct-SQL bypass.
- **No `forwardRef` in UsersModule** — verified by grep that `scenarios.module.ts` does not import `UsersModule`. Direct `ScenariosModule` import is clean.
- **Valid v4 UUID constants at top of spec** — pattern carried from Plan 03 (which discovered that `zod .uuid()` and various decorators reject literals like `'role-uuid'`). All test IDs are proper v4 UUIDs assigned to named constants for readability.
- **Restore `mockReturnThis()` after `jest.clearAllMocks()`** in `beforeEach` — pattern carried from Plan 02. `clearAllMocks` wipes the `mockReturnThis` behavior, breaking chained QueryBuilder semantics in the second test onward. Inline restore in `beforeEach` keeps the chain alive across the entire suite.

## Threat Surface Coverage (from PLAN `<threat_model>`)

| Threat ID | Mitigation in this plan |
|-----------|-------------------------|
| T-12.4-03 (Self-role-edit privilege escalation) | Guard 1 in `update`: `id === callerId && dto.roleId !== victim.role.id` throws BadRequestException. callerId is JWT-derived (Plan 01 uses `@CurrentUser('id')`) — cannot be spoofed without forging JWT. Test: `'throws BadRequestException No puedes cambiar tu propio rol cuando victim === caller y roleId distinto'`. |
| T-12.4-04 (Self-delete privilege escalation) | Guard 1 in `remove`: `id === callerId` throws fast-fail before any DB I/O. Same callerId trust chain as T-12.4-03. Test: `'throws self-delete con No puedes borrarte a vos mismo cuando id === callerId'` + assertion that `findOne` was NOT called. |
| T-12.4-05 (Last-active-admin loss) | Guard 3 in both `update` and `remove`: parametrized `createQueryBuilder` count, throws if 0. Tests: 2 cases in `update` (demote + deactivate) + 1 in `remove`. |
| T-12.4-07 (transferOwnership + delete atomicity) | QueryRunner wraps both ops; if transferOwnership throws, rollback fires and user is NOT deleted. Test: `'rollbackea cuando transferOwnership falla'` asserts commit NOT called + manager.delete NOT called + release CALLED. |
| T-12.4-22 (repo.merge leaking unintended fields) | Service uses explicit field assignment, never `repo.merge`. Verified by grep returning 0. |
| T-12.4-23 (role re-assignment to invalid role) | `roleRepository.findOne({ where: { id: dto.roleId } })` validates existence; throws NotFoundException('Rol no encontrado'). Test: `'throws NotFoundException con Rol no encontrado cuando roleId no existe'`. |
| T-12.4-24 (Forged victim.id matching admin) | Upstream `@RequirePermission('can_manage_users')` from Plan 01 gates the endpoints. Inside the service, JWT-derived `callerId` is the source of truth for self-rule comparisons. |
| T-12.4-25 (Race between two admins deactivating each other) | Accepted per RESEARCH Finding 5 — theoretical race window at single-digit user count, mitigation cost (SERIALIZABLE / SELECT FOR UPDATE) exceeds benefit. Documented in PLAN. |
| T-12.4-SC (slopsquat / package install) | N/A — no new packages installed. |
