    **1. Install next-auth**:
    ```bash
    cd nemea-front

    ```

    **2. Create TypeScript module augmentation** at `nemea-front/src/types/next-auth.d.ts`:
    - Augment `next-auth` module:
      - `Session`: add `accessToken: string`, extend `user` with `id: string`, `role: string`
    - Augment `next-auth/jwt` module:
      - `JWT`: add `backendToken?: string`, `role?: string`, `userId?: string`
    - This MUST be done first so all other auth files have proper types

    **3. Create auth config** at `nemea-front/src/auth.ts`:
    - Import `NextAuth` from `next-auth` and `Google` from `next-auth/providers/google`
    - Export `{ handlers, auth, signIn, signOut }` from `NextAuth({ ... })`
    - Configuration:
      - `providers: [Google]` â€” Auth.js v5 auto-reads `AUTH_GOOGLE_ID` and `AUTH_GOOGLE_SECRET` env vars
      - `session: { strategy: 'jwt' }`
      - `pages: { signIn: '/login', error: '/login' }`
      - `callbacks`:
        - `signIn({ account })`: If provider is 'google', return true. All whitelist validation happens in the jwt callback. If provider is not google, return false.
        - `jwt({ token, account })`: When `account?.provider === 'google'` (initial sign-in only):
          1. Call backend `POST /api/auth/google` with `{ idToken: account.id_token }`
          2. If response is OK: store `data.accessToken` as `token.backendToken`, `data.user.role` as `token.role`, `data.user.id` as `token.userId`
          3. If response is NOT OK (401 = not whitelisted): do NOT store backendToken. On next page load, proxy.ts will see no session and redirect to `/acceso-denegado`
          4. Return token
        - `session({ session, token })`: Copy `token.backendToken` to `session.accessToken`, `token.role` to `session.user.role`, `token.userId` to `session.user.id`. Return session.
    - **CRITICAL per RESEARCH.md Pitfall 2**: Only call the backend in the `jwt` callback, NOT in `signIn`. This avoids double-calling POST /auth/google.
    - **CRITICAL per RESEARCH.md Pitfall 5**: Use `AUTH_SECRET` env var (not NEXTAUTH_SECRET). Auth.js v5 auto-detects it.
    - Use `process.env.NEXT_PUBLIC_API_URL` for the backend URL (available server-side and client-side).

    **4. Create NextAuth route handler** at `nemea-front/src/app/api/auth/[...nextauth]/route.ts`:
    ```typescript
    import { handlers } from '@/auth';
    export const { GET, POST } = handlers;
    ```
    - This is the entire file. Auth.js v5 handles all OAuth routes automatically.

    **5. Create proxy.ts** at `nemea-front/src/proxy.ts`:
    - Import `auth` from `@/auth`
    - Export `proxy` function using `auth((req) => { ... })`:
      - If `!req.auth` AND path is NOT `/login` AND path is NOT `/acceso-denegado`:
        - Build login URL: `new URL('/login', req.nextUrl.origin)`
        - Add `callbackUrl` search param: `req.nextUrl.pathname` (per user decision: redirect back after login)
        - Return `Response.redirect(loginUrl)`
      - If `req.auth` AND `!req.auth.accessToken` AND path is NOT `/acceso-denegado` AND path is NOT `/login`:
        - This means the user authenticated with Google but backend rejected them (no backendToken)
        - Return `Response.redirect(new URL('/acceso-denegado', req.nextUrl.origin))`
    - Export `config` with matcher: `'/((?!api|_next/static|_next/image|favicon.ico|fonts|images).*)'`
    - **CRITICAL per RESEARCH.md Pitfall 6**: The matcher MUST exclude `api` routes so NextAuth's `/api/auth/*` routes work

    **6. Create SessionProvider** at `nemea-front/src/providers/SessionProvider.tsx`:
    - `'use client'` directive at top
    - Import `SessionProvider as NextAuthSessionProvider` from `next-auth/react`
    - Export named function `SessionProvider` accepting `{ children: ReactNode }`
    - Return `<NextAuthSessionProvider>{children}</NextAuthSessionProvider>`
    - Follow the same pattern as existing `ThemeProvider.tsx`

    **7. Update layout.tsx** at `nemea-front/src/app/layout.tsx`:
    - Import `SessionProvider` from `@/providers/SessionProvider`
    - Wrap `{children}` with `<SessionProvider>` INSIDE `<ThemeProvider>`:
      ```tsx
      <ThemeProvider>
        <SessionProvider>
          {children}
          <Toaster />
        </SessionProvider>
      </ThemeProvider>
      ```

    **8. Create api fetch wrapper** at `nemea-front/src/lib/api.ts`:
    - Import `auth` from `@/auth`
    - Export async function `apiFetch<T>(path: string, options?: RequestInit): Promise<T>`
    - Implementation:
      1. Call `const session = await auth()` to get server-side session
      2. If `!session?.accessToken`, throw `new Error('No authenticated session')`
      3. Fetch `${process.env.NEXT_PUBLIC_API_URL}${path}` with:
         - Merge `options.headers` with `Authorization: Bearer ${session.accessToken}` and `Content-Type: application/json`
      4. If `!res.ok`, throw `new Error(`API error: ${res.status}`)`
      5. Return `res.json() as Promise<T>`
    - This is a SERVER-SIDE utility (uses `auth()` which only works in server components/actions)

    **9. Create .env.local.example** at `nemea-front/.env.local.example`:
    ```
    # NextAuth
    AUTH_SECRET=generate-with-npx-auth-secret
    AUTH_GOOGLE_ID=your-google-client-id.apps.googleusercontent.com
    AUTH_GOOGLE_SECRET=your-google-client-secret

    # Backend API
    NEXT_PUBLIC_API_URL=http://localhost:4000
    ```
