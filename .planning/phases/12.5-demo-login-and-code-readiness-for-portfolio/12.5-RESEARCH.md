## Summary

Phase 12.5 delivers six concrete surface areas across two repos. The research uncovers no architecture-level surprises: every required pattern already exists in the codebase and only needs to be extended or applied to new targets.

**Demo login (R1–R3):** `auth.service.ts:validateGoogleToken` is a 5-step function; the demo variant is the same 3 steps minus Google token verification. The env flag must be read via NestJS `ConfigService` (not `process.env`) because env validation runs at boot and DEMO_LOGIN_ENABLED must not be declared `required`. The frontend `auth.ts` callback structure already branches on `account?.provider === 'google'`; adding `'credentials'` is additive. The login page is a Server Component that already imports `signIn` from `@/auth` — the demo button calls `signIn('credentials', { redirect: false })` from a client sub-component or a Server Action.

**Testing (R4–R6):** The existing `users.service.spec.ts`, `roles.service.spec.ts`, and `scenarios.service.spec.ts` establish a completely uniform NestJS `Test.createTestingModule` pattern with flat mock-object repositories. The three new backend specs mirror this exactly. For the frontend, `EditUserDialog.test.tsx` is the canonical RTL pattern: mock `next-auth/react`, `next/navigation`, `sonner`, and `global.fetch`. The home-page ESM fix is mechanical: `jest.mock('@/auth', () => ({ auth: jest.fn() }))` before the `import` of `../page` short-circuits the NextAuth v5 ESM chain; `render(await Home())` then works because RTL can render pre-resolved RSC output.

**Surface cleanup (R8–R10):** `error.tsx` requires `'use client'` and receives `({ error, reset })`. `loading.tsx` is a plain Server Component with a centered spinner. Process comments span 5 backend files and 5 frontend files (confirmed by grep). **Primary recommendation:** Follow the existing patterns exactly — no new libraries, no new test infrastructure, no new config patterns. Every piece needed already exists.

## Finding 1: NextAuth v5 Credentials Provider

### Current auth.ts structure (confirmed by reading the file)

`nemea-front/src/auth.ts` currently:
- `providers: [Google]` (line 20)
- `signIn` callback: `if (account?.provider === 'google') return true; return false;` (lines 27–31)
- `jwt` callback: `if (account?.provider === 'google') { ... fetch /api/auth/google ... }` (lines 33–51)
- `session` callback: reads `token.backendToken`, `token.permissions`, `token.userId` (lines 69–76)

### How to add Credentials provider

```typescript
// Source: auth.ts — ADDITIVE changes, nothing removed
import Credentials from 'next-auth/providers/credentials';

const DEMO_EMAIL = 'demo@nemea.app'; // matches seed

export const { handlers, auth, signIn, signOut } = NextAuth({
  providers: [
    Google,
    Credentials({
      credentials: { email: { type: 'text' } },
      async authorize(credentials): Promise<{ backendToken: string; permissions: Permissions; userId: string; email: string } | null> {
        if (!credentials?.email) return null;
        try {
          const res = await fetch(
            `${process.env.NEXT_PUBLIC_API_URL}/api/auth/demo-login`,
            {
              method: 'POST',
              headers: { 'Content-Type': 'application/json' },
              body: JSON.stringify({ email: credentials.email }),
            },
          );
          if (!res.ok) return null;
          const body = (await res.json()) as BackendAuthResponse;
          return {
            backendToken: body.data.accessToken,
            permissions: body.data.user.permissions,
            userId: String(body.data.user.id),
            email: body.data.user.email,
          };
        } catch {
          return null;
        }
      },
    }),
  ],
  // ...
  callbacks: {
    signIn({ account }): boolean {
      if (account?.provider === 'google') return true;
      if (account?.provider === 'credentials') return true; // ADD
      return false;
    },
    async jwt({ token, account, user }): Promise<typeof token> {
      if (account?.provider === 'google') {
        // ...existing fetch to /api/auth/google...
      }
      if (account?.provider === 'credentials') {
        // user = what authorize() returned
        const u = user as { backendToken: string; permissions: Permissions; userId: string };
        token.backendToken = u.backendToken;
        token.permissions = u.permissions;
        token.userId = u.userId;
      }
      // ...stale JWT detection unchanged...
      return token;
    },
    // session callback unchanged
  },
});
```

**Critical pitfall — `user` type in jwt callback:** When `account?.provider === 'credentials'`, the `user` parameter of the `jwt` callback is the object returned by `authorize()`. For the Google case, `user` is not used (the backend fetch happens inline). The two branches are independent and don't interfere. [ASSUMED — NextAuth v5 jwt callback signature behavior with credentials]

**Critical pitfall — `redirect: false` in signIn call:** `signIn('credentials', { redirect: false, email: DEMO_EMAIL })` from a client component returns `{ ok: boolean, error?: string }` instead of redirecting. The client component must handle the redirect manually (`router.push('/')`) on success. [ASSUMED — NextAuth v5 credentials signIn return shape]

**Login page:** Currently a Server Component (`async function LoginPage`). The demo button must either be a Client Component (because it calls `signIn` from `next-auth/react`) or use a Server Action. Since the Google button already uses a Server Action (`'use server'` function), the cleanest approach is a separate `DemoLoginButton` Client Component that calls `signIn('credentials', ...)` from `next-auth/react`. This is shown by the `NEXT_PUBLIC_DEMO_LOGIN_ENABLED` env var: the button is conditionally rendered server-side based on the env var, so the LoginPage Server Component renders it or not. [VERIFIED: codebase read — existing login page structure]

### Current validateGoogleToken (auth.service.ts:25–62)

```typescript
  const payload = await this.verifyGoogleIdToken(idToken);      // Google-specific
  if (!payload || !payload.email) throw new UnauthorizedException('Token de Google invalido');

}
```

### Demo-login method (what to write)

```typescript
// In auth.service.ts — new method
async validateDemoLogin(email: string): Promise<AuthResponseDto> {
  // 1. Check env flag — reject early if disabled
  const enabled = this.configService.get<string>('DEMO_LOGIN_ENABLED');
  if (enabled !== 'true') {
    throw new UnauthorizedException('Demo login no disponible');
  }

  // 2. Find user (Role is eager-loaded on User entity — user.role is immediately available)
  const user = await this.usersService.findActiveByEmail(email);
  if (!user) {
    throw new UnauthorizedException('Usuario demo no encontrado');
  }

  // 3. Extract permissions and sign JWT — identical shape to Google path
  const permissions = extractPermissions(user.role);
  const accessToken = this.jwtService.sign({
    sub: user.id,
    email: user.email,
    permissions,
  });

  return {
    accessToken,
    user: {
      id: user.id,
      email: user.email,
      permissions,
      name: user.name ?? null,
      pictureUrl: user.pictureUrl ?? null,
    },
  };
}
```

**Key insight:** `User` entity has `@ManyToOne(() => Role, { eager: true })` (confirmed: `user.entity.ts:20`). So `findActiveByEmail` returns a user with `role` already populated — `extractPermissions(user.role)` works directly, identical to the Google path.

### Env flag strategy — ConfigService, not process.env

`auth.service.ts` already uses `ConfigService.getOrThrow()` for `GOOGLE_CLIENT_ID` (line 21). The same pattern applies. However, `DEMO_LOGIN_ENABLED` must NOT be `getOrThrow` because it is optional (default false). Use `configService.get<string>('DEMO_LOGIN_ENABLED')`.

**Env validation schema:** `env.validation.ts` uses Joi. Add the optional field:
```typescript
DEMO_LOGIN_ENABLED: Joi.string().valid('true', 'false').default('false'),
```
This keeps the validation schema the single source of truth and makes the default explicit. [VERIFIED: codebase read — env.validation.ts uses Joi, pattern confirmed]

## Finding 3: Demo-User Seed Migration

### Pattern from existing migrations

The admin user was originally seeded inline in `1772369233961-CreateUserTable.ts`:
```sql
INSERT INTO "users" ("email", "role", "name", "is_active") VALUES ('basualdofelipe@gmail.com', 'admin', 'Felipe Basualdo', true)
```

After Phase 12.1, `1773000000000-CreateRolesAndMigrateUsers.ts` restructured users with a `role_id` FK to `roles`. That migration seeded ADMIN and USER roles and migrated existing users.

**The demo-user seed must be a new migration** (per immutability constraint). It needs the ADMIN role ID — which is not a fixed UUID (generated by `uuid_generate_v4()` at migration time). The seed migration must query for the ADMIN role by name:

```typescript
// New migration: 1773200000000-SeedDemoUser.ts
export class SeedDemoUser1773200000000 implements MigrationInterface {
  name = 'SeedDemoUser1773200000000';

  public async up(queryRunner: QueryRunner): Promise<void> {
    // Fetch the ADMIN role id by name (not hardcoded — uuid was generated at migration time)
    const adminRoleResult = await queryRunner.query(
      `SELECT id FROM "roles" WHERE "name" = 'ADMIN' LIMIT 1`,
    );
    const adminRoleId = adminRoleResult[0]?.id;
    if (!adminRoleId) {
      throw new Error('ADMIN role not found — ensure CreateRolesAndMigrateUsers migration ran first');
    }

    // Insert demo user (upsert to be idempotent on re-run)
    await queryRunner.query(
      `INSERT INTO "users" ("email", "name", "is_active", "role_id")
       VALUES ('demo@nemea.app', 'Demo Nemea', true, $1)
       ON CONFLICT ("email") DO NOTHING`,
      [adminRoleId],
    );
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(
      `DELETE FROM "users" WHERE "email" = 'demo@nemea.app'`,
    );
  }
}
```

**`ON CONFLICT DO NOTHING`** makes the migration idempotent — safe to run on a DB that already has the demo user. [VERIFIED: codebase read — existing migrations use queryRunner.query pattern; ASSUMED — ON CONFLICT syntax for idempotency]

**Migration filename timestamp:** Must be greater than `1773100000000` (the last migration). Use `1773200000000`. [VERIFIED: codebase read — migration list]

## Finding 4: Jest + NextAuth v5 ESM — Home Page Test

### Why the current test fails (confirmed)

`nemea-front/src/app/(app)/__tests__/page.test.tsx` (current state):
1. `render(<Home />)` — passes `Home` synchronously but `Home` is `async function Home(): Promise<ReactElement>` (line 15 of `(app)/page.tsx`). RTL can't render a Promise as a React element.
2. The import `import Home from '../page'` triggers `import { auth } from '@/auth'` which imports from `next-auth` — a package using ESM-only modules. `next/jest` handles most of the transform, but the NextAuth v5 internal ESM chain can cause parse errors in some environments.
3. Assertions `getByText('NEMEA')` and subtitle — both text strings no longer exist in `(app)/page.tsx` (confirmed: page renders quick-link cards with 'Productos', 'Calculadora', 'Insumos', 'Gastos').

## Common Pitfalls

### Pitfall 1: Credentials provider authorize() returning `null` vs throwing
**What goes wrong:** If `authorize()` throws an exception, NextAuth treats it as an unexpected error (500-class). If it returns `null`, NextAuth maps it to a sign-in error that redirects to the error page (configured as `/login` in current `auth.ts`).
**How to avoid:** Always return `null` on failure (backend 4xx, network error) — never throw. Wrap the fetch in try/catch.

### Pitfall 2: `signIn('credentials')` redirect behavior from client
**What goes wrong:** Calling `signIn('credentials', { email })` without `{ redirect: false }` from a Client Component triggers a full page redirect to `/` which can cause layout flashes or miss error handling.
**How to avoid:** Always pass `{ redirect: false }` from the Client Component. Then check the returned `{ ok, error }` and `router.push('/')` manually on success.

### Pitfall 3: DEMO_LOGIN_ENABLED required vs optional in Joi schema
**What goes wrong:** Using `Joi.string().required()` for `DEMO_LOGIN_ENABLED` causes the NestJS app to fail to start in any environment where the var is not set (all existing deployments).
**How to avoid:** Use `Joi.string().valid('true', 'false').default('false')` — optional with explicit default.

### Pitfall 4: findActiveByEmail returns User WITHOUT role if eager is broken
**What goes wrong:** If `extractPermissions(user.role)` is called when `user.role` is undefined, `NO_PERMISSIONS` is returned (the function handles null/undefined gracefully — line 71 of `permission.ts`). The demo user appears to log in but has zero permissions.
**How to avoid:** Confirm `@ManyToOne(() => Role, { eager: true })` is present (it is — confirmed `user.entity.ts:20`). No action needed, but the spec should assert that returned permissions are non-empty.

### Pitfall 5: `render(await Home())` — Lucide icons need SVG support in jsdom
**What goes wrong:** `ShoppingBag`, `Calculator`, `Package`, `Receipt` from `lucide-react` render SVG elements. jsdom supports SVG but some implementations miss specific SVG methods.
**How to avoid:** The existing `jest.setup.ts` already has ResizeObserver, scrollIntoView, and pointer capture shims. Lucide icons typically render fine. If errors occur, mock `lucide-react` returning `<svg data-testid="icon" />`.

### Pitfall 6: Migration timestamp collision
**What goes wrong:** A migration timestamp that is less than or equal to an existing one causes TypeORM to skip or error.
**How to avoid:** Use `1773200000000` which is greater than `1773100000000-AddSortOrderToProductSizes.ts` (confirmed last migration). [VERIFIED: codebase read — migration list]

### Pitfall 7: Process-comment removal breaks logic
**What goes wrong:** Some process comments are on the same line as code. Removing the wrong characters breaks the statement.
**How to avoid:** For inline comments (`code; // WR-A4: explanation`), remove only from `//` to end of line. For standalone comment lines (`// WR-A4: long explanation`), remove the entire line. Run `tsc --noEmit` and `npm run lint` after.
