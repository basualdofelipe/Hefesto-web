# Phase 12.5: Demo login and code-readiness for portfolio — Specification

**Created:** 2026-05-26 (revised 2026-05-26 — corrected testing scope)
**Ambiguity score:** 0.12 (gate: ≤ 0.20)
**Requirements:** 10 locked

## Goal

The repository becomes clone-and-run for a reviewer without a whitelisted Google account, and the codebase reaches a professional portfolio bar: a demo login path (alongside Google OAuth), a 100%-green test suite in both repos with meaningful coverage of the untested business logic, App Router error/loading boundaries, no internal process comments in source, and Tiendanube slug literals centralized in application code.

## Background

The app is functionally complete through Phase 12.4 and the code quality is genuinely good (zero `any`, TS strict, clean NestJS architecture, real domain logic). The blockers to portfolio-readiness are clone-and-run access, test gaps, and presentation noise:

- **Login is Google-OAuth-only.** `nemea-front/src/auth.ts:20` declares `providers: [Google]`; the `signIn`/`jwt` callbacks only handle `account?.provider === 'google'`. The backend only exposes `POST /api/auth/google` (`auth.controller.ts`), which verifies a Google `id_token` via `validateGoogleToken` (`auth.service.ts:25`). A reviewer cloning the repo cannot log in without a whitelisted Google identity. This — not Docker/DB — is the real "clone-and-run" blocker.
- **Test gaps in real business logic (corrected scope, 2026-05-26).** Three substantial backend services have NO spec: `products.service.ts` (504 lines — BOM/SKU/price history), `supplies.service.ts` (274 lines — feeds cost calc), `tiendanube-config.service.ts` (257 lines — rate parsing). The frontend's flagship client logic (`CalculadoraClient.tsx` — forward/inverse, cascading gateways, debounce; `ScenarioEditorClient.tsx`) has no test. **Historical root cause:** early in the project the user said "prefiero testear con curl en verify-work" (manual UAT), which got over-generalized into "no test files." That was a misunderstanding — the user wants tests at a professional level. An earlier "LEAN / defer tests" framing of this phase was a calibration error and is reverted: **tests are first-class deliverables of 12.5.**
- **Home test is broken.** `nemea-front/src/app/(app)/__tests__/page.test.tsx` asserts `getByText('NEMEA')` and a subtitle that no longer exist (home is now permission-aware quick-link cards), and `render()`s `Home` synchronously though `Home` is an async Server Component that `await auth()`s (`(app)/page.tsx:15`). It fails to parse under jest because importing `../page` pulls in `@/auth` (NextAuth v5, ESM). It will be replaced (not deleted): an async RSC can be unit-tested with `render(await Home())` + a mock of `@/auth`, which also short-circuits the ESM chain — giving real coverage of the card-per-permission and empty-state logic without Playwright.
- **No error/loading boundaries.** No `error.tsx`/`loading.tsx` under the `(app)/` route group — a failed SSR fetch crashes Next instead of degrading.
- **Process comments leaked into source.** Internal tracker markers (`WR-…`, `IN-A…`, `CRITICAL: review fix`) in ~6 backend production files + ~7 frontend production files, plus test files.
- **Tiendanube slug literals duplicated in code.** The Tiendanube data IS in the database (Phase 10 seeded `tn_plans`, `tn_payment_gateways`, etc.) and the frontend reads it from the API — no hardcoded plan list. The only smell is a small number of places where application code branches on a slug literal: `calculadora.service.ts:116` (`gatewaySlug === 'pago_nube'`), `GatewaySelectors.tsx:30` (`DEFAULT_PLAN_SLUG = 'esencial'`), and a couple of spots in `scenarios.service.ts` / DTOs. The seed migration defines these slugs as canonical values.

This phase delivers the **corrected code scope** (demo-login + professional tests + surface cleanup). Manual portfolio artifacts (READMEs, screenshots, making repos public, secrets git-history audit, coverage badge in README) are done outside GSD and are out of scope.

## Requirements

1. **Backend demo-login endpoint**: An env-gated endpoint mints the same JWT as Google login without Google verification.
   - Current: `AuthController` exposes only `POST /auth/google` and `GET /auth/me`; the only token-minting path requires a valid Google `id_token`.
   - Target: `POST /api/auth/demo-login` accepts an email, mirrors `validateGoogleToken` minus the Google check (`findActiveByEmail` → `extractPermissions(user.role)` → `jwtService.sign({ sub, email, permissions })`), gated by env flag `DEMO_LOGIN_ENABLED` (default `false`). When off/unset the endpoint rejects.
   - Acceptance: With `DEMO_LOGIN_ENABLED=true`, `POST /api/auth/demo-login` with the seeded demo email returns `{ accessToken, user }` with populated permissions; with the flag false/unset it returns a 4xx and no token. A unit/e2e test covers both states.

2. **Frontend Credentials provider + demo entry**: NextAuth gains a Credentials provider alongside Google, and `/login` offers a one-click demo entry.
   - Current: `auth.ts` has `providers: [Google]`; `signIn`/`jwt` callbacks handle only `provider === 'google'`; `/login` shows the Google button only.
   - Target: add a `Credentials` provider that calls `POST /api/auth/demo-login`; extend `signIn`/`jwt` callbacks to handle `account?.provider === 'credentials'` identically to google (store `backendToken`, `permissions`, `userId`); add an "Entrar como demo" one-click button on `/login`, shown only when demo login is enabled. `PermissionsGuard` and `@RequirePermission` are NOT modified.
   - Acceptance: Clicking "Entrar como demo" yields an authenticated session whose `session.user.permissions` are populated from the demo user's role; the Google flow is unchanged; no guard or `@RequirePermission` decorator changed.

3. **Seed demo user**: A demo account exists so a fresh clone can log in immediately.
   - Current: the only seeded account is the admin (`basualdofelipe@gmail.com`).
   - Target: a new seed/migration adds an active demo user (e.g. `demo@nemea.app`) with the admin role.
   - Acceptance: a fresh `migration:run` + seed creates the demo user as active with the admin role; `demo-login` with that email succeeds.

4. **Backend service specs for untested business logic**: The three substantial untested services get behavior-focused specs.
   - Current: `products.service.ts` (504 lines), `supplies.service.ts` (274 lines), `tiendanube-config.service.ts` (257 lines) have no `.spec.ts`.
   - Target: add `products.service.spec.ts`, `supplies.service.spec.ts`, `tiendanube-config.service.spec.ts` covering real cases (happy / error / edge) for their domain logic — e.g. SKU generation, BOM versioning and active-record swap, selling-price history; supply price-history "current price"; rate parsing/append-only history. Trivial pass-through CRUD need not be exhaustively tested.
   - Acceptance: the three spec files exist and pass; each asserts behavior (inputs→outputs/state), not just that a method was called; `npm test` in `nemea-back` includes them green.

5. **Frontend client-logic tests**: The flagship client components are tested.
   - Current: `CalculadoraClient.tsx` and `ScenarioEditorClient.tsx` have no tests.
   - Target: add RTL tests for `CalculadoraClient` (forward/inverse mode, cascading gateway selection, debounced input behavior) and `ScenarioEditorClient` (override entry, recalculated margin display), mocking the API layer.
   - Acceptance: both test suites exist and pass under jsdom; they assert user-visible behavior (rendered results / state transitions), not implementation details; `npm test` in `nemea-front` includes them green.

6. **Home page test replaced**: The broken home test is rewritten to cover the real home.
   - Current: `page.test.tsx` asserts removed `'NEMEA'`/subtitle text and `render()`s an async RSC synchronously; fails to parse (NextAuth v5 ESM via `@/auth`).
   - Target: rewrite as `render(await Home())` with `@/auth` mocked, asserting permission-card rendering for a mocked permission set and the empty-state when no permissions are granted. NOT deleted.
   - Acceptance: the rewritten test passes; it asserts cards render per mocked permissions and the empty-state renders; no `'NEMEA'`/subtitle assertions remain.

7. **Full test suite green in both repos**: Every test passes; coverage is reported informationally.
   - Current: `nemea-back` is 121/121; `nemea-front` fails on the home test (ESM parse).
   - Target: after R4–R6, `npm test` is 100% green in both `nemea-front` and `nemea-back`. Coverage is generated and reported as an informational number — NOT enforced as a hard gate (no padding trivial code to hit a percentage).
   - Acceptance: `npm test` exits 0 in `nemea-front` AND `nemea-back`; a coverage report can be produced (`--coverage`) without errors.

8. **Error and loading boundaries in (app)/**: The app degrades gracefully on SSR/fetch failure.
   - Current: no `error.tsx`/`loading.tsx` under `(app)/`; a failed server fetch crashes Next.
   - Target: add `error.tsx` (graceful error UI with retry) and `loading.tsx` (loading state) in the `(app)/` route group.
   - Acceptance: both files exist under `(app)/`; a simulated backend/SSR failure renders the error boundary rather than an unhandled Next crash.

9. **Remove process comments (prod + tests)**: Internal tracker markers are gone from all source.
   - Current: `WR-…`, `IN-A…`, `CRITICAL: review fix` markers in backend and frontend production files and in test files.
   - Target: remove all such process/tracker comments from production source AND test files in both repos, preserving genuinely explanatory comments.
   - Acceptance: `grep -rE 'WR-[0-9]|IN-A[0-9]|CRITICAL: review fix'` over `nemea-front/src` and `nemea-back/src` (including tests) returns zero matches; lint, `tsc`, and tests remain green.

10. **Centralize Tiendanube slug constants (front + back, migration untouched)**: Slug literals used for branching live in one named place per repo.
    - Current: slug literals (`'pago_nube'`, `'esencial'`, `'tarjeta_debito_credito'`, etc.) written inline at comparison/default sites (`calculadora.service.ts`, `GatewaySelectors.tsx`, `scenarios.service.ts`, DTOs).
    - Target: a shared constants module in the frontend and one in the backend; replace inline literals at comparison/default sites with named constants. The Phase 10 seed migration is NOT modified (migrations are immutable historical records).
    - Acceptance: application code branching/defaulting on a Tiendanube slug imports from a single constants module per repo; no bare slug literal in a comparison/default position in app code (migration excluded); the calculadora round-trip test still passes.

## Constraints

- **Zero new npm dependencies** — NextAuth Credentials, JWT signing, jest/RTL are already available; consistent with the v1.1 "no new deps" rule.
- **Demo login must be env-gated and OFF by default in real production** — a password-less admin entry must never be reachable in real prod.
- **Historical migrations are immutable** — no edits to existing migration files; the demo-user seed is a new migration/seed step.
- **NextAuth v5 is ESM** — jest fixes are via module mocks of `@/auth`, not by weakening `transformIgnorePatterns` project-wide.
- **`PermissionsGuard` and `@RequirePermission` must not be modified** — the demo path reuses the identical JWT shape so authorization is unaware of the entry method.
- **Coverage is reported, not gated** — tests target named business-logic units with real cases; do not pad trivial CRUD/UI primitives to reach a number.
- **UI copy in español argentino**; technical identifiers, code, and slugs stay as-is.
