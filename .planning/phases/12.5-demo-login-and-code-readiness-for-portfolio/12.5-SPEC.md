# Phase 12.5: Demo login and code-readiness for portfolio — Specification

**Created:** 2026-05-26
**Ambiguity score:** 0.15 (gate: ≤ 0.20)
**Requirements:** 7 locked

## Goal

The repository becomes clone-and-run for a reviewer without a whitelisted Google account, and the codebase is presentation-clean: a demo login path (alongside the existing Google OAuth), a green test suite in both repos, App Router error/loading boundaries, no internal process comments in source, and Tiendanube slug literals centralized in application code.

## Background

The app is functionally complete through Phase 12.4 and the code quality is genuinely good (zero `any`, TS strict, clean NestJS architecture, real domain logic). The blockers to portfolio-readiness are presentation and "clone-and-run":

- **Login is Google-OAuth-only.** `nemea-front/src/auth.ts:20` declares `providers: [Google]`; the `signIn` and `jwt` callbacks only handle `account?.provider === 'google'`. The backend only exposes `POST /api/auth/google` (`auth.controller.ts`), which verifies a Google `id_token` via `validateGoogleToken` (`auth.service.ts:25`). A reviewer cloning the repo cannot log in without a whitelisted Google identity. This — not Docker/DB — is the real "clone-and-run" blocker.
- **Home test is broken.** `nemea-front/src/app/(app)/__tests__/page.test.tsx` asserts `getByText('NEMEA')` and a subtitle that no longer exist (the home is now permission-aware quick-link cards), and it `render()`s `Home` synchronously even though `Home` is an async Server Component that `await auth()`s (`(app)/page.tsx:15`). It fails to even parse under jest because importing `../page` pulls in `@/auth` (NextAuth v5, ESM) — a pre-existing failure noted in STATE.md (12.4 P07).
- **No error/loading boundaries.** No `error.tsx` or `loading.tsx` exists under the `(app)/` route group — a failed SSR fetch crashes Next instead of degrading.
- **Process comments leaked into source.** Internal tracker markers (`WR-…`, `IN-A…`, `CRITICAL: review fix`) appear in ~6 backend production files (`scenarios.service.ts`, `users.service.ts`, `update-user.dto.ts`) and ~7 frontend production files (`api-client.ts`, `UsersClient.tsx`, `EditUserDialog.tsx`), plus test files.
- **Tiendanube slug literals duplicated in code.** The Tiendanube data (plans, gateways, rates) IS in the database (Phase 10 seeded `tn_plans`, `tn_payment_gateways`, `tn_gateway_rates`, `tn_installment_rates`, `tn_tax_config`) and the frontend reads it from the API — there is no hardcoded plan list. The only smell is a small number of places where application code branches on a slug literal: `calculadora.service.ts:116` (`gatewaySlug === 'pago_nube'`), `GatewaySelectors.tsx:30` (`DEFAULT_PLAN_SLUG = 'esencial'`), and a couple of spots in `scenarios.service.ts` / DTOs. The seed migration `1772800000000-CreateTiendanubeConfigTables.ts` defines these slugs as canonical seed values.

This phase delivers the **LEAN code scope** decided 2026-05-26. Manual artifacts (READMEs, screenshots, making repos public, secrets git-history audit) are done outside GSD and are out of scope here.

## Requirements

1. **Backend demo-login endpoint**: An env-gated endpoint mints the same JWT as Google login without Google verification.
   - Current: `AuthController` exposes only `POST /auth/google` and `GET /auth/me`; the only token-minting path requires a valid Google `id_token`.
   - Target: `POST /api/auth/demo-login` accepts an email, mirrors `validateGoogleToken` minus the Google check (`findActiveByEmail` → `extractPermissions(user.role)` → `jwtService.sign({ sub, email, permissions })`), and is gated by env flag `DEMO_LOGIN_ENABLED` (default `false`). When the flag is off/unset the endpoint rejects the request.
   - Acceptance: With `DEMO_LOGIN_ENABLED=true`, `POST /api/auth/demo-login` with the seeded demo email returns `{ accessToken, user }` with populated permissions; with the flag false/unset it returns a 4xx and no token. A unit/e2e test covers both states.

2. **Frontend Credentials provider + demo entry**: NextAuth gains a Credentials provider alongside Google, and `/login` offers a one-click demo entry.
   - Current: `auth.ts` has `providers: [Google]`; `signIn`/`jwt` callbacks handle only `provider === 'google'`; `/login` shows the Google button only.
   - Target: add a `Credentials` provider that calls `POST /api/auth/demo-login`; extend the `signIn` and `jwt` callbacks to handle `account?.provider === 'credentials'` identically to google (store `backendToken`, `permissions`, `userId`); add an "Entrar como demo" one-click button on `/login`, shown only when demo login is enabled. The `PermissionsGuard` and `@RequirePermission` are NOT modified.
   - Acceptance: Clicking "Entrar como demo" yields an authenticated session whose `session.user.permissions` are populated from the demo user's role; the Google login flow is unchanged; no change to any guard or `@RequirePermission` decorator.

3. **Seed demo user**: A demo account exists so a fresh clone can log in immediately.
   - Current: the only seeded account is the admin (`basualdofelipe@gmail.com`).
   - Target: a seed/migration adds an active demo user (e.g. `demo@nemea.app`) with the admin role, so a local cloner sees the full app.
   - Acceptance: a fresh `migration:run` + seed creates the demo user as active with the admin role; `demo-login` with that email succeeds.

4. **Home test rewritten + full suite green**: The home test verifies the real permission-card home, and the whole test suite passes in both repos.
   - Current: `page.test.tsx` asserts removed text and fails to parse (NextAuth v5 ESM via `@/auth`); the other two front suites (EditUserDialog, DeleteUserAlertDialog) pass; backend is 121/121.
   - Target: rewrite `page.test.tsx` to test the permission-aware home (mock `@/auth` to short-circuit the ESM import chain, assert visible cards per mocked permissions and the empty-state), remove the dead `'NEMEA'`/subtitle assertions, and ensure the entire suite is green. Any other broken/stale tests in either repo are fixed as part of "leave the tests tidy."
   - Acceptance: `npm test` exits 0 in BOTH `nemea-front` and `nemea-back`; the rewritten home test asserts cards render for a mocked permission set and the empty-state renders when no permissions are granted.

5. **Error and loading boundaries in (app)/**: The app degrades gracefully instead of crashing on SSR/fetch failure.
   - Current: no `error.tsx` or `loading.tsx` exists under `(app)/`; a failed server fetch crashes Next.
   - Target: add `error.tsx` (graceful error UI with retry) and `loading.tsx` (loading state) in the `(app)/` route group.
   - Acceptance: both files exist under `(app)/`; a simulated backend/SSR failure renders the error boundary UI rather than an unhandled Next crash.

6. **Remove process comments (prod + tests)**: Internal tracker markers are gone from all source.
   - Current: `WR-…`, `IN-A…`, and `CRITICAL: review fix` markers appear in backend and frontend production files and in test files.
   - Target: remove all such process/tracker comments from production source AND test files in both repos, preserving any genuinely explanatory comments.
   - Acceptance: `grep -rE 'WR-[0-9]|IN-A[0-9]|CRITICAL: review fix'` over `nemea-front/src` and `nemea-back/src` (including tests) returns zero matches; lint, `tsc`, and tests remain green afterward.

7. **Centralize Tiendanube slug constants (front + back, migration untouched)**: Slug literals used for branching live in one named place per repo.
   - Current: slug literals (`'pago_nube'`, `'esencial'`, `'tarjeta_debito_credito'`, etc.) are written inline where code compares/defaults on them (`calculadora.service.ts`, `GatewaySelectors.tsx`, `scenarios.service.ts`, DTOs).
   - Target: introduce a shared constants module in the frontend and one in the backend; replace the inline literals at comparison/default sites with the named constants. The Phase 10 seed migration is NOT modified (migrations are immutable historical records and must reproduce their original state independent of app constants).
   - Acceptance: application code that branches/defaults on a Tiendanube slug imports from a single constants module per repo; no bare slug literal remains in a comparison/default position in app code (migration excluded); the calculadora round-trip test still passes.

- Editing the Phase 10 seed migration or any historical migration — immutability principle

## Constraints

- **Zero new npm dependencies** — NextAuth Credentials and JWT signing are already available; consistent with the v1.1 "no new deps" rule.
- **Demo login must be env-gated and OFF by default in real production** — a password-less admin entry must never be reachable in the real prod deployment.
- **Historical migrations are immutable** — no edits to existing migration files; the demo-user seed is a new migration or seed step, not an edit to Phase 2/Phase 10 migrations.
- **NextAuth v5 is ESM** — the jest fix is via a module mock of `@/auth` in the home test, not by weakening `transformIgnorePatterns` project-wide.
- **`PermissionsGuard` and `@RequirePermission` must not be modified** — the demo path reuses the identical JWT shape (`{ sub, email, permissions }`) so authorization is unaware of the entry method.
- **UI copy in español argentino**; technical identifiers, code, and slugs stay as-is.
