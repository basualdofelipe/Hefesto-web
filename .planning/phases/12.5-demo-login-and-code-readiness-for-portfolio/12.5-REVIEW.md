# Phase 12.5: Code Review Report

**Reviewed:** 2026-05-26
**Depth:** standard
**Files Reviewed:** 33
**Status:** issues_found

## Summary

This phase adds an env-gated demo-login auth path (backend `POST /auth/demo-login` + a frontend NextAuth Credentials provider), a demo-user seed migration, error/loading App Router boundaries, centralized Tiendanube constants, and a batch of backend/frontend tests.

The core security posture of the demo path is mostly sound: the backend `validateDemoLogin` re-checks `DEMO_LOGIN_ENABLED` server-side before issuing a JWT, so the flag cannot be bypassed by hitting the API directly, and the issued JWT carries the user's real role-derived permissions (no privilege escalation). The seed migration correctly resolves the ADMIN role id at runtime (no hardcoded UUID) and is idempotent via `ON CONFLICT DO NOTHING`.

However the demo-login is not a "demo-account-only" path: when the flag is enabled, the Credentials provider authorizes login for **any active user's email with no credential check**, and the issued token reflects that user's real permissions. That is the one finding that rises to BLOCKER. The remaining findings are robustness/quality issues — most notably an unsafe `as` cast in the NextAuth jwt callback, a stale-token cleanup branch that does not clear `permissions`, and a frontend bulk-override rounding rule that silently zeroes sub-1-peso prices.

## Critical Issues

### CR-01: Demo-login grants full access as ANY active user, not just the demo account

**File:** `nemea-back/src/auth/auth.service.ts:68-96`, `nemea-front/src/auth.ts:23-53`

**Issue:** The demo path was designed to log in the seeded `demo@nemea.app` account, but neither layer constrains it to that account. The frontend `Credentials` provider (`auth.ts:31`) passes `credentials.email` straight through to `POST /auth/demo-login`, and the backend `validateDemoLogin(email)` accepts *any* email, looks it up with `findActiveByEmail(email)`, and — if the user exists and is active — signs a JWT carrying that user's **real role permissions** (`extractPermissions(user.role)`, line 79). The `DemoLoginButton` happens to hardcode `demo@nemea.app`, but the server action `signIn('credentials', { email })` and the underlying endpoint accept arbitrary input.

Net effect: in any environment where `DEMO_LOGIN_ENABLED=true` (the portfolio deploy is the explicit intent), anyone who knows or guesses a real user's email — e.g. the seeded admin `basualdofelipe@gmail.com`, or any whitelisted Google user — can POST it to `/api/auth/demo-login` and receive a fully-privileged JWT with **no password, no Google token, no second factor**. This is a passwordless authentication bypass for every real account, not a sandboxed demo login.

The DTO `@IsEmail()` does not mitigate this — it only validates format, not that the email is the demo account.

**Fix:** Constrain the demo path to the demo account explicitly. Pin the email server-side rather than trusting the request body:

```ts
// auth.service.ts
private readonly DEMO_EMAIL = 'demo@nemea.app';

async validateDemoLogin(email: string): Promise<AuthResponseDto> {
  if (this.configService.get<string>('DEMO_LOGIN_ENABLED') !== 'true') {
    throw new UnauthorizedException('Demo login no disponible');
  }
  // Reject any email that is not the demo account — prevents logging in
  // as a real privileged user via the passwordless path.
  if (email !== this.DEMO_EMAIL) {
    throw new UnauthorizedException('Usuario demo no encontrado');
  }
  const user = await this.usersService.findActiveByEmail(email);
  ...
}
```

Optionally also assert the demo user is non-admin / read-only by seeding it with a restricted role (see CR-01 interaction with the seed below). At minimum the email allow-list is mandatory before this ships with the flag enabled.

## Warnings

### WR-01: Demo user is seeded as ADMIN — demo visitors get full management rights

**File:** `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:8-23`

**Issue:** The demo user is inserted with the `ADMIN` role (`role_id = adminRoleId`). Combined with the demo-login path, anyone using "Entrar como demo" obtains `canManageUsers`, `canManageConfig`, etc. A demo/portfolio visitor can therefore create/delete users, mutate Tiendanube tax config, and delete other users' scenarios (see `scenarios.service.ts:183` admin bypass). For a public portfolio demo this is a data-integrity and abuse risk even after CR-01 is fixed.

**Fix:** Seed the demo user with a read-only or limited role rather than ADMIN. If no suitable role exists, create a `DEMO`/`VIEWER` role in the same migration with only `canView*`/`canUseCalculator` flags set. This bounds blast radius regardless of the auth path.

### WR-02: Unsafe `as` cast in NextAuth jwt callback can store `undefined` token fields

**File:** `nemea-front/src/auth.ts:89-98`

**Issue:** In the credentials branch the code casts `user` to a fully-populated shape (`backendToken: string; permissions; userId`) with no runtime guard. The `authorize` callback can return `null` (lines 31, 41, 49), in which case NextAuth blocks sign-in — but the type assertion still represents a fragile contract: if `authorize`'s return shape ever drifts, the cast silently propagates `undefined` into `token.backendToken`/`token.permissions`, which then flows to the session and yields a session with `accessToken: ''` and `NO_PERMISSIONS` instead of a hard failure. This is the kind of "looks authenticated, silently has no permissions" state that is hard to debug.

**Fix:** Validate before assigning instead of casting:

```ts
if (account?.provider === 'credentials' && user) {
  const u = user as { backendToken?: string; permissions?: Permissions; userId?: string };
  if (u.backendToken && u.permissions && u.userId) {
    token.backendToken = u.backendToken;
    token.permissions = u.permissions;
    token.userId = u.userId;
  }
}
```

### WR-03: Stale-token cleanup clears `backendToken` but not `permissions`

**File:** `nemea-front/src/auth.ts:102-109`

**Issue:** The "stale JWT detection" branch fires when an old-format token has `role` set and `permissions === undefined`. It deletes `role` and sets `token.backendToken = undefined`, but never sets/clears `token.permissions`. Because the branch condition requires `permissions === undefined`, the subsequent `session` callback falls back to `NO_PERMISSIONS` — so the immediate effect is benign. However the branch also runs on *every* jwt invocation (not only at login), and it nulls `backendToken` without forcing a real re-auth. A user with a stale token keeps a session whose `accessToken` is `''`; `apiClientFetch` will then 401 and redirect to `/login` only on the next API call, not proactively. The comment claims it "will trigger re-auth via the signIn flow," but nothing here triggers `signIn`.

**Fix:** Make the intent explicit — either return a sentinel that invalidates the session (so middleware redirects), or document that recovery is lazy via the 401 path. At minimum align the comment with behavior and ensure `permissions` is also cleared/normalized so a half-migrated token cannot leak stale permissions if the condition logic changes.

### WR-04: Bulk-override rounding can silently zero out low prices

**File:** `nemea-front/src/app/(app)/escenarios/[id]/ScenarioEditorClient.tsx:110`

**Issue:** `const newPrice = Math.round(basePrice * (1 + percentage / 100));` rounds to the nearest integer peso. For a product priced below ~0.5 with a negative percentage adjustment, this rounds to `0`; that `0` is then written into `overridesState` and persisted as an override. A `0` override later flows through `calculate()` where `effectivePrice > 0` is false, so the product silently drops out of margin results with no warning to the user. More generally, rounding to whole pesos is a silent precision-loss decision applied uniformly; if the business uses centavos this corrupts override prices.

**Fix:** Avoid integer rounding (or round to 2 decimals), and guard against producing a non-positive override:

```ts
const raw = basePrice * (1 + percentage / 100);
const newPrice = Math.round(raw * 100) / 100;
if (newPrice <= 0) continue; // skip rather than persist a 0 override
newOverrides[product.id] = newPrice;
```

### WR-05: Demo Credentials provider is always registered regardless of the env flag

**File:** `nemea-front/src/auth.ts:23-53`, `nemea-front/src/app/(auth)/login/page.tsx:79`

**Issue:** The login page hides the demo button unless `NEXT_PUBLIC_DEMO_LOGIN_ENABLED === 'true'`, but the `Credentials` provider and the `signIn('credentials', ...)` server action in `DemoLoginButton` are registered unconditionally in `auth.ts`. The provider's NextAuth route (`/api/auth/callback/credentials`) is therefore live even when the UI hides the button. Real gating depends entirely on the backend `DEMO_LOGIN_ENABLED` check (which is correct), but the frontend exposes an always-on credentials surface that does nothing useful when the backend flag is off and adds attack surface. Defense-in-depth: the provider should not exist when demo is disabled.

**Fix:** Conditionally include the provider based on the server-side env flag:

```ts
const providers = [Google];
if (process.env.DEMO_LOGIN_ENABLED === 'true') {
  providers.push(Credentials({ ... }));
}
```

Use the server-only `DEMO_LOGIN_ENABLED` (not the `NEXT_PUBLIC_` mirror) since `auth.ts` runs server-side.

### WR-06: `findActiveByEmail` is case-sensitive — demo/whitelist lookup can fail or be bypassed by case

**File:** `nemea-back/src/users/users.service.ts:30-34`

**Issue:** `findActiveByEmail` matches `email` with an exact equality `where: { email, isActive: true }`. Email is case-insensitive by RFC for the domain part and conventionally treated case-insensitively for the local part too. A user seeded/whitelisted as `Demo@Nemea.app` would not match a login attempt for `demo@nemea.app`, and conversely the auth allow-list semantics depend on exact stored casing. This affects both the Google whitelist path and the demo path. Not a vulnerability on its own, but it makes the CR-01 fix (`email !== DEMO_EMAIL`) brittle if any caller normalizes casing differently.

**Fix:** Normalize email to lowercase on write (create/seed) and on lookup, or use a case-insensitive comparison (`LOWER(email) = LOWER(:email)`). Apply consistently so the demo allow-list check and the seed agree on casing.

## Info

### IN-01: Tiendanube constants duplicated across front and back

**File:** `nemea-back/src/constants/tiendanube.ts:1-3`, `nemea-front/src/constants/tiendanube.ts:1-4`

**Issue:** The two constant files define the same three slugs (`TN_PLAN_ESENCIAL`, `TN_GATEWAY_PAGO_NUBE`, `TN_PAYMENT_TARJETA`) with values that must stay in lockstep across repos, but they are physically separate (and even ordered differently). A change to a slug on one side silently desyncs the calculadora rate matching (`calculadora.service.ts:77-83` matches `gateway.slug === gatewaySlug`). Acceptable given the two-repo split, but there is no test or comment cross-referencing them.

**Fix:** Add a brief comment in each file noting the counterpart must be kept in sync, or extract to a shared package if the monorepo gains one. Low priority.

### IN-02: `paymentMethod` default hardcoded as a string literal instead of the constant

**File:** `nemea-front/src/app/(app)/escenarios/[id]/ScenarioEditorClient.tsx:56`

**Issue:** `paymentMethod: scenario.paymentMethod ?? 'tarjeta_debito_credito'` uses a raw literal while the file already imports `TN_GATEWAY_PAGO_NUBE` and `TN_PLAN_ESENCIAL` from `@/constants/tiendanube`. The matching `TN_PAYMENT_TARJETA` constant exists in that file but is not used here, defeating the centralization this phase introduced.

**Fix:** Import and use `TN_PAYMENT_TARJETA`.

### IN-03: `apiClientFetch` 401 handler does nothing useful on the server

**File:** `nemea-front/src/lib/api-client.ts:20-26`

**Issue:** The 401 branch only redirects when `typeof window !== 'undefined'`; on the server it throws `'Sesion expirada'`. The file is marked `'use client'`, so this is mostly fine, but if it is ever invoked from a server action the user gets an opaque thrown error with no redirect. Minor.

**Fix:** Document that this helper is client-only, or surface a typed error the caller can map to a redirect server-side.

### IN-04: `error.tsx` shows only a digest code, never the message — and ignores the unused `error` for users

**File:** `nemea-front/src/app/(app)/error.tsx:16-19`

**Issue:** The boundary displays `error.digest` when present, otherwise a generic string. In development `error.digest` is typically undefined, so developers never see the actual error message in the UI (must check console). Acceptable for production (avoids leaking internals) but reduces debuggability during the portfolio-readiness work this phase targets.

**Fix:** Optionally render `error.message` when `process.env.NODE_ENV === 'development'`. Low priority.
