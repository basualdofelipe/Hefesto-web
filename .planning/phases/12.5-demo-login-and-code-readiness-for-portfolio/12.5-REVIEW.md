## Summary

This is a re-review of phase 12.5 after fixes were applied to the prior review's findings. The verdict on each prior finding is tracked below; new findings carry fresh IDs.

**CR-01 (privilege escalation via demo-login) is genuinely RESOLVED.** `validateDemoLogin` now hard-pins the path to `DEMO_USER_EMAIL` (`'demo@nemea.app'`): it rejects any other email with `UnauthorizedException` *before* any DB lookup (`auth.service.ts:81-83`), and then looks up only the pinned constant — never the client-supplied address (`auth.service.ts:86`). The backend env-gate (`DEMO_LOGIN_ENABLED !== 'true'`) still runs server-side first (`auth.service.ts:76-79`), and the `auth.service.spec.ts` suite proves both properties (a foreign email is rejected without `findActiveByEmail` ever being called, lines 285-294; the lookup uses the pinned address, lines 296-306). The passwordless-as-any-user bypass is closed.

The remaining picture: one finding still rises to BLOCKER — the demo user is seeded as **ADMIN**, so the now-correctly-scoped demo login still hands a public visitor full management rights (create/delete users, mutate tax config, admin-bypass scenario deletion). This was WR-01 in the prior review but, with CR-01 fixed, the demo path is now the *intended* entry point for untrusted portfolio visitors, which elevates the over-privileged seed from "warning" to "blocker": it is the one defect that lets an anonymous visitor cause real data loss. The rest are robustness/quality issues that survived the fix pass (unsafe cast in the jwt callback, always-on Credentials provider, bulk-override integer rounding, case-sensitive email lookup).

## Critical Issues

### CR-01: Demo user is seeded as ADMIN — public demo visitors get full management rights

> **DISMISSED — false positive (clone-and-run local deployment, not a hosted public demo). The demo ADMIN role is intentional; the cloner is their own admin on their own DB. See Author disposition at the top of this report.**

**File:** `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:8-23`

**Issue:** The seed migration resolves the **ADMIN** role id (`SELECT id FROM "roles" WHERE "name" = 'ADMIN'`) and inserts `demo@nemea.app` with it. Now that the prior CR-01 is fixed, `validateDemoLogin` deliberately mints a JWT for exactly this account, and the login page exposes "Entrar como demo" to anyone whenever `NEXT_PUBLIC_DEMO_LOGIN_ENABLED === 'true'` — which is the explicit intent for the portfolio deploy. The JWT carries `extractPermissions(user.role)` (`auth.service.ts:91`), i.e. the full ADMIN permission set: `canManageUsers`, `canManageConfig`, etc.

Concrete blast radius for an anonymous visitor logged in as demo:
- Create and delete other users (`UsersService.create` / `remove`), bounded only by the last-admin guard.
- Mutate Tiendanube tax/commission config (`canManageConfig`).
- Delete *any* user's scenarios via the admin bypass keyed on `canManageUsers` (`scenarios.service.ts:183-184`) — including scenarios shared by the real owner.
- The demo account itself counts as an active admin in `countOtherActiveAdmins`, so a malicious visitor could attempt to demote/deactivate the real seeded admin (`basualdofelipe@gmail.com`) as long as the demo admin keeps the system above the one-admin floor.

For a public portfolio demo this is a data-integrity and abuse risk, not a sandboxed read-only experience. A `USER` role already exists in the prior migration (`1773000000000-CreateRolesAndMigrateUsers.ts:48-61`) with read + calculator + scenarios + dashboard and all `canEdit*`/`canManage*` flags false — exactly the right shape for a demo.

**Fix:** Seed the demo user against the `USER` role (or a dedicated `DEMO`/`VIEWER` role) instead of ADMIN:

```ts
const roleResult = await queryRunner.query(
  `SELECT id FROM "roles" WHERE "name" = 'USER' LIMIT 1`,
);
const roleId = (roleResult[0] as { id: string } | undefined)?.id;
if (!roleId) {
  throw new Error(
    'USER role not found — run CreateRolesAndMigrateUsers1773000000000 first',
  );
}
await queryRunner.query(
  `INSERT INTO "users" ("email", "name", "is_active", "role_id")
   VALUES ('demo@nemea.app', 'Demo Nemea', true, $1)
   ON CONFLICT ("email") DO NOTHING`,
  [roleId],
);
```

Note the `ON CONFLICT DO NOTHING` means re-running the migration after this change will **not** downgrade an already-seeded ADMIN demo row — verify the demo row's role in the deployed DB and correct it manually (or write a follow-up migration) before enabling the flag in production.

## Warnings

### WR-01: Unsafe `as` cast in the NextAuth jwt callback can silently store `undefined`

**File:** `nemea-front/src/auth.ts:89-98`

**Issue:** The credentials branch casts `user` to a fully-populated shape (`{ backendToken: string; permissions: Permissions; userId: string }`) with no runtime guard, then assigns each field onto `token`. The `authorize` callback returns either that exact shape or `null`, and NextAuth blocks sign-in on `null`, so today this works. But the cast encodes a non-load-bearing contract: if `authorize`'s return shape ever drifts (e.g. a field renamed, or a partial-success path added), the assertion silently propagates `undefined` into `token.backendToken`/`token.permissions`. That then flows through the `session` callback's `?? ''` / `?? NO_PERMISSIONS` fallbacks (`auth.ts:114-117`) and yields a session that *looks* authenticated but has an empty access token and no permissions — a hard-to-debug "logged in but can do nothing / every API call 401s" state instead of a clean failure.

**Fix:** Validate before assigning:

```ts
if (account?.provider === 'credentials' && user) {
  const u = user as {
    backendToken?: string;
    permissions?: Permissions;
    userId?: string;
  };
  if (u.backendToken && u.permissions && u.userId) {
    token.backendToken = u.backendToken;
    token.permissions = u.permissions;
    token.userId = u.userId;
  }
}
```

### WR-02: Stale-token cleanup clears `backendToken` but never normalizes `permissions`; comment misstates behavior

**File:** `nemea-front/src/auth.ts:100-109`

**Issue:** The stale-JWT branch fires when an old token has `role` set and `permissions === undefined`. It deletes `role` and sets `token.backendToken = undefined`, but never touches `token.permissions`. Today the effect is benign because the branch condition already requires `permissions === undefined` and the `session` callback falls back to `NO_PERMISSIONS`. Two problems remain:
1. The inline comment claims clearing the backend token "will trigger re-auth via the signIn flow" — nothing here calls `signIn`. Recovery is actually lazy: the next `apiClientFetch` sees an empty `accessToken`, the backend returns 401, and `api-client.ts:20-26` redirects to `/login`. The comment misdescribes the mechanism and will mislead a future maintainer.
2. The branch runs on *every* jwt invocation, not just login. It is defensive code whose correctness depends on the exact `permissions === undefined` guard; if that guard is ever loosened, a half-migrated token could leak stale `permissions` because they are never explicitly cleared.

**Fix:** Align the comment with the actual lazy-401 recovery path, and explicitly normalize `permissions` (e.g. `token.permissions = undefined`) inside the branch so the cleanup is self-contained rather than dependent on the entry condition.

### WR-03: Bulk-override integer rounding can silently produce a 0 override that drops the product from results

**File:** `nemea-front/src/app/(app)/escenarios/[id]/ScenarioEditorClient.tsx:114`

**Issue:** `const newPrice = Math.round(basePrice * (1 + percentage / 100));` rounds to whole pesos and is written unconditionally into `overridesState` (line 115) and later persisted. For a low base price with a sufficiently negative `percentage`, the result rounds to `0`. That `0` is saved as an override; on the backend `calculate()` the guard `effectivePrice !== null && effectivePrice > 0` (`scenarios.service.ts:317`) is then false, so the product silently disappears from the margin results with no feedback to the user. Separately, rounding every adjusted price to whole pesos is a uniform precision-loss decision — if the business prices in centavos it corrupts the override.

**Fix:** Round to 2 decimals and skip non-positive results instead of persisting them:

```ts
const raw = basePrice * (1 + percentage / 100);
const newPrice = Math.round(raw * 100) / 100;
if (newPrice <= 0) continue; // don't persist a 0 override that vanishes from results
newOverrides[product.id] = newPrice;
count++;
```

### WR-04: Demo Credentials provider is registered unconditionally regardless of the env flag

**File:** `nemea-front/src/auth.ts:21-54`, `nemea-front/src/app/(auth)/login/page.tsx:79`

**Issue:** The login page hides the demo button unless `NEXT_PUBLIC_DEMO_LOGIN_ENABLED === 'true'`, but the `Credentials` provider in `auth.ts` is always in the `providers` array. That means the NextAuth credentials callback route (`/api/auth/callback/credentials`) and the `signIn('credentials', ...)` server action in `DemoLoginButton` are live even when the UI hides the button. Real gating still holds because the backend `validateDemoLogin` re-checks `DEMO_LOGIN_ENABLED` and returns 401 (and the frontend `authorize` returns `null` on `!res.ok`), so this is not an exploitable bypass — it is an unnecessary always-on auth surface that contradicts the env-gating model. Defense-in-depth: the provider should not exist when demo is disabled.

**Fix:** Build the provider list conditionally using the server-only flag (this file runs server-side, so use `DEMO_LOGIN_ENABLED`, not the `NEXT_PUBLIC_` mirror):

```ts
const providers = [Google];
if (process.env.DEMO_LOGIN_ENABLED === 'true') {
  providers.push(Credentials({ /* ... */ }));
}
export const { handlers, auth, signIn, signOut } = NextAuth({ providers, /* ... */ });
```

### WR-05: Flagship calculadora/scenario tests assert presence, not the computed value (senior audit)

**File:** `nemea-front/src/app/(app)/calculadora/__tests__/CalculadoraClient.test.tsx:168-177`, `nemea-front/src/app/(app)/escenarios/[id]/__tests__/ScenarioEditorClient.test.tsx:245-258`

**Issue:** The two new front tests verify the *correct endpoint* is hit (`/api/calculadora/forward` vs `/inverse`; `PUT /overrides`) and method (`POST`/`PUT`) — that half is good and meaningful. But neither asserts the **request payload** nor the **computed result value**. The mocked `DesglosePanel` is checked only for presence (`getByTestId('desglose')`), and `MarginSummary` only for `results.length` (`'1 results'`). The mock `fetch` returns a fixed body (`ganancia: 30, margen: 0.3, precioVenta: 130` / `simResult.margen: 30`), so a regression that computed or forwarded the *wrong* number (wrong price in the request body, wrong product id, wrong gateway slug mapping) would still pass green. For the flagship feature of a portfolio repo where test quality is the primary signal, this is a meaningful gap — the test proves "a request went out and a panel rendered", not "the calculation is correct end to end". This is NOT the classic mock-stubs-contract anti-pattern (the logic under test lives in the client, not the mocked child) — the only weakness is the assertion strength. Ref senior PM "Execute and check later" item 4.

**Fix:** Assert the request body carries the entered values (e.g. `JSON.parse(call[1].body)` contains `precioVenta: 130` / the selected `productId` / `gatewaySlug: 'pago_nube'`), and assert the mocked child receives/renders the concrete returned value (pass `results`/`ganancia` into a visible node in the mock and assert on it, not just `length`). Low effort, high signal for the calculadora suite.

## Info

### IN-01: `findActiveByEmail` is case-sensitive

**File:** `nemea-back/src/users/users.service.ts:30-34`

**Issue:** `findActiveByEmail` matches with exact equality (`where: { email, isActive: true }`). Email comparison is conventionally case-insensitive; a user seeded/whitelisted as `Demo@Nemea.app` would not match a login for `demo@nemea.app`. With CR-01 fixed, the demo path compares against the lowercase literal `DEMO_USER_EMAIL` and the seed inserts lowercase `demo@nemea.app`, so the demo path itself is internally consistent — hence this is downgraded from the prior WR-06 to Info. It still affects the Google whitelist path (`validateGoogleToken` → `findActiveByEmail(payload.email)`): a user whitelisted with different casing than Google returns would fail to match.

**Fix:** Normalize email to lowercase on write (create/seed) and on lookup, or compare with `LOWER(email) = LOWER(:email)`. Apply consistently across both auth paths.

### IN-02: Tiendanube constants duplicated across front and back with no sync guard

**File:** `nemea-back/src/constants/tiendanube.ts:1-3`, `nemea-front/src/constants/tiendanube.ts:1-3`

**Issue:** Both files define the same three slugs (`TN_PLAN_ESENCIAL`, `TN_GATEWAY_PAGO_NUBE`, `TN_PAYMENT_TARJETA`) with values that must stay in lockstep across the two repos (the calculadora rate-matching in `calculadora.service.ts:77-83` and the scenario defaults in `scenarios.service.ts:281-285` both depend on the slug strings matching what the frontend sends). They are physically separate and even ordered differently between files, with no test or comment cross-referencing them. Acceptable given the two-repo split, but a slug change on one side silently desyncs rate matching.

**Fix:** Add a brief comment in each file noting the counterpart must be kept in sync (or extract to a shared package if the monorepo gains one). Low priority.

### IN-03: `error.tsx` shows only the digest, never the message

**File:** `nemea-front/src/app/(app)/error.tsx:16-19`

**Issue:** The boundary renders `error.digest` when present, otherwise a generic string, and never `error.message`. In development `error.digest` is typically undefined, so a developer never sees the actual error in the UI and must check the console. Appropriate for production (avoids leaking internals) but reduces debuggability during this portfolio-readiness work.

**Fix:** Optionally render `error.message` when `process.env.NODE_ENV === 'development'`. Low priority.

## Narrative notes (no finding)

- **`update-user.dto.ts:33` comment is inaccurate but behavior is safe.** The comment claims `@ValidateIf((_, value) => value !== null)` "rejects explicit null at the validation layer." It does the opposite: when `roleId` is `null` the predicate returns `false`, which *skips* `@IsUUID`, so `null` passes validation and reaches the service. Safety is actually provided by the explicit service guard (`users.service.ts:131-133`, throws `BadRequestException` on `roleId === null`), not by the DTO. Not a bug — the guard catches it — but the comment should be corrected to avoid a future maintainer removing the service guard on the false belief the DTO blocks null.
- **Demo DTO `@IsEmail()` is now purely cosmetic for security.** Since `validateDemoLogin` pins the email server-side, the DTO validation no longer carries any auth weight; this is fine and worth noting only so a reviewer doesn't mistake it for the security control.
- **Test quality is solid.** `auth.service.spec.ts` directly asserts the CR-01 fix (foreign email rejected with no lookup, pinned lookup on success). `EditUserDialog.test.tsx` and `ScenarioEditorClient.test.tsx` use real `Headers` instances matching the `api-client.ts` contract and pin documented edge behaviors (empty-token PATCH on loading/unauthenticated sessions).
