<decisions>
## Implementation Decisions

### Demo login entry
- **D-01:** One-click, fixed demo identity (passwordless). The "Entrar como demo" button internally calls the NextAuth Credentials login with a known demo email ‚Äî no email/password form. The backend `POST /api/auth/demo-login` already takes email only; the Credentials provider sends the known demo email, no password field.
- **D-02:** Button label is exactly **"Entrar como demo"**, on `/login`, reusing the existing login page layout (secondary to the Google button).
- **D-03:** Demo entry reuses the existing auth path end-to-end ‚Äî JWT shape `{ sub, email, permissions }` is identical to Google login, so `PermissionsGuard` / `@RequirePermission` are untouched (carried forward from Phase 12.1).

### Demo gating / prod safety
- **D-04:** Frontend detects demo availability via build-time env var **`NEXT_PUBLIC_DEMO_LOGIN_ENABLED`** (mirror of the backend `DEMO_LOGIN_ENABLED`). The button is hidden entirely when off ‚Äî no disabled/failing button.
- **D-05:** In real production both flags are `false`. Backend endpoint rejects (4xx) when its flag is off even if the front somehow calls it ‚Äî defense in depth, not just hidden UI.

### Backend spec style
- **D-06:** The 3 new service specs (`products.service`, `supplies.service`, `tiendanube-config.service`) are **unit tests with mocked repositories**, matching the existing convention (`users`/`roles`/`scenarios` specs). This is the correct layer of the test pyramid for service-logic specs and keeps `npm test` infra-free.
- **D-07:** Integration-against-real-Postgres is NOT added to these specs ‚Äî that layer is already demonstrated by the Phase 12.4 e2e suite (3/3 against real Postgres). Mixing integration into unit specs would blur the pyramid.

### Frontend test style
- **D-08:** `CalculadoraClient` and `ScenarioEditorClient` tests follow the existing front pattern (RTL, behavior-focused, mock the API layer), reusing `jest.setup.ts` Radix polyfills (carried forward from Phase 12.4 P03). Home test uses `render(await Home())` + `jest.mock('@/auth')` to both cover the permission-card logic and short-circuit the NextAuth v5 ESM import chain.

### Error / loading UX
- **D-09:** `error.tsx` shows a **branded Nemea** error message in Spanish with a **"Reintentar"** button wired to the App Router `reset()`.
- **D-10:** `loading.tsx` shows a **centered spinner** (not per-screen skeletons) ‚Äî consistent, low-maintenance.

### Claude's Discretion
- Exact location/name of the shared Tiendanube constants module per repo (e.g., `nemea-front/src/constants/tiendanube.ts`, `nemea-back/src/tiendanube-config/constants.ts`).
- Seed mechanism for the demo user (new migration vs seed step) ‚Äî lean to a new migration matching the existing admin-seed pattern; never edit a historical migration.
- Specific assertions/cases inside each new spec (happy/error/edge), provided they assert behavior not call-spying.

</decisions>
