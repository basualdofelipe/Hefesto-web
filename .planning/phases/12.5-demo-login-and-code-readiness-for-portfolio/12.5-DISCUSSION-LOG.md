## Estilo de tests backend

| Option | Description | Selected |
|--------|-------------|----------|
| Unit con repos mockeados | Patrón existente (users/roles/scenarios), sin DB | ✓ |
| Integración con Postgres real | Como e2e de 12.4, más lento, requiere DB | |
| Mixto | Unit + algunos de integración | |

**User's choice:** Unit con repos mockeados (tras analizar el tema de CI/Postgres).
**Notes:** El usuario preguntó cómo corre Postgres en CI (GitHub Actions service container) y empujó sobre "lo más profesional". Se reconcilió con la pirámide de tests: unit es la capa correcta para specs de lógica de servicio; integración ya está demostrada por el e2e de 12.4; el flex de CI+Postgres+badge se agenda como fase 12.6 dedicada. Senior reconoció que el framing inicial ("docker rompe clone-and-run") estaba flojo — el motivo real es la separación unit/integración de la pirámide, no evitar Docker.
