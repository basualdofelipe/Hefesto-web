# Phase 12.6: Branding configurable por env — Specification

**Created:** 2026-06-01
**Ambiguity score:** 0.07 (gate: ≤ 0.20)
**Requirements:** 10 locked

## Goal

El nombre de la app y la identidad del usuario demo dejan de estar hardcodeados ("Nemea") y pasan a leerse de variables de entorno con fallback en código (default **Hefesto** / **demo@hefesto.com**), de modo que rebrandear = setear envs + reemplazar el logo + rebuild.

## Background

La marca está hardcodeada en varios puntos visibles del front y en la identidad del demo:
- `nemea-front/src/app/layout.tsx:37` → `title: 'Nemea'` (objeto `metadata` estático).
- 3 `alt='NEMEA'` en `login/page.tsx:58`, `AppSidebar.tsx:83`, `:90`.
- `acceso-denegado/page.tsx` → heading "NEMEA" (:17), texto "…acceder a Nemea…" (:23), y mail de contacto hardcodeado "…en `basualdofelipe@gmail.com`" (:24, **visible al usuario bloqueado**).
- `NEXT_PUBLIC_APP_NAME=Nemea` YA existe en `nemea-front/.env.example` pero está **muerto** (no se consume en código).
- Back: `auth.service.ts:15` `DEMO_USER_EMAIL = 'demo@nemea.app'`; migración `1773200000000-SeedDemoUser.ts` seedea `'demo@nemea.app'` / `'Demo Nemea'`; `demo-login.dto.ts` con `@IsEmail()`; `main.ts` Swagger `.setTitle('Nemea API')` + `.setDescription('Nemea leather goods…')`.
- Front: `DemoLoginButton.tsx:8` `DEMO_EMAIL = 'demo@nemea.app'` hardcodeado.
- Tests: `auth.service.spec.ts` hardcodea `'demo@nemea.app'` en 7 asserts.

Motivación: el deploy público no debe decir "Nemea" (marca registrada por un tercero para marroquinería) y la app debe poder revenderse rebrandeando por env (white-label ready). El logo se reemplaza como asset a mano (no entra por env/DB en esta fase).

## Requirements

1. **App name por env (front)**: el nombre visible del front sale de `NEXT_PUBLIC_APP_NAME` con fallback en código.
   - Current: `title` (layout.tsx:37), 3 `alt='NEMEA'` (login + sidebar), el heading "NEMEA" (acceso-denegado:17) y el texto "…acceder a Nemea…" (acceso-denegado:23) hardcodeados; `NEXT_PUBLIC_APP_NAME` existe pero no se usa.
   - Target: todos esos puntos renderizan `process.env.NEXT_PUBLIC_APP_NAME ?? 'Hefesto'`.
   - Acceptance: con el env sin setear, todos renderizan "Hefesto"; con `NEXT_PUBLIC_APP_NAME=Foo` (rebuild), todos renderizan "Foo"; `grep -i nemea` sobre `nemea-front/src` (excluyendo tests) no devuelve la marca.

2. **App name por env (back Swagger)**: el título de Swagger sale del env.
   - Current: `main.ts` → `.setTitle('Nemea API')` y `.setDescription('Nemea leather goods cost management API')` hardcodeados (Swagger es dev-only, off en prod).
   - Target: title y description usan `process.env.APP_NAME ?? 'Hefesto'`.
   - Acceptance: con `APP_NAME` sin setear, el título de Swagger = "Hefesto API"; con `APP_NAME=Foo` → "Foo API"; la description ya no contiene "Nemea".

3. **Demo email por env (back)**: el endpoint demo-login se pinea al email del env.
   - Current: `auth.service.ts:15` const `DEMO_USER_EMAIL = 'demo@nemea.app'`; no está en `env.validation.ts`.
   - Target: `DEMO_USER_EMAIL = process.env.DEMO_EMAIL ?? 'demo@hefesto.com'`; `DEMO_EMAIL` agregado a `env.validation.ts` como `Joi.string().email().default('demo@hefesto.com')`.
   - Acceptance: demo-login acepta el email del env y rechaza cualquier otro (401/403); con `DEMO_EMAIL=demo@foo.com`, acepta `demo@foo.com`.

4. **Demo email por env (front)**: el botón demo usa `NEXT_PUBLIC_DEMO_EMAIL`.
   - Current: `DemoLoginButton.tsx:8` `DEMO_EMAIL = 'demo@nemea.app'` hardcodeado.
   - Target: `process.env.NEXT_PUBLIC_DEMO_EMAIL ?? 'demo@hefesto.com'`.
   - Acceptance: el botón postea el email del env; con `NEXT_PUBLIC_DEMO_EMAIL=demo@foo.com` (rebuild) postea `demo@foo.com`. Default del front == default del back (`demo@hefesto.com`).

5. **Nombre del demo derivado de APP_NAME**: la migración seedea `Demo ${APP_NAME}` y el email del env (seed-from-env).
   - Current: `SeedDemoUser` seedea `'demo@nemea.app'` / `'Demo Nemea'` hardcodeados.
   - Target: la migración seedea email `process.env.DEMO_EMAIL ?? 'demo@hefesto.com'` y name `Demo ${process.env.APP_NAME ?? 'Hefesto'}`; el `down` borra `WHERE email = <mismo valor del env>`.
   - Acceptance: en DB **fresca** con defaults, la fila demo es `demo@hefesto.com` / "Demo Hefesto"; con `APP_NAME=Foo` + `DEMO_EMAIL=demo@foo.com` → "Demo Foo" / `demo@foo.com`.

6. **Validez del email**: el `DEMO_EMAIL` elegido debe pasar `@IsEmail` + validación de boot.
   - Current: `DemoLoginDto` tiene `@IsEmail()` + ValidationPipe global; nada valida `DEMO_EMAIL` al bootear.
   - Target: `env.validation.ts` rechaza un `DEMO_EMAIL` con formato inválido; defaults y ejemplos de docs usan emails válidos.
   - Acceptance: `DEMO_EMAIL=demo@demo` (inválido, sin TLD) hace fallar la validación al bootear; `demo@hefesto.com` bootea OK.

7. **Tests actualizados + test de override**: los asserts apuntan a la fuente única y un test nuevo prueba el override.
   - Current: `auth.service.spec.ts` hardcodea `'demo@nemea.app'` en 7 asserts.
   - Target: los 7 asserts referencian la constante/env; test nuevo: con `DEMO_EMAIL=demo@foo.com`, `validateDemoLogin('demo@foo.com')` resuelve y otro email se rechaza.
   - Acceptance: `npm test` (back) verde; el test de override **falla** si el código hardcodea el email en vez de leer el env.

8. **Docs de rebranding**: README explica cómo rebrandear; `.env.example` y refs cosméticas actualizadas.
   - Current: `.env.example` (front) trae `NEXT_PUBLIC_APP_NAME=Nemea`; comments mencionan `demo@nemea.app`; `ARCHITECTURE.md:116` y `demo-login.dto.ts:6` mencionan `demo@nemea.app`.
   - Target: `.env.example` (front+back) con defaults Hefesto / `demo@hefesto.com` y comments; sección **"Rebranding"** en el README (setear `APP_NAME` + `DEMO_EMAIL` + sus `NEXT_PUBLIC_*` + reemplazar el logo + rebuild); `ARCHITECTURE.md` y el `example` del DTO actualizados.
   - Acceptance: el README tiene una sección "Rebranding" que lista las env vars + el swap del logo + el requisito de rebuild; sin "Nemea"/"demo@nemea.app" en source/docs (grep, excluyendo historial git).

9. **App admin por env**: el primer usuario admin se seedea desde env con defaults neutros (coherencia white-label).
   - Current: migración 1 `CreateUserTable` hardcodea `'basualdofelipe@gmail.com'` / `'Felipe Basualdo'` / `'admin'`.
   - Target: la migración seedea email `process.env.ADMIN_EMAIL ?? 'admin@hefesto.com'` y name `process.env.ADMIN_NAME ?? 'Admin'` (query parametrizada); `ADMIN_EMAIL` agregado a `env.validation.ts` como `Joi.string().email().default('admin@hefesto.com')`.
   - Acceptance: en DB fresca sin envs, el admin seedeado es `admin@hefesto.com` / "Admin"; con `ADMIN_EMAIL=x@y.com` + `ADMIN_NAME=Foo` → seedea esos; `ADMIN_EMAIL` con formato inválido falla la validación de boot.

10. **Contact email por env (front)**: el mail de contacto mostrado al usuario bloqueado sale de env.
   - Current: `acceso-denegado/page.tsx:24` hardcodea "contacta al administrador en `basualdofelipe@gmail.com`" — visible a cualquier usuario bloqueado (incluido el demo público).
   - Target: el texto usa `process.env.NEXT_PUBLIC_CONTACT_EMAIL ?? 'admin@hefesto.com'`.
   - Acceptance: con el env sin setear muestra `admin@hefesto.com`; con `NEXT_PUBLIC_CONTACT_EMAIL=x@y.com` (rebuild) muestra `x@y.com`; `grep -i basualdofelipe` sobre `nemea-front/src` no devuelve nada.

## Boundaries

**In scope:**
- `APP_NAME` por env: texto visible del front (title, 3 alt, heading + texto de acceso-denegado) + título y descripción de Swagger del back, con fallback `?? 'Hefesto'`.
- `DEMO_EMAIL` (back) + `NEXT_PUBLIC_DEMO_EMAIL` (front), fallback `?? 'demo@hefesto.com'`.
- Nombre del demo derivado: `Demo ${APP_NAME}`.
- `SeedDemoUser` migración editada a seed-from-env (aplica en DB fresca).
- `DEMO_EMAIL` en `env.validation.ts` (`.email()`, default).
- `ADMIN_EMAIL` + `ADMIN_NAME` por env en la migración 1 `CreateUserTable` (defaults neutros `admin@hefesto.com` / `Admin`) + `ADMIN_EMAIL` en `env.validation.ts`.
- `NEXT_PUBLIC_CONTACT_EMAIL` por env (default `admin@hefesto.com`) para el mail de contacto en `acceso-denegado:24`.
- Actualizar los 7 asserts de `auth.service.spec.ts` + agregar test de override.
- `.env.example` (front+back) + `ARCHITECTURE.md` + sección "Rebranding" en README.

**Out of scope:**
- Logo/branding en **DB + UI de upload** — otra arquitectura (file storage, serving, admin UI); innecesario para resale de un comprador.
- Logo por env (path/URL) — fricción con next/image; el logo queda como **asset swappable** que el user reemplaza a mano.
- Reemplazar los archivos de imagen del logo — lo hace el user fuera de esta fase.
- Rename interno (package.json, `nemea_db` en docker-compose, carpetas locales, repo GitHub) — no visible en el deploy; riesgo de romper deploy/lockfiles/`DATABASE_URL` por cero beneficio.
- Migración de reconciliación para DB ya seedeadas — decisión del user: las DB actuales son descartables (se wipean); solo se editan `SeedDemoUser` y la migración 1 `CreateUserTable`.

## Constraints

- **Cero deps npm nuevas** (regla del proyecto v1.1) — se usa `process.env` + Joi/ConfigService existentes.
- **`NEXT_PUBLIC_*` se inlinea en build time** → rebrandear requiere rebuild del front (documentar).
- **`DEMO_EMAIL` debe ser email válido** (`@IsEmail` + ValidationPipe + Joi `.email()`).
- **Todo valor por env DEBE tener fallback en código** (`?? 'Hefesto'` / `?? 'demo@hefesto.com'`) → la app nunca renderiza `undefined` ni falla al bootear si falta un env.
- **Seed-from-env aplica en DB fresca solamente**; las DB ya seedeadas se wipean (sin reconciliación).
- **Defaults front == back** para el demo email (`demo@hefesto.com`).
- **`ADMIN_EMAIL` debe ser una cuenta Google real del deployer** para poder loguearse como admin (el default neutro `admin@hefesto.com` no es logueable — es solo un seed válido). Documentar en el README.

## Acceptance Criteria

- [ ] DB fresca + sin envs de branding → demo-login entra como "Demo Hefesto" / `demo@hefesto.com`; el title de la pestaña dice "Hefesto".
- [ ] DB fresca + sin envs → el admin seedeado es `admin@hefesto.com` / "Admin"; con `ADMIN_EMAIL`/`ADMIN_NAME` seteados, seedea esos valores.
- [ ] La página acceso-denegado muestra el `NEXT_PUBLIC_CONTACT_EMAIL` (default `admin@hefesto.com`); `grep -i basualdofelipe` sobre `nemea-front/src` no devuelve nada.
- [ ] Override: `APP_NAME=Foo` + `DEMO_EMAIL=demo@foo.com` + `NEXT_PUBLIC_APP_NAME=Foo` + `NEXT_PUBLIC_DEMO_EMAIL=demo@foo.com` (rebuild) → todo el texto visible + identidad del demo reflejan "Foo"/`demo@foo.com`.
- [ ] `grep -i nemea` sobre source + docs de front y back (excluyendo historial git) no devuelve la marca.
- [ ] Si falta cualquier env de branding → la app bootea y renderiza los defaults (sin `undefined`, sin crash).
- [ ] `DEMO_EMAIL` o `ADMIN_EMAIL` con formato inválido → el back falla la validación de env al bootear.
- [ ] `npm test` (back) verde; el test de override falla si el email está hardcodeado en vez de leído del env.
- [ ] El README tiene una sección "Rebranding" que lista `APP_NAME`, `DEMO_EMAIL`, sus contrapartes `NEXT_PUBLIC_*`, el paso de reemplazar el logo, y el requisito de rebuild.
