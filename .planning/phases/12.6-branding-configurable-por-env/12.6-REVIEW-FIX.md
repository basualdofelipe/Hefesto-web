## Skipped Issues

### CR-01: Demo login enabled by default grants public full-ADMIN access

**File:** `nemea-back/.env.example:28`, `nemea-front/.env.example:12`, `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:11-26`
**Reason:** skipped — confirmed FALSE POSITIVE per senior audit. This is a clone-and-run
LOCAL app: each user clones and runs against their own DB, so the "demo admin" is the
cloner's own admin on their own machine — there is no anonymous public attacker. demo=ADMIN
and `DEMO_LOGIN_ENABLED=true` in `.env.example` are intentional and user-confirmed. Per
explicit adjudication: do NOT seed a DEMO role, do NOT change the seed migration's role, do
NOT flip the `DEMO_LOGIN_ENABLED`/`NEXT_PUBLIC_DEMO_LOGIN_ENABLED` defaults. The Joi boot
default remains fail-closed `false`.
**Original issue:** Reviewer argued a fresh clone exposes a public, password-less,
full-admin account; assumes a hosted public demo (not the actual deployment model).

### WR-03: Env-default for `DEMO_EMAIL` bypassed at migration time (back/front drift risk)

**File:** `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:7`, `nemea-back/src/constants/branding.ts:29-30`
**Reason:** skipped — out of scope per senior audit. The must-match requirement is already
documented (README:269). The only suggested code change (an email-format guard inside the
migration) is optional behavioral hardening that changes migration failure behavior; per the
adjudication ("prefer SKIP-with-reason over a behavioral change"), not applied in this
branding phase.
**Original issue:** Mismatched `DEMO_EMAIL` / `NEXT_PUBLIC_DEMO_EMAIL` across back/front can
seed an unreachable demo account; a malformed `DEMO_EMAIL` seeds silently at migration time.

### WR-04: `validateDemoLogin` re-reads `DEMO_LOGIN_ENABLED` as untyped string, no normalization

**File:** `nemea-back/src/auth/auth.service.ts:70-73`
**Reason:** skipped — downgraded to minor per senior audit. The current `!== 'true'`
comparison is defensible fail-closed behavior, and Joi already validates the value to the
`'true' | 'false'` literal set at boot. The suggested normalize+log change is a behavioral
tweak to the gate; per the adjudication, skipped to avoid altering fail-closed behavior in a
branding phase.
**Original issue:** Comparison is whitespace/case-sensitive and silently fails closed with no
diagnostic for misconfigured-but-intended-enabled deployments.

### WR-05: `getProfile` / `auth/me` returns the raw TypeORM `User` entity

**File:** `nemea-back/src/auth/auth.service.ts:65-67`, `nemea-back/src/auth/auth.controller.ts:75-83`
**Reason:** skipped — explicitly OUT OF SCOPE per senior audit. This is a pre-existing

**Original issue:** Returning the full entity means any future column added to `User` leaks
through `/api/auth/me`; hardening would return an explicit DTO.
