# Phase 12.6: Branding configurable por env - Research

**Researched:** 2026-06-01
**Domain:** Env-driven configuration (white-label) — Next.js build-time inlining + NestJS/TypeORM seed-from-env + Jest env-override testing
**Confidence:** HIGH (todo verificado contra el código del repo y docs oficiales)

## Summary

Esta fase no introduce tecnología nueva: usa `process.env` + Joi (ya presentes) y replica el patrón de constantes centralizadas de la fase 12.5 (`constants/tiendanube.ts`). Todo el trabajo es mover strings hardcodeados ("Nemea", `demo@nemea.app`, `basualdofelipe@gmail.com`) a lecturas de env con fallback (`?? 'Hefesto'` / `?? 'demo@hefesto.com'` / `?? 'admin@hefesto.com'`), centralizadas en un nuevo `src/constants/branding.ts` por repo.

Hay tres mecanismos distintos en juego, cada uno con su timing y su landmine: (1) en el **front**, las `NEXT_PUBLIC_*` se **inlinean en build time** por el compilador de Next — cambiar el env sin rebuild no tiene efecto, y por eso el README debe exigir rebuild; (2) en el **back runtime** (services, Swagger en `main.ts`), `process.env` se lee al arrancar el proceso vía `ConfigModule`/dotenv; (3) en las **migraciones** TypeORM, `process.env` está poblado porque `data-source.ts` hace `import 'dotenv/config'` en la primera línea — esto es lo que habilita el seed-from-env.

El riesgo técnico principal está concentrado en **un solo lugar**: el test de override del back. `DEMO_USER_EMAIL` hoy es una `const` a nivel de módulo evaluada **una sola vez al import**. Si pasa a `process.env.DEMO_EMAIL ?? '...'`, un test que solo mute `process.env` después de importar el módulo NO verá el cambio. El patrón correcto exige `jest.resetModules()` + re-`import()` dinámico del módulo con el env ya seteado, o mover la lectura del env dentro de la función `validateDemoLogin` (no a nivel de módulo). Esta decisión condiciona si el test de override del req 7 efectivamente prueba lo que dice probar.

**Primary recommendation:** Crear `src/constants/branding.ts` en ambos repos. En el back, exportar el valor de demo como **getter/función** (no `const` congelada) o leer el env dentro de `validateDemoLogin`, para que el test de override funcione con `jest.resetModules()`. En el front, importar las `NEXT_PUBLIC_*` desde `branding.ts` y documentar rebuild en el README.

### Pattern 1: Constantes centralizadas por repo (replica de 12.5)

**What:** Un único módulo `constants/branding.ts` lee `process.env` con fallback; todos los puntos de uso importan de ahí. NO reads inline dispersos.
**When to use:** Siempre en esta fase (D-01/D-02/D-03).

```typescript
// nemea-front/src/constants/branding.ts
// NEXT_PUBLIC_* se inlinean en build time — rebuild requerido para cambios.
export const APP_NAME = process.env.NEXT_PUBLIC_APP_NAME ?? 'Hefesto';
export const DEMO_EMAIL = process.env.NEXT_PUBLIC_DEMO_EMAIL ?? 'demo@hefesto.com';
export const CONTACT_EMAIL = process.env.NEXT_PUBLIC_CONTACT_EMAIL ?? 'admin@hefesto.com';
```

> **Nota Next/inlining:** `process.env.NEXT_PUBLIC_FOO` debe escribirse con el nombre **literal completo** para que el compilador lo reemplace. NO destructurar (`const { NEXT_PUBLIC_APP_NAME } = process.env`) ni acceder dinámicamente (`process.env['NEXT_PUBLIC_' + x]`) — eso rompe el inlining y devuelve `undefined` en el cliente. [CITED: nextjs.org/docs/app/building-your-application/configuring/environment-variables]

### Pattern 2: Back branding — lectura segura para tests

**What:** En el back, evitar congelar el demo email en una `const` de módulo si un test lo va a overridear.

```typescript
// nemea-back/src/constants/branding.ts
export const APP_NAME = process.env.APP_NAME ?? 'Hefesto';
export const ADMIN_EMAIL = process.env.ADMIN_EMAIL ?? 'admin@hefesto.com';
export const ADMIN_NAME = process.env.ADMIN_NAME ?? 'Admin';

// Opción A (segura para test sin re-import): getter en vez de const congelada
export const getDemoEmail = (): string => process.env.DEMO_EMAIL ?? 'demo@hefesto.com';
// Opción B: const + el test usa jest.resetModules() + import() dinámico (ver Pitfall 1)
export const DEMO_EMAIL = process.env.DEMO_EMAIL ?? 'demo@hefesto.com';
```

**When to use:** El planner elige (Claude's Discretion en CONTEXT). Si elige `const`, el test DEBE usar `jest.resetModules()`.

### Pattern 3: Seed-from-env en migración (query parametrizada)

```typescript
// SeedDemoUser up()
const demoEmail = process.env.DEMO_EMAIL ?? 'demo@hefesto.com';
const demoName = `Demo ${process.env.APP_NAME ?? 'Hefesto'}`;
await queryRunner.query(
  `INSERT INTO "users" ("email", "name", "is_active", "role_id")
   VALUES ($1, $2, true, $3) ON CONFLICT ("email") DO NOTHING`,
  [demoEmail, demoName, adminRoleId],
);
// down(): DELETE ... WHERE "email" = $1  con [demoEmail] (mismo valor del env)
```

> **CreateUserTable** hoy usa string interpolado directo (`VALUES ('basualdofelipe@gmail.com', 'admin', ...)`). Al meter env, **parametrizar** (`$1`) — req 9 lo exige explícitamente ("query parametrizada"). Evita inyección si un día el env viene de fuente no confiable.

### Anti-Patterns to Avoid

- **Destructurar `process.env` en el front:** rompe el inlining de Next → `undefined` en cliente.
- **Leer `process.env` inline en cada sitio:** viola D-01. Centralizar en `branding.ts`.
- **Usar `ConfigService` para branding:** viola D-02 (migraciones no tienen DI; dos fuentes divergen).
- **`const DEMO_EMAIL = process.env...` + test que solo muta `process.env`:** el test pasa en falso (la const ya quedó congelada). Ver Pitfall 1.
- **String interpolado en INSERT de migración admin:** usar parámetros `$1`.

## Common Pitfalls

### Pitfall 1: El test de override no prueba nada (const congelada)

**What goes wrong:** `DEMO_USER_EMAIL` hoy es `export const DEMO_USER_EMAIL = 'demo@nemea.app'` evaluado una vez al importar el módulo. Si pasa a `process.env.DEMO_EMAIL ?? '...'` pero sigue siendo `const` de módulo, un test que haga `process.env.DEMO_EMAIL = 'demo@foo.com'` **después** de que el módulo ya se importó verá el viejo valor. El test "pasa" sin probar el override → falso verde. El req 7 exige específicamente que el test "**falle si el código hardcodea el email**".
**Why it happens:** ESM/CJS evalúan el módulo una sola vez; la const captura el valor del env del momento del primer import. Jest cachea módulos.
**How to avoid:** Dos opciones (Claude's Discretion):
- **(A) Getter/lectura inline:** leer `process.env.DEMO_EMAIL` *dentro* de `validateDemoLogin` (o vía `getDemoEmail()`), no en una const de módulo. El test setea `process.env` antes de llamar la función. Más simple, sin re-import.
- **(B) Re-import dinámico:** mantener la const pero en el test:
```typescript
it('respeta DEMO_EMAIL del env (override)', async () => {
  process.env.DEMO_EMAIL = 'demo@foo.com';
  jest.resetModules();                              // limpia cache de módulos
  const { AuthService } = await import('./auth.service'); // re-evalúa con env nuevo
  // ... recrear el TestingModule con el AuthService re-importado
});
```
**Warning signs:** El test pasa aunque borres la lectura del env y vuelvas a hardcodear → el test no sirve.

### Pitfall 2: NEXT_PUBLIC_* no cambia sin rebuild

**What goes wrong:** Cambian `NEXT_PUBLIC_APP_NAME` en Vercel pero la app sigue diciendo el valor viejo.
**Why it happens:** Next inlinea estas vars en build time, no en runtime.
**How to avoid:** Documentar rebuild obligatorio en el README (D-04, las 2 líneas son no-negociables). Acceptance de req 1/4/10 incluye "(rebuild)".
**Warning signs:** Cambio de env sin redeploy → texto viejo persiste.

### Pitfall 3: Mock de ConfigService en el spec no cubre DEMO_LOGIN_ENABLED

**What goes wrong:** `validateDemoLogin` hace `this.configService.get('DEMO_LOGIN_ENABLED')`. El spec usa `mockConfigService = { get: jest.fn() }`. Si el test de override no mockea `get` para devolver `'true'`, `validateDemoLogin` corta antes (línea 76-78) y nunca llega a comparar el email.
**Why it happens:** El demo-login está gateado por `DEMO_LOGIN_ENABLED`. [VERIFIED: auth.service.ts:75-86]
**How to avoid:** En el test de override, `mockConfigService.get.mockReturnValue('true')` (o mock por key) antes de llamar `validateDemoLogin`.
**Warning signs:** El test tira "demo login deshabilitado" en vez de probar el email.

### Pitfall 4: `down()` de la migración borra la fila equivocada

**What goes wrong:** El `down` de `SeedDemoUser` hace `DELETE WHERE email='demo@nemea.app'` hardcodeado. Si el `up` ahora seedea desde env, el `down` debe borrar el **mismo** valor del env, no el viejo string.
**Why it happens:** Asimetría up/down.
**How to avoid:** `down` usa la misma expresión `process.env.DEMO_EMAIL ?? 'demo@hefesto.com'` (req 5 lo exige).

### Pitfall 5: grep de verificación atrapa el historial git / node_modules

**What goes wrong:** Acceptance pide `grep -i nemea` sin resultados; el grep ingenuo matchea `.git/`, `node_modules/`, o el propio `.planning/`.
**How to avoid:** Acotar el grep a `nemea-front/src` y `nemea-back/src` + docs específicas, excluir historial git (los acceptance ya dicen "excluyendo historial git"). Ojo: rename interno (carpeta `nemea-front/`, `package.json name`, `Logger('Nemea')`) está **fuera de scope** — esos matches son esperados y NO se tocan.
