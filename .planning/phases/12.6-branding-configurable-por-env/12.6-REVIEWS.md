# Structured Feedback: Phase 12.6 Branding Implementation Plans

I have reviewed the four implementation plans for **Phase 12.6: Branding configurable por env**. The plans are highly surgical, demonstrating a deep understanding of the technical constraints (Next.js inlining, NestJS testing pitfalls, and TypeORM migration nuances).

## Summary
The plans successfully translate the phase's requirements into actionable tasks with a strong focus on technical integrity. By centralizing branding constants, employing getters for testability in the backend, and strictly adhering to Next.js inlining rules in the frontend, the strategy effectively achieves "white-label" readiness while neutralizing information disclosure risks (e.g., the author's personal email). The inclusion of a blocking manual verification for migrations is a critical safeguard for data integrity.

---

## Strengths
- **Test Integrity:** Plan 01 Task 3 specifically addresses the "frozen const" pitfall by using a getter for `DEMO_EMAIL`, ensuring the override test in `auth.service.spec.ts` is actually valid and fails if hardcoded.
- **Security by Design:** Plan 02 mandates parametrized queries ($1, $2) for the new `SeedDemoUser` and `CreateUserTable` seeds, effectively mitigating SQL injection risks from environment variables.
- **Framework Compliance:** Plan 03 correctly identifies and enforces the "full literal" rule for `NEXT_PUBLIC_*` variables to ensure Next.js build-time inlining works as expected.
- **Operational Clarity:** Plan 04 includes the mandatory "rebuild" and "manual logo swap" instructions in the README, which are essential for a successful rebranding by a third party.
- **Parallelization:** Plans 01 (Back) and 03 (Front) are correctly identified as independent, allowing for parallel execution across repos.

---

## Concerns
- **Migration Verification (MEDIUM):** Task 3 of Plan 02 relies on a manual checkpoint. While appropriate given the lack of automated migration testing infra, it is a high-gravity task. Ensure the `DATABASE_URL` is pointed at a safe, non-production target during this "wipe and run" verification.
- **Admin Email Log-ability (LOW):** The constraint that `ADMIN_EMAIL` must be a real Google account for login is well-documented in the plans, but it remains a potential point of friction for users who forget to set it. The README note is the correct mitigation.
- **Grep Noise (LOW):** Plan 04 correctly anticipates noise from internal identifiers (e.g., `Logger('Nemea')`). The plan to document these expected matches in the summary is a good practice to avoid "completion anxiety" during verification.

---

## Suggestions
- **Joi Validation for Admin Name:** While the spec labels it as optional, adding a simple `.default('Admin')` or a length check in `env.validation.ts` for `ADMIN_NAME` would provide extra hardening against accidental empty env vars.
- **Dry-run Migrations:** Before the final `migration:run` in Plan 02, a quick `npm run typeorm migration:show` can confirm that only the intended migrations are pending/running.

## Senior Pre-mortem

**Reviewer:** Senior (internal)
**Date:** 2026-06-02

Verified every plan assumption against the live code (migration column names, the `DEMO_USER_EMAIL` export's importers, the spec's mock setup, the static `metadata` export, `package.json` scripts, and the brand greps). The plans are tight and the phase's stated #1 risk — the override-test false-green — is genuinely mitigated by the `getDemoEmail()` call-time getter (a hardcoded email makes the override test fail). Migration column names match byte-for-byte (`email`/`role`/`name`/`is_active`; demo seeds `role_id` FK) so the raw-SQL column-name trap is not triggered, and the planner correctly flagged `CreateUserTable` as the pre-roles-table enum era. Every visible/shipped `nemea`/`basualdofelipe` reference maps to a plan; the two leftover back matches (`Logger('Nemea')`, `app:'nemea-back'`) are correctly scoped out.

### Will break

_None._ No assumption failed verification; no item requires user judgment or a replan before executing.

### Might break

1. **Override-test env contamination is a SILENT failure mode (test isolation).**
   Plan 01 Task 3 sets `process.env.DEMO_EMAIL = 'demo@foo.com'` inside the override `it()`. The spec's `beforeEach` runs `jest.clearAllMocks()` (line 74) which resets mocks but does **not** touch `process.env`. If the env isn't restored, later tests read the polluted value — and because the retargeted asserts reference `getDemoEmail()` (which reads the same polluted env), they'd still pass *for the wrong reason*. Suite stays green; isolation is silently broken. The plan does instruct an `afterEach`/`finally` restore — this finding is "executor must not skip it," not "plan is wrong."
   - Audit followup: `grep -nA3 "DEMO_EMAIL = 'demo@foo.com'" nemea-back/src/auth/auth.service.spec.ts` → confirm a matching `afterEach`/`finally` deletes or restores `process.env.DEMO_EMAIL`. Also confirm the override `it()` calls `mockConfigService.get.mockReturnValue('true')` (gate, Pitfall 3).

### Missing from the plan

1. **README Rebranding section must state that the demo-email vars MUST match across repos.**
   Demo-login is a functional coupling: the front posts `NEXT_PUBLIC_DEMO_EMAIL` and the back pins `validateDemoLogin` to `DEMO_EMAIL`. If a rebrander sets one but not the other (or to different values), demo-login returns 401. Plan 04 Task 2 lists all 7 vars but doesn't tell the reader the pair must be set to the **same** value. Add one line: "`DEMO_EMAIL` (back) and `NEXT_PUBLIC_DEMO_EMAIL` (front) must be identical or the demo login fails." (Defaults already match, so the footgun only fires on deliberate change — but it's a silent 401 when it does.)
   - Audit followup: `grep -i "must match\|mismo valor\|identical\|igual" nemea-back/README.md` in the Rebranding section.

2. **Preserve the `DEMO_USER_EMAIL` security-pin rationale.**
   `auth.service.ts:9-14` carries a JSDoc explaining *why* the endpoint is hard-pinned (it never authenticates an arbitrary email even with the gate on). `DEMO_USER_EMAIL` has zero external importers (verified), so Plan 01 Task 2 will likely delete the export. Make sure that security rationale moves to `getDemoEmail()` or `validateDemoLogin` rather than getting dropped with the const.
   - Audit followup: confirm a comment explaining the single-email pin survives near `validateDemoLogin` or `getDemoEmail`.

### Execute and check later

1. **Blocking migration run (Plan 02 Task 3) — point `DATABASE_URL` at a throwaway DB.**
   The checkpoint says "drop/recreate the DB and run migrations." This app is clone-and-run-local (each dev runs against their own DB), so "wipe and run" is the intended flow — but the human doing the checkpoint must not run it against a DB holding data they care about. This is a one-line operational caution, not a plan defect.
   - Will surface as: data loss only if pointed at a populated DB — otherwise clean seed rows.
   - Expected fix: verify `DATABASE_URL` targets a local/disposable DB before `npm run migration:run`.
   - Audit followup: N/A (manual checkpoint; confirm the SUMMARY records both the default-run and override-run row inspections).

### Notes (not blocking)

- **Gemini's `ADMIN_NAME` Joi suggestion — I'd skip it.** The plan deliberately leaves `ADMIN_NAME` unvalidated because a display name has no format to validate. An empty name seeds an empty string (cosmetic), not a boot failure. Adding a Joi default duplicates the `?? 'Admin'` fallback already in `branding.ts` for zero functional gain. Disagree with the suggestion; the plan's choice is correct.
- **STATE.md staleness (FYI):** STATE.md "Known Issues" claims ~4 pre-existing `tsc --noEmit` errors in `nemea-front` test files. I ran `npx tsc --noEmit` in `nemea-front` → exit 0. Those were cleaned up; Plan 03's tsc verify gate is safe. Worth updating STATE.md.
- **Double-default consistency:** `DEMO_EMAIL`/`ADMIN_EMAIL` get a Joi `.default(...)` (boot validation) AND a `?? '...'` fallback in `branding.ts` (the actual read path, since branding reads `process.env` directly per D-02). Both defaults are set to identical values in this same phase, so no divergence today — just be aware they're two places to keep in sync if defaults ever change.

**SAFE TO EXECUTE — audit followups attached.** Plans verified against live code, the headline risk is properly defended, and every shipped brand reference is covered.
