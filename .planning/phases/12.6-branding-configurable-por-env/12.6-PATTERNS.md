## Pattern Assignments

### NEW `nemea-front/src/constants/branding.ts` (config, transform)

**Analog:** `nemea-front/src/constants/tiendanube.ts` (the entire file — only 3 lines).

**Analog full content** (`nemea-front/src/constants/tiendanube.ts:1-3`):
```typescript
export const TN_PLAN_ESENCIAL = 'esencial' as const;
export const TN_GATEWAY_PAGO_NUBE = 'pago_nube' as const;
export const TN_PAYMENT_TARJETA = 'tarjeta_debito_credito' as const;
```

**Pattern to copy:** flat module of `export const` named exports, no default export, no class. For branding, replace the literal RHS with `process.env.NEXT_PUBLIC_* ?? '<fallback>'` (the value comes from RESEARCH Pattern 1, lines 98-103):
```typescript
// NEXT_PUBLIC_* se inlinean en build time — rebuild requerido para cambios.
export const APP_NAME = process.env.NEXT_PUBLIC_APP_NAME ?? 'Hefesto';
export const DEMO_EMAIL = process.env.NEXT_PUBLIC_DEMO_EMAIL ?? 'demo@hefesto.com';
export const CONTACT_EMAIL = process.env.NEXT_PUBLIC_CONTACT_EMAIL ?? 'admin@hefesto.com';
```

**CRITICAL inlining rule** (RESEARCH Pitfall 2 + Anti-Patterns, lines 105/143): write the full literal `process.env.NEXT_PUBLIC_X`. Do NOT destructure (`const { NEXT_PUBLIC_X } = process.env`) and do NOT index dynamically — Next's build-time replacement only fires on the literal form, otherwise the client gets `undefined`.

**Import convention** (front uses `@/` alias, confirmed at `GatewaySelectors.tsx:16`): consumers import via `import { APP_NAME } from '@/constants/branding';`.

### NEW `nemea-back/src/constants/branding.ts` (config, transform)

**Analog:** `nemea-back/src/constants/tiendanube.ts:1-3` (same flat `export const` shape as the front sibling above).

**Pattern to copy:** plain `process.env.X ?? 'fallback'` (no `ConfigService` — per CONTEXT D-02, because migrations run outside Nest DI and `data-source.ts` already does `import 'dotenv/config'`). From RESEARCH Pattern 2 (lines 112-121):
```typescript
export const APP_NAME = process.env.APP_NAME ?? 'Hefesto';
export const ADMIN_EMAIL = process.env.ADMIN_EMAIL ?? 'admin@hefesto.com';
export const ADMIN_NAME = process.env.ADMIN_NAME ?? 'Admin';
```

**DEMO email — test-safety decision (Open Question, Claude's Discretion):** RESEARCH recommends **Option A** — expose the demo email as a getter so the override test sees env mutations without `jest.resetModules()`:
```typescript
// Option A (recommended): getter, not a frozen module-level const
export const getDemoEmail = (): string => process.env.DEMO_EMAIL ?? 'demo@hefesto.com';
```
A frozen `const DEMO_EMAIL = process.env...` is the anti-pattern flagged in Pitfall 1 (lines 173-188): the value is captured once at import, so a test that only mutates `process.env` afterward gives a false green. If the planner keeps a `const`, the override test MUST use `jest.resetModules()` + dynamic `import()`.

**Import convention** (back uses relative paths, confirmed at `calculadora.service.ts:9` → `'../constants/tiendanube'`): services import `from '../constants/branding'`; migrations import `from '../../constants/branding'` (one level deeper, under `database/migrations/`).

## Shared Patterns

### Centralized constants module (the spine of this phase)
**Source:** `constants/tiendanube.ts` (both repos) — flat `export const` named exports, no default, no class.
**Apply to:** both new `constants/branding.ts`. Every point of use imports from it; **zero inline `process.env` reads** scattered in components/services (CONTEXT D-01/D-02/D-03).
```typescript
export const APP_NAME = process.env.NEXT_PUBLIC_APP_NAME ?? 'Hefesto'; // front
export const APP_NAME = process.env.APP_NAME ?? 'Hefesto';            // back
```

### Import path convention
**Source:** `GatewaySelectors.tsx:16` (front), `calculadora.service.ts:9` (back).
**Apply to:** all consumers.
- Front: `import { APP_NAME } from '@/constants/branding';` (`@/` alias, never relative).
- Back services/main: `import { APP_NAME } from '../constants/branding';` (relative).
- Back migrations (one dir deeper): `import { ... } from '../../constants/branding';`.

### env read with code fallback (never undefined)
**Source:** existing pattern in `main.ts:57` (`?? 4000`) and `auth.service.ts` defaults.
**Apply to:** every branding value. SPEC constraint: `?? 'Hefesto'` / `?? 'demo@hefesto.com'` / `?? 'admin@hefesto.com'` so the app boots and renders with defaults if any env is missing.

### Joi env validation at boot
**Source:** `env.validation.ts:3-13` (`Joi.string()...default(...)`).
**Apply to:** `DEMO_EMAIL`, `ADMIN_EMAIL` — `Joi.string().email().default(...)`. Default `.email()` (no `tlds` override) so `demo@demo` fails boot (req 6).

### Parametrized seed query (SQL injection guard)
**Source:** `1773200000000-SeedDemoUser.ts:18-23` (`$1`/`$2`/`$3` + values array).
**Apply to:** the admin seed in `CreateUserTable` (currently string-interpolated). Never interpolate env into the SQL string (Security Domain — Tampering, req 9).

### dotenv in migrations (already present — do NOT touch)
**Source:** `data-source.ts:1` `import 'dotenv/config'`.
**Why it matters:** this is the single reason `process.env` is populated inside migrations, enabling seed-from-env without `ConfigService`. No change needed; just rely on it.
