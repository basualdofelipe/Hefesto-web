# Phase 12.6: Code Review Report

**Reviewed:** 2026-06-02
**Depth:** standard
**Files Reviewed:** 18
**Status:** issues_found

## Summary

This phase makes branding (app name, demo/admin/contact emails) configurable via environment variables across both backend and frontend, with neutral `Hefesto` defaults. The core branding mechanism is clean: `process.env.* ?? 'fallback'` on the backend (correctly NOT routed through ConfigService for migration-time access, with `getDemoEmail()` deliberately reading at call-time so Jest overrides work), and `NEXT_PUBLIC_*` inlining on the frontend. The demo-login pinning logic is well-tested and resistant to privilege escalation via a foreign email.

The dominant concern is not the branding plumbing itself but a **security posture decision that ships in this phase**: the demo account is seeded with the full `ADMIN` role (all 11 permissions including `can_manage_users`), AND demo login is enabled by default in both shipped `.env.example` files. A fresh clone therefore exposes a public, password-less, full-admin account. Combined with documentation that contradicts the shipped defaults, this is the chief risk. Several documentation/code drift issues round out the findings.

## Critical Issues

### CR-01: Demo login enabled by default grants public full-ADMIN access

**File:** `nemea-back/.env.example:28`, `nemea-front/.env.example:12`, `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:11-26`
**Issue:** Three facts compose into a real exposure:
1. `SeedDemoUser` seeds the demo account with the `ADMIN` role (`SELECT id FROM "roles" WHERE "name" = 'ADMIN'`), which has every permission flag `true` — including `can_manage_users`, `can_manage_config`, and all edit/delete permissions (see `1773000000000-CreateRolesAndMigrateUsers.ts:42-44`).
2. `DEMO_LOGIN_ENABLED=true` is the shipped default in `nemea-back/.env.example:28`, and `NEXT_PUBLIC_DEMO_LOGIN_ENABLED=true` in `nemea-front/.env.example:12`.
3. `POST /api/auth/demo-login` is `@Public()` and requires no credentials — only the pinned email (`auth.controller.ts:48-63`, `auth.service.ts:69-102`).

Result: anyone who clones and runs with the example env, or deploys without overriding these, gets an unauthenticated path to a full administrator session (create/delete users, change roles, mutate all business data). The architecture doc (`ARCHITECTURE.md:116`, `:382`) defends the *email pinning* as the safeguard, but pinning only restricts *which* account is minted — it does nothing to limit that account's privilege. The seeded demo account being full ADMIN defeats the "static credential, not a bypass" rationale.

**Fix:** Seed the demo account with a read-only / limited role rather than ADMIN, and/or default the flag to off in the shipped examples. Minimum viable fix — create a dedicated `DEMO` (read-only) role in a migration and point the seed at it:
```ts
// SeedDemoUser up()
const demoRoleResult = await queryRunner.query(
  `SELECT id FROM "roles" WHERE "name" = 'DEMO' LIMIT 1`,
);
const demoRoleId = (demoRoleResult[0] as { id: string } | undefined)?.id;
if (!demoRoleId) {
  throw new Error('DEMO role not found — seed a read-only role first');
}
// ...INSERT with demoRoleId instead of adminRoleId
```
If the product intent is genuinely "out-of-the-box full demo," then at minimum ship `DEMO_LOGIN_ENABLED=false` in `.env.example` (matching the README, see WR-01) and require the deployer to opt in consciously. Do not ship both "enabled by default" and "ADMIN-privileged" together.

## Warnings

### WR-01: README contradicts the shipped `.env.example` for `DEMO_LOGIN_ENABLED`

**File:** `nemea-back/README.md:114`, `nemea-back/.env.example:28`
**Issue:** README local-setup block documents `DEMO_LOGIN_ENABLED=false` (described as "Optional — enables /api/auth/demo-login endpoint"), but the actual shipped `.env.example` sets `DEMO_LOGIN_ENABLED=true` with a comment saying it is enabled by default. A deployer copying `.env.example` (as README step 2 instructs: `cp .env.example .env`) gets `true`, contradicting the documented `false`. This drift directly feeds CR-01 — the operator believes demo is off when it is on.
**Fix:** Make them consistent. Either set `.env.example` to `false` and update its comment, or update README:114 to `true` and document the security implication explicitly. Given CR-01, prefer `false` in both.

### WR-02: ARCHITECTURE.md documents the wrong demo-login route

**File:** `nemea-back/docs/ARCHITECTURE.md:116`
**Issue:** The doc states "A demo login path exists (`POST /api/auth/demo`)" but the actual route is `POST /api/auth/demo-login` (`auth.controller.ts:48`). A reader following the architecture doc to test or audit the endpoint will hit a 404 and may wrongly conclude the endpoint does not exist.
**Fix:** Change `POST /api/auth/demo` to `POST /api/auth/demo-login` in ARCHITECTURE.md:116.

### WR-03: Env-default for `DEMO_EMAIL` is bypassed at migration time, allowing back/front drift to seed an unreachable demo account

**File:** `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:7`, `nemea-back/src/constants/branding.ts:29-30`
**Issue:** The seed migration reads `process.env.DEMO_EMAIL ?? 'demo@hefesto.com'` directly (Joi validation does not run for the standalone migration runner / CLI DataSource). If a deployer sets `DEMO_EMAIL` on the backend but forgets `NEXT_PUBLIC_DEMO_EMAIL` on the frontend (or vice-versa), the seeded account email and the email the frontend posts will diverge, and `validateDemoLogin` rejects with 401 (`auth.service.ts:75`). README:269 notes this requirement in prose, but nothing enforces it. Worse: if the backend env has a malformed `DEMO_EMAIL`, the migration seeds a malformed account silently (no email-format check at migration time), even though Joi would reject it at app boot — producing an inconsistent state where the migration succeeded but the app refuses to start.
**Fix:** This is an accepted-config-coupling risk; at minimum document it as a single sourced value. Consider validating `DEMO_EMAIL` format inside the migration before seeding, e.g. throw if it does not match a basic email regex, so a bad value fails the migration loudly rather than seeding an unusable account.

### WR-04: `validateDemoLogin` re-reads `DEMO_LOGIN_ENABLED` as untyped string with no normalization

**File:** `nemea-back/src/auth/auth.service.ts:70-73`
**Issue:** `this.configService.get<string>('DEMO_LOGIN_ENABLED')` is compared with `!== 'true'`. Joi validates the value to the literal set `'true' | 'false'` (`env.validation.ts:12`), so this works at boot — but the comparison is whitespace/case-sensitive and any value other than the exact lowercase `true` silently disables demo login. Combined with the call-time read pattern, a value like `TRUE`, ` true`, or `1` (which Joi would reject, but which could appear if validation is ever bypassed or relaxed) fails closed silently with no log. This is defensible as fail-closed, but the lack of any diagnostic makes a misconfigured-but-intended-enabled deployment hard to debug.
**Fix:** Normalize and optionally log when disabled-by-config:
```ts
const enabled = (this.configService.get<string>('DEMO_LOGIN_ENABLED') ?? 'false').trim().toLowerCase() === 'true';
if (!enabled) {
  throw new UnauthorizedException('Demo login no disponible');
}
```

### WR-05: `getProfile` / `auth/me` returns the raw TypeORM `User` entity

**File:** `nemea-back/src/auth/auth.service.ts:65-67`, `nemea-back/src/auth/auth.controller.ts:75-83`
**Issue:** `getProfile` returns the full `User` entity (with its loaded `role` relation) directly to the client, unlike `validateGoogleToken`/`validateDemoLogin` which map to a controlled DTO shape. The README itself flags this as a known trade-off (README:301), but it is in the changed surface for this phase (the file was modified) and is worth recording: any future column added to `User` (e.g. an internal flag, a token, a soft-delete marker) leaks automatically through `/api/auth/me`. No secret column exists today, so this is a hardening WARNING, not a live leak.
**Fix:** Return an explicit response DTO from `auth/me` mirroring the `AuthResponseDto.user` shape (id, email, name, pictureUrl, permissions) rather than the entity.

## Info

### IN-01: ARCHITECTURE.md "No auto-increment integers anywhere" contradicts the initial users migration

**File:** `nemea-back/docs/ARCHITECTURE.md:162`, `nemea-back/src/database/migrations/1772369233961-CreateUserTable.ts:14`
**Issue:** The doc asserts "UUID primary keys across the board. No auto-increment integers anywhere." The first migration creates `users.id` as `SERIAL` (later converted to UUID by `1772380000000-AlterUsersIdToUuid.ts`). The end state matches the doc, but the absolute claim is technically false for the migration history. Minor doc precision issue.
**Fix:** Soften to "All entities end up with UUID primary keys; an early users migration used SERIAL and was migrated to UUID."

### IN-02: `DemoLoginDto.email` is validated and required but functionally ignored

**File:** `nemea-back/src/auth/dto/demo-login.dto.ts:9-10`, `nemea-back/src/auth/auth.service.ts:75-80`
**Issue:** The DTO requires a valid email, but the service only uses it to compare against the pinned `getDemoEmail()` and then looks up the pinned email regardless. The client-supplied email is never used as a lookup key (correct, by design for security), so the field is effectively a "prove you know the demo email" token. This is fine, but the DTO/`@ApiProperty` presents it as if the caller chooses the account, which can mislead API consumers reading Swagger.
**Fix:** Add a description note to `@ApiProperty` clarifying the email must equal the configured demo email and is not a free-choice account selector.

### IN-03: Hardcoded brand string literal in spec fixture diverges from configurable default

**File:** `nemea-back/src/auth/auth.service.spec.ts:240`
**Issue:** `mockDemoUser.name` is the literal `'Demo Hefesto'` while the seed migration computes `Demo ${process.env.APP_NAME ?? 'Hefesto'}` (`SeedDemoUser:8`). The test fixture name is never asserted against, so this is cosmetic, but it bakes the default brand into a test that the phase is explicitly trying to make configurable. If `APP_NAME` default ever changes, this fixture silently drifts.
**Fix:** Either derive from the branding constant or leave a comment that the literal is intentional fixture data.

### IN-04: Login page redirect-target `redirectTo` computed but demo flow ignores it

**File:** `nemea-front/src/app/(auth)/login/page.tsx:46-47`, `nemea-front/src/app/(auth)/login/DemoLoginButton.tsx:29`
**Issue:** `LoginPage` computes `redirectTo = callbackUrl ?? '/'` and uses it for the Google flow, but `DemoLoginButton` hardcodes `router.push('/')` and never receives/honors `callbackUrl`. A user deep-linked to a protected page who chooses "Entrar como demo" lands on `/` instead of their intended destination. Minor UX inconsistency; not in the branding scope strictly, but the demo button is part of this phase's surface.
**Fix:** Pass `callbackUrl` into `DemoLoginButton` as a prop and `router.push(callbackUrl ?? '/')` after a successful sign-in.
