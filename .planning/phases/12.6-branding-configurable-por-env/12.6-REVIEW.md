## Summary

This phase makes branding (app name, demo/admin/contact emails) configurable via environment variables across backend and frontend, with neutral `Hefesto` fallbacks, plus an env-gated demo-login path. The branding plumbing is clean and consistent: `process.env.* ?? 'fallback'` on the backend (correctly NOT routed through `ConfigService` because migrations run outside Nest's DI container), `getDemoEmail()` deliberately reading at call-time so Jest overrides work without `resetModules`, `NEXT_PUBLIC_*` inlining on the frontend, and migrations reading the same vars with the same defaults. The demo-login pinning is well-tested and resists privilege escalation via a foreign email (`auth.service.ts:75-83`, spec at `auth.service.spec.ts:302-323`).

**Adjudication carried forward (ground truth):** The previous report raised a Critical (CR-01) that "demo seeded as ADMIN + `DEMO_LOGIN_ENABLED=true` default ⇒ public passwordless full-admin." The senior audit (preserved below) reclassified this as a **false positive** for this project's **clone-and-run-local** distribution model — there is no anonymous public attacker; each cloner runs against their own DB and the demo account is the cloner's own admin. The user explicitly confirmed admin is intended; a future hosted demo would use a hand-made read-only DEMO role created post-deploy (no code change now). This re-review does **not** revive that finding, and `--fix` must not neuter demo login or flip the default.

## Warnings

### WR-01: Demo login failure is swallowed with no user feedback

**File:** `nemea-front/src/app/(auth)/login/DemoLoginButton.tsx:24-30`
**Issue:** When `signIn('credentials', ...)` returns an error (backend `DEMO_LOGIN_ENABLED=false`, a `DEMO_EMAIL` vs `NEXT_PUBLIC_DEMO_EMAIL` mismatch — the exact case README:270 warns about — or backend unreachable), the handler does `setIsPending(false)` and silently returns. The user clicks "Entrar como demo", the spinner stops, and nothing visible happens: no toast, no message, no diagnostic. For a portfolio app where evaluators will click this button first, a silent dead-end is a real UX defect with zero discoverability.

**Fix:** A `Toaster` is already mounted in `layout.tsx:57`. Surface the failure:
```ts
import { toast } from 'sonner';
// ...
if (result?.error !== undefined && result.error !== null) {
  setIsPending(false);
  toast.error('No se pudo iniciar el login demo. Verificá la configuración.');
  return;
}
```

### WR-02: `APP_NAME` and `ADMIN_NAME` are consumed but absent from the Joi env schema

**File:** `nemea-back/src/config/env.validation.ts:3-15`, `nemea-back/src/constants/branding.ts:12,15`
**Issue:** The schema validates `DEMO_EMAIL`, `ADMIN_EMAIL`, and `DEMO_LOGIN_ENABLED`, but `APP_NAME` and `ADMIN_NAME` — both read in `branding.ts` and used in `main.ts:48-49` (Swagger title/description) and the `CreateUserTable` migration (`adminName`) — are not in the schema. The README advertises "Joi env validation (fail-fast)" as a feature; these two branding vars sit outside that contract, so a typo like `APP_NAEM` is never caught and silently falls back to `Hefesto`. This is a consistency gap in the phase's own feature (configurable branding), not just generic hardening.

**Fix:** Add the missing keys so validation stays the single source of truth:
```ts
APP_NAME: Joi.string().default('Hefesto'),
ADMIN_NAME: Joi.string().default('Admin'),
```

### WR-03: `SeedDemoUser.down()` deletes any user matching the demo email, with no guard against the admin or a missing env

**File:** `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:29-33`
**Issue:** `down()` runs an unconditional `DELETE FROM "users" WHERE "email" = $1` using `process.env.DEMO_EMAIL ?? 'demo@hefesto.com'`. The `up()` path is safely idempotent (`ON CONFLICT DO NOTHING`), but the revert path is not symmetric: if `DEMO_EMAIL` is ever set equal to `ADMIN_EMAIL`, or if the env value at revert time differs from seed time, reverting this migration deletes a real account (and cascades to its scenarios) rather than just the seeded demo row. There is no marker proving the deleted row was the one this migration inserted.

**Fix:** Guard the destructive revert:
```ts
const demoEmail = process.env.DEMO_EMAIL ?? 'demo@hefesto.com';
const adminEmail = process.env.ADMIN_EMAIL ?? 'admin@hefesto.com';
if (demoEmail === adminEmail) return; // never delete the admin on revert
await queryRunner.query(`DELETE FROM "users" WHERE "email" = $1`, [demoEmail]);
```

## Info

### IN-01: Migration-time `DEMO_EMAIL` bypasses Joi, so a malformed value seeds an unusable account silently

**File:** `nemea-back/src/database/migrations/1773200000000-SeedDemoUser.ts:7`, `nemea-back/src/config/env.validation.ts:13`
**Issue:** Joi validates `DEMO_EMAIL` as an email at app boot, but the standalone migration runner reads `process.env.DEMO_EMAIL` directly (correctly — migrations run outside DI). A malformed `DEMO_EMAIL` therefore seeds a malformed/unreachable demo account during `migrationsRun`, and only later does Joi reject the value at app startup — an inconsistent state where the DB seed succeeded but the app refuses to start. The front/back must-match requirement is documented (README:270) but unenforced.
**Fix:** Optional hardening — validate `DEMO_EMAIL` format inside the migration before seeding and throw on a bad value, so a misconfiguration fails the migration loudly rather than persisting an unusable row.

### IN-02: `acceso-denegado` renders the contact email as plain text, not a `mailto:` link

**File:** `nemea-front/src/app/(auth)/acceso-denegado/page.tsx:25`
**Issue:** `...contacta al administrador en {CONTACT_EMAIL}.` is bare text. A user told to contact the admin cannot click through.
**Fix:** `<a href={\`mailto:${CONTACT_EMAIL}\`} className='underline'>{CONTACT_EMAIL}</a>`.

### IN-03: Demo button ignores `callbackUrl`, dropping the deep-link target

**File:** `nemea-front/src/app/(auth)/login/page.tsx:46-47`, `nemea-front/src/app/(auth)/login/DemoLoginButton.tsx:29`
**Issue:** `LoginPage` computes `redirectTo = callbackUrl ?? '/'` and honors it for the Google flow, but `DemoLoginButton` hardcodes `router.push('/')`. A user deep-linked to a protected page who chooses "Entrar como demo" lands on `/` instead of the intended destination. Minor UX inconsistency on this phase's surface.
**Fix:** Pass `callbackUrl` into `DemoLoginButton` as a prop and `router.push(callbackUrl ?? '/')` after a successful sign-in.

### IN-04: `branding.ts` JSDoc pins a file:line that will silently rot

**File:** `nemea-back/src/constants/branding.ts:9`
**Issue:** The comment references `import 'dotenv/config' (already present in data-source.ts:1)`. Hard-coding a line number in prose breaks silently if `data-source.ts` is ever edited.
**Fix:** Drop the `:1` and reference the file only.
