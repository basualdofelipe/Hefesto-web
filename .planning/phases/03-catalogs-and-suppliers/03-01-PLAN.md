  <name>Task 1: Create UUID migration for users table</name>
  <files>nemea-back/src/database/migrations/TIMESTAMP-AlterUsersIdToUuid.ts</files>
  <action>
Create a new TypeORM migration file in `nemea-back/src/database/migrations/`. Use a timestamp-based name like `{timestamp}-AlterUsersIdToUuid.ts`.

The migration must:

1. **Enable uuid-ossp extension** (required by TypeORM's `@PrimaryGeneratedColumn('uuid')` which generates `uuid_generate_v4()`):
   ```sql
   CREATE EXTENSION IF NOT EXISTS "uuid-ossp"
   ```

2. **Drop the existing PK constraint** on users table:
   ```sql
   ALTER TABLE "users" DROP CONSTRAINT "PK_a3ffb1c0c8416b9fc6f907b7433"
   ```

3. **Add a temporary UUID column**, populate it with generated UUIDs:
   ```sql
   ALTER TABLE "users" ADD COLUMN "new_id" uuid NOT NULL DEFAULT uuid_generate_v4()
   ```

4. **Drop old integer id column and rename new one**:
   ```sql
   ALTER TABLE "users" DROP COLUMN "id"
   ALTER TABLE "users" RENAME COLUMN "new_id" TO "id"
   ```

5. **Recreate PK**:
   ```sql
   ALTER TABLE "users" ADD CONSTRAINT "PK_users_id" PRIMARY KEY ("id")
   ```

The `down()` method should reverse this (add integer id back, drop UUID id). Since data will be lost on rollback, a simple drop + recreate approach is acceptable for down().

**Critical:** This migration MUST run BEFORE any new entity tables are created (TypeORM runs migrations in filename order, so ensure the timestamp is before any Phase 3 table creation migrations).

**Note:** No FK references to users.id exist from other tables yet (only 1 table), so no FK drops/recreates needed. The seeded admin user will get a new UUID id. Since JWT tokens contain email (not id) for validation, existing tokens will still work for the whitelist check. The `sub` field in JWT payload will contain a UUID string after this change.
