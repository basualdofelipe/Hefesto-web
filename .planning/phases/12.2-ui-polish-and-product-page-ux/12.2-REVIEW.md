Phase 12.2 covers the `sort_order` migration for product sizes, home page RSC rewrite, logo/sidebar PNG swap, and cascade collapse/typography polish across products and supplies components. ## Warnings

### WR-01: Migration size name mismatch — `'Talle Unico'` not in CASE list

**File:** `nemea-back/src/database/migrations/1773100000000-AddSortOrderToProductSizes.ts:14-25`

**Issue:** The CASE expression in Step 2 handles `'Unico'` (sort_order = 0), but the UI in `ProductFinishGroup.tsx` (line 145-147) checks for `product.size.name === 'Talle Unico'` to display an em-dash. If `'Talle Unico'` exists as a separate row in `product_sizes`, it falls into the `ELSE 50` branch instead of getting sort_order 0. This would sort "Talle Unico" after numeric belt sizes (which receive their numeric value, potentially 70–120+), creating a misleading ordering in size selectors and any future size-sorted UI.

**Fix:** Either ensure the DB only ever contains `'Unico'` (and update the UI check to match), or add `'Talle Unico'` to the CASE explicitly:

```sql
WHEN 'Unico' THEN 0
WHEN 'Talle Unico' THEN 0
```

And align the UI display check in `ProductFinishGroup.tsx` line 145 to whichever canonical name is stored in the DB:

```tsx
{product.size.name === 'Unico' ? '\u2014' : product.size.name}
```

### WR-02: BOM total row missing `text-right` alignment on cost cell

**File:** `nemea-front/src/components/products/ProductExpandedRow.tsx:269-278`

**Issue:** The "Total" summary row at the bottom of the BOM table has `text-right` on the colSpan label cell (line 270), but the cost value cell (line 272-277) has no alignment class. All other numeric cells in the same table use `className='text-right'`. This causes the total cost to left-align visually while all other cost cells are right-aligned — a visual inconsistency that looks like a bug.

```tsx
// Current (line 272)
<TableCell>
  {formatCost(totalCost > 0 ? totalCost : product.cost)}
</TableCell>
```

**Fix:**
```tsx
<TableCell className='text-right'>
  {formatCost(totalCost > 0 ? totalCost : product.cost)}
</TableCell>
```

### WR-03: Home page renders empty grid with no feedback when user has zero permissions

**File:** `nemea-front/src/app/(app)/page.tsx:46-65`

**Issue:** When `visibleCards` is empty (a viewer with no permissions, or a broken session returning `NO_PERMISSIONS`), the component renders a `<div className='p-6'>` containing an empty `<div className='grid ...'>` — completely blank UI with no message. In practice this is unlikely for real users, but a new user before role assignment, or a session decode failure (which falls back to `NO_PERMISSIONS`), would see a blank page with no explanation.

**Fix:** Add an empty-state guard:

```tsx
if (visibleCards.length === 0) {
  return (
    <div className='p-6'>
      <p className='text-muted-foreground text-sm'>
        No tenés acceso a ninguna sección. Contactá al administrador.
      </p>
    </div>
  );
}
```

### WR-04: `handleTogglePublic` in `ScenarioEditorClient` passes `void` to `Switch.onCheckedChange`

**File:** `nemea-front/src/app/(app)/escenarios/[id]/ScenarioEditorClient.tsx:199`

**Issue:** `Switch`'s `onCheckedChange` prop expects `(checked: boolean) => void`. The current code passes `handleTogglePublic` directly — a `Promise<void>`-returning async function that ignores its boolean argument. While this does not crash at runtime (the argument is silently dropped and the Promise is implicitly discarded), it bypasses the project's pattern of explicitly voiding async handlers, and TypeScript strict mode may produce a type error depending on Shadcn/ui's typings.

```tsx
// Current (line 199)
<Switch checked={isPublic} onCheckedChange={handleTogglePublic} />
```

**Fix:** Wrap to be explicit and consistent with how `ProductExpandedRow` handles the same pattern:

```tsx
<Switch
  checked={isPublic}
  onCheckedChange={() => void handleTogglePublic()}
/>
```

### IN-01: `catalogs.service.ts` — `create()` does not auto-assign `sortOrder` for new `product-sizes`

**File:** `nemea-back/src/catalogs/catalogs.service.ts:117-143`

**Issue:** The `create()` method auto-assigns `skuCode` when not provided (line 124-129), but has no equivalent logic for `sortOrder` on `product-sizes`. New sizes created via the catalog UI will receive `sort_order = 0` (the column default), which puts them before `'Unico'` in the sort order. This is not a crash, but it means any new size created post-migration will sort to the top of the list regardless of intended order. Whether this is intentional ("admin sets sort_order manually") or an oversight should be confirmed.

**Fix (if auto-assign is desired):**
```typescript
if (dimension === 'product-sizes' && (dto as { sortOrder?: number }).sortOrder == null) {
  const result = await this.productSizeRepo
    .createQueryBuilder('item')
    .select('COALESCE(MAX(item.sortOrder), 0)', 'maxOrder')
    .getRawOne<{ maxOrder: string }>();
  // assign at end of existing sizes
  (dto as { sortOrder?: number }).sortOrder = (parseInt(result.maxOrder, 10) || 0) + 10;
}
```

### IN-02: `ProductNameGroup` — divergence detection accumulates state across re-renders from child callbacks

**File:** `nemea-front/src/components/products/ProductNameGroup.tsx:59, 100, 124-126`

**Issue:** `hasDivergence` is set via `handleDivergenceDetected` called by child `BomGroupEditorDialog`. The `costBasedDivergence` memo correctly recomputes from props, but `hasDivergence` from the child callback is never reset to `false` when the BOM editor closes without changes. This means once a divergence is detected and the dialog is closed, the amber badge persists even if the user subsequently saves a uniform BOM. The cost-based heuristic would eventually catch the correction after `router.refresh()`, but `hasDivergence` remains `true` until the component unmounts.

**Fix:** Reset `hasDivergence` to `false` on `router.refresh()` (after successful BOM group save) or when the dialog closes:

```tsx
<BomGroupEditorDialog
  ...
  onSuccess={() => {
    setHasDivergence(false);
    router.refresh();
  }}
  ...
/>
```
