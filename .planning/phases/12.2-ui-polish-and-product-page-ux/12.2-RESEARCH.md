### D-03/D-04: Home Page Quick-Link Cards

**What:** Replace current `page.tsx` (brand splash + theme toggle) with RSC quick-link cards.

**Current state:** `nemea-front/src/app/(app)/page.tsx` is a `'use client'` component rendering "NEMEA" h1 + subtitle + theme toggle button. No data fetching. [VERIFIED: Read tool]

**Decision:** D-04 says RSC (no `'use client'` directive, no API calls). Cards use `usePermissions()` — but `usePermissions()` uses `useSession()` which is client-side. Therefore the page needs a client wrapper component or a direct server-side session read.

**Correct pattern for permission-aware RSC:**

Since `usePermissions()` uses `useSession()` (client hook), the permission-aware cards CANNOT be a pure RSC if they use that hook. Two valid approaches:

Option A (recommended, matches D-04 intent): The page.tsx is an RSC that reads the session server-side via `auth()` (the same `auth` import already used in `login/page.tsx`), extracts permissions from the session, and renders cards conditionally as pure server markup. No client hooks needed. [VERIFIED: `login/page.tsx` already imports `auth` from `@/auth` for server-side session check]

Option B: Thin RSC outer page + `HomeCardsClient` client component that calls `usePermissions()`. This is heavier than needed since no interactivity is required.

**Recommendation:** Use Option A. Read session server-side with `auth()`, extract `session.user.permissions`, render card grid with `next/link` — fully RSC, no client boundary needed.

**Card structure:** 4 cards mapping to sidebar nav items:
- Productos → `/productos` — requires `canViewProducts`
- Calculadora → `/calculadora` — requires `canUseCalculator`
- Insumos → `/insumos` — requires `canViewSupplies`
- Gastos → `/finanzas/gastos` — requires `canViewExpenses`

Shadcn Card is installed. Use `<Card>` with `<CardHeader>` and `<CardContent>` wrapped in `<Link>` for navigation. Lucide icons (ShoppingBag, Calculator, Package, Receipt) are already imported in AppSidebar — reuse the same icons for visual consistency.

**Permission type:** `session.user.permissions` contains the `Permissions` object (same type returned by `usePermissions()`). The `NO_PERMISSIONS` fallback from `@/types/permissions` should be used when session is null. [VERIFIED: Read of usePermissions.ts]

**Part 1 — Migration:**

## Common Pitfalls

### Pitfall 1: usePermissions() in RSC Home Page
**What goes wrong:** Calling `usePermissions()` directly in an RSC page causes "useState can only be used in a Client Component" error.
**Why it happens:** `usePermissions()` calls `useSession()` which is a React hook — hooks cannot run in RSC.
**How to avoid:** Use `auth()` (server-side session) in the page.tsx RSC and pass permissions as props, OR wrap in a `'use client'` component. The `auth()` import pattern is already established in `login/page.tsx`.
**Warning signs:** Build error mentioning "use client" boundary at page.tsx.

### Pitfall 2: next/image with local public/ assets — missing dimensions
**What goes wrong:** `<Image src="/brand/logo.png">` throws "Image with src requires width and height" error.
**Why it happens:** Next.js Image requires explicit `width` and `height` for non-responsive images (or `fill` layout).
**How to avoid:** Always provide `width` and `height` props. The logos are square (1080x1080, 3000x3000) so use `width={N} height={N}` with `className="object-contain"`.
**Warning signs:** Build error or runtime warning about missing dimensions.

### Pitfall 3: sort_order migration DEFAULT 0 causes all existing rows to get sort_order = 0
**What goes wrong:** Adding `NOT NULL DEFAULT 0` and then UPDATE-ing looks correct in migration, but if the UPDATE WHERE clause doesn't match actual DB data (e.g., "Talle Unico" vs "Unico"), some rows stay at 0 and appear first.
**Why it happens:** The seed migration inserts `'Unico'` but the frontend code checks for `'Talle Unico'` — there may be renamed sizes in production. [VERIFIED: seed has 'Unico'; frontend TypeScript has 'Talle Unico' hardcoded in ProductFinishGroup.tsx:145 and types.ts:67]
**How to avoid:** Run `SELECT name FROM product_sizes ORDER BY name` before writing the migration to confirm exact current names. The UPDATE in the migration must match exact DB values.
**Warning signs:** Sizes appearing in wrong order after migration.

### Pitfall 4: Typography audit introduces font-normal where font-medium was appropriate
**What goes wrong:** Blanket replace of `font-medium` → `font-normal` on table data cells makes numbers appear too light, especially in dense tables.
**Why it happens:** D-08 rule says body/data → font-normal, but numeric table data often reads better at font-medium than font-normal.
**How to avoid:** The rule in D-08 is: "headers/labels/badges → font-semibold, body/descriptions/placeholders → font-normal". Numeric table data in cells (costs, prices, margins) should become `font-normal tabular-nums`. Check that `tabular-nums` is preserved on all financial cells.
**Warning signs:** Numbers in tables look inconsistent in visual weight compared to labels.

### Pitfall 5: Sidebar logo PNG breaks at collapsed icon width
**What goes wrong:** The isotipo PNG (3000x3000) rendered at `width={28}` in the collapsed sidebar looks blurry because Next.js Image generates incorrect srcset sizes.
**Why it happens:** The sidebar icon width (`--sidebar-width-icon`) is approximately 40px. Images wider than their container get clipped without `object-contain`.
**How to avoid:** Always use `className="object-contain"` and ensure `width`/`height` props match the intended display size in pixels, not the source image dimensions.

### Home Page RSC with Server-Side Permissions
```tsx
// nemea-front/src/app/(app)/page.tsx
// Source: login/page.tsx pattern for auth() usage [VERIFIED: codebase]
import { auth } from '@/auth';
import { NO_PERMISSIONS } from '@/types/permissions';
import type { Permissions } from '@/types/permissions';
import Link from 'next/link';
import { Card, CardHeader, CardTitle } from '@/components/ui/card';
import { ShoppingBag, Calculator, Package, Receipt } from 'lucide-react';

export default async function Home() {
  const session = await auth();
  const permissions: Permissions = session?.user?.permissions ?? NO_PERMISSIONS;

  const cards = [
    { label: 'Productos', href: '/productos', icon: ShoppingBag, show: permissions.canViewProducts },
    { label: 'Calculadora', href: '/calculadora', icon: Calculator, show: permissions.canUseCalculator },
    { label: 'Insumos', href: '/insumos', icon: Package, show: permissions.canViewSupplies },
    { label: 'Gastos', href: '/finanzas/gastos', icon: Receipt, show: permissions.canViewExpenses },
  ].filter((c) => c.show);

  return (
    <div className='grid grid-cols-2 gap-4 p-6 sm:grid-cols-4'>
      {cards.map((card) => (
        <Link key={card.href} href={card.href}>
          <Card className='hover:bg-muted/50 cursor-pointer transition-colors'>
            <CardHeader className='items-center gap-2 text-center'>
              <card.icon className='text-primary size-8' />
              <CardTitle className='text-base font-semibold'>{card.label}</CardTitle>
            </CardHeader>
          </Card>
        </Link>
      ))}
    </div>
  );
}
```

### Sort Order Migration Pattern
```typescript
// nemea-back/src/database/migrations/TIMESTAMP-AddSortOrderToProductSizes.ts
// Source: established migration pattern [VERIFIED: codebase]
import { MigrationInterface, QueryRunner } from 'typeorm';

export class AddSortOrderToProductSizes1773100000000 implements MigrationInterface {
  name = 'AddSortOrderToProductSizes1773100000000';

  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(
      `ALTER TABLE "product_sizes" ADD COLUMN "sort_order" SMALLINT NOT NULL DEFAULT 0`
    );
    // Seed known sizes with logical display order
    await queryRunner.query(`
      UPDATE "product_sizes" SET "sort_order" = CASE "name"
        WHEN 'Unico' THEN 0
        WHEN 'Chico' THEN 10
        WHEN 'Mediano' THEN 20
        WHEN 'Grande' THEN 30
        ELSE 50
      END
    `);
    // Numeric belt sizes: ordered by numeric value ascending
    await queryRunner.query(`
      UPDATE "product_sizes" SET "sort_order" = CAST("name" AS SMALLINT) * 10
      WHERE "name" ~ '^[0-9]+$'
    `);
  }

  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(
      `ALTER TABLE "product_sizes" DROP COLUMN "sort_order"`
    );
  }
}
```

async findAll(dimension: string): Promise<CatalogEntity[]> {
