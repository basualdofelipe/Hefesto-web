## Directory Purposes

**`.claude/`:**
- Purpose: Framework for AI-assisted development (GSD — Get Shit Done)
- Contains: Agent definitions, command scripts, documentation templates, workflow automation
- Key files: `.claude/commands/commit`, `.claude/commands/push`, `.claude/docs/planning/` (roadmap, scope, decisions)

**`.planning/codebase/`:**
- Purpose: Auto-generated codebase analysis documents
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md, STACK.md, INTEGRATIONS.md
- Generated by: `/gsd:map-codebase` orchestrator
- Consumed by: `/gsd:plan-phase` and `/gsd:execute-phase` for context-aware code generation

**`nemea-front/src/`:**
- Purpose: Frontend application source code
- Contains: Next.js pages, React components, type definitions, utilities, tests

  - `app/`: Next.js App Router directory structure
    - `layout.tsx`: Root layout component (fonts, metadata, ThemeProvider)
    - `page.tsx`: Home page (client component with theme toggle)
    - `__tests__/`: Colocated tests (Jest convention)
    - `globals.css`: Global Tailwind CSS styles

  - `components/`:
    - `ui/`: Shadcn/ui primitive components (from CLI, auto-installed)
      - `button.tsx`, `card.tsx`, `input.tsx`, `label.tsx`, `select.tsx`, `tabs.tsx`, `collapsible.tsx`, `separator.tsx`, `sonner.tsx`
    - (Future) Business components (e.g., `ProductTable`, `SuppliesForm`, `CalculatorWidget`)

  - `lib/`: Utility functions
    - `utils.ts`: `cn()` function for merging Tailwind classes

  - `providers/`:
    - `ThemeProvider.tsx`: Wraps next-themes, provides dark/light theme toggle

  - `hooks/`: (Planned) Custom React hooks
    - Future examples: `useAuth`, `useSupplies`, `useProducts`

  - `types/`: (Planned) TypeScript type definitions
    - Future examples: `api.ts` (request/response shapes), `entities.ts` (Supply, Product, User)

**`nemea-front/public/`:**
- Purpose: Static assets served as-is
- Contains: Fonts (Engraving CC, Engraving Shaded CC), brand images, favicon
- Accessed via: Absolute paths starting with `/` (e.g., `src="../../public/fonts/EngravingCC.ttf"`)

**`nemea-back/src/`:**
- Purpose: Backend application source code (scaffolded, awaiting Phase 2)
- Planned structure (from decisions.md):
  - `main.ts`: Entry point (initialize NestJS app, start server on :4000)
  - `auth/`: AuthModule (Google OAuth validation, JWT, whitelist guard)
  - `supplies/`: SuppliesModule (CRUD supplies, prices, suppliers, types)
  - `products/`: ProductsModule (CRUD products, catalogs)
  - `costs/`: CostsModule (cost calculation logic)
  - `config/`: ConfigModule (CRUD Tiendanube config)
  - Each module will follow NestJS standard: controller, service, entity, DTO

## Key File Locations

**Entry Points:**

- `nemea-front/src/app/layout.tsx`: Root layout, initializes fonts and providers
- `nemea-front/src/app/page.tsx`: Home page (currently just branding + theme toggle)
- `nemea-back/src/main.ts`: Backend entry point (not yet implemented)

**Configuration:**

- `nemea-front/next.config.mjs`: Next.js build config (Turbopack, SVG webpack loader)
- `nemea-front/tsconfig.json`: TypeScript compiler options (strict mode, path aliases `@/*` → `src/*`)
- `nemea-front/.prettierrc.json`: Prettier formatting rules (single quotes, trailing comma, 2-space indent)
- `nemea-front/eslint.config.mjs`: ESLint rules (cognitive complexity max 30, no `any`, explicit return types)
- `nemea-front/jest.config.js`: Jest test config (jsdom environment, moduleNameMapper for `@/*` alias)
- `nemea-front/components.json`: Shadcn/ui CLI config (new-york style, Lucide icons, path aliases)

**Core Logic:**

- `nemea-front/src/lib/utils.ts`: Tailwind class merging utility (`cn()`)
- `nemea-front/src/components/ui/*.tsx`: Shadcn/ui primitives (auto-generated, not hand-written)

**Testing:**

- `nemea-front/src/app/__tests__/page.test.tsx`: Test for home page (renders NEMEA heading)
- `nemea-front/jest.setup.ts`: Jest setup file (imports testing-library/jest-dom)

## Naming Conventions

**Files:**

- Pages: `page.tsx` in route directory (Next.js convention)
- Components: PascalCase filename, single default export (e.g., `ThemeProvider.tsx` exports `ThemeProvider`)
- Utilities: camelCase filename (e.g., `utils.ts`)
- Tests: `*.test.tsx` or `*.spec.tsx` colocated with source
- Hooks: `use*` prefix (e.g., `useAuth.ts`)

**Directories:**

- Feature modules: plural, lowercase (e.g., `components/`, `hooks/`, `lib/`)
- Routes in App Router: lowercase, hyphens (e.g., `/calculadora`, `/configuracion`, `/productos`)
- UI components: `components/ui/` (Shadcn convention)

**TypeScript/JavaScript:**

- Types: PascalCase (e.g., `ThemeProviderProps`)
- Constants: UPPER_SNAKE_CASE if immutable
- Functions: camelCase (e.g., `cn()`, `toggleTheme()`)
- Variables: camelCase (e.g., `mounted`, `theme`)

## Where to Add New Code

**New Feature (e.g., Products page):**

1. Create route directory: `nemea-front/src/app/productos/`
2. Add page component: `nemea-front/src/app/productos/page.tsx`
3. Add layout (if needed): `nemea-front/src/app/productos/layout.tsx`
4. Add test: `nemea-front/src/app/productos/__tests__/page.test.tsx`
5. Add types: `nemea-front/src/types/product.ts` (if not already in planned `types/`)
6. Add hooks (if needed): `nemea-front/src/hooks/useProducts.ts`

**New Component (e.g., ProductTable):**

1. Create: `nemea-front/src/components/ProductTable.tsx`
2. Add test: `nemea-front/src/components/__tests__/ProductTable.test.tsx`
3. Import in page: `import { ProductTable } from '@/components/ProductTable'`

**Utilities:**

- Shared helpers: `nemea-front/src/lib/`
- Example: `nemea-front/src/lib/calculations.ts` for cost computation logic

**Hooks:**

- `nemea-front/src/hooks/`
- Example: `useAuth.ts` (fetch current user), `useSupplies.ts` (CRUD supplies via API)

**Backend Feature (e.g., SuppliesModule in Phase 4):**

1. Create module directory: `nemea-back/src/supplies/`
2. Add entity: `nemea-back/src/supplies/entities/supply.entity.ts` (TypeORM entity)
3. Add DTO: `nemea-back/src/supplies/dto/create-supply.dto.ts`, `update-supply.dto.ts`
4. Add repository: `nemea-back/src/supplies/supplies.repository.ts` (TypeORM repository methods)
5. Add service: `nemea-back/src/supplies/supplies.service.ts` (business logic)
6. Add controller: `nemea-back/src/supplies/supplies.controller.ts` (HTTP routes)
7. Add module: `nemea-back/src/supplies/supplies.module.ts` (wire controller, service, entity)
8. Register in `nemea-back/src/app.module.ts`

## Special Directories

**`nemea-front/.next/`:**
- Purpose: Next.js build output and dev artifacts
- Generated: Yes (by `npm run dev` and `npm run build`)
- Committed: No (in .gitignore)

**`nemea-front/node_modules/`:**
- Purpose: Installed npm packages
- Generated: Yes (by `npm install`)
- Committed: No (in .gitignore)

**`nemea-front/.swc/`:**
- Purpose: SWC compiler cache (faster builds than Babel)
- Generated: Yes
- Committed: No (in .gitignore)

**`nemea-front/public/fonts/`:**
- Purpose: Custom font files (Engraving CC, Engraving Shaded CC)
- Generated: No
- Committed: Yes (binary files, tracked in git)

**`nemea-back/.env`, `nemea-back/.env.local`:**
- Purpose: Environment variables (database URL, JWT secret, etc.)
- Generated: No (manually created from .env.example)
- Committed: No (in .gitignore for both nemea-back and root)

**`.claude/plans/nemea-front/` and `.claude/plans/nemea-back/`:**
- Purpose: Phase-specific implementation plans
- Generated: Yes (created by `/gsd:plan-phase`)
- Committed: Yes (part of project history)

**`.planning/codebase/`:**
- Purpose: Codebase analysis snapshots
- Generated: Yes (by `/gsd:map-codebase`)
- Committed: Yes (refreshed when architecture changes)

## Import Path Aliases

**Frontend:**
- `@/*` → `nemea-front/src/*` (configured in tsconfig.json and components.json)
- Usage examples:
  - `import { cn } from '@/lib/utils'`
  - `import { Button } from '@/components/ui/button'`
  - `import { ThemeProvider } from '@/providers/ThemeProvider'`

## Monorepo Structure Notes

- **Root `Nemea-web/`**: Orchestrator only (CLAUDE.md, .claude/, git config)
- **`nemea-front/`**: Independent git history (`.git/` subfolder)
- **`nemea-back/`**: Independent git history (`.git/` subfolder)
- Each sub-repo has its own `package.json`, `tsconfig.json`, linting config
- No shared monorepo tooling (no `npm workspaces` or `lerna`) — intentionally simple
- Root `.gitignore` ignores `node_modules/`, `dist/`, `.env*` globally
- Sub-repos have their own `.gitignore` (e.g., nemea-front ignores `.next/`)
